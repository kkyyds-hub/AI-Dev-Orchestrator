# AI 开发调度系统总规划冻结文档

- 项目：`AI-Dev-Orchestrator`
- 范围：面向个人开发者的本地优先 AI 开发调度系统
- 文档版本：`v1.1`
- 冻结日期：`2026-03-08`
- 文档性质：**总规划冻结文档**
- 目标：为 `V1 / V2 / V3`、阶段文档和每日计划提供统一上位约束

---

## 0. 范围定位（冻结）

1. 本项目不是普通聊天机器人，也不是单纯的 AI IDE 外壳，而是一个“任务驱动”的开发调度系统。
2. 系统的核心目标是：把用户的项目设想转成任务、调度执行、验证结果、记录成本，并在必要时请求人工介入。
3. 当前优先级不是“功能越多越好”，而是先建立最小闭环，再逐步增强调度质量、成本控制和观测能力。
4. 所有下位文档都必须服从本文件；若发生冲突，默认以本文件为准。

---

## 1. 边界决策（已冻结）

1. 目标用户定位为单用户、项目作者本人：**YES**
2. 运行方式以本地优先、自托管为主：**YES**
3. 允许依赖大模型 API，但不依赖第三方基础设施平台：**YES**
4. 系统核心对象以 `Task / Run / Event` 为中心，而不是聊天消息：**YES**
5. 通过规则和路由选择模型，而不是默认暴力使用强模型：**YES**
6. 执行结果必须可验证、可恢复、可追踪：**YES**
7. UI 的角色是“控制台”，不是花哨演示界面：**YES**
8. 多用户 SaaS、复杂协同平台、高风险密钥托管：**NO**
9. 无边界多智能体自由协商、复杂自动部署平台：**NO**
10. 为了“看起来高级”而引入分布式基础设施：**NO**

## 2. 核心问题基线（冻结）

### 2.1 任务拆解问题

用户给的是高层目标，系统执行需要的是可验收的最小任务，因此必须具备任务树或任务 DAG 的拆解能力。

### 2.2 模型选择问题

不同任务的复杂度、风险和上下文成本不同，系统必须能按任务类型与风险路由模型。

### 2.3 上下文污染问题

长链路开发会快速积累冗余上下文，因此系统必须优先依赖摘要、上下文包和结构化状态，而不是不断堆叠历史对话。

### 2.4 执行可靠性问题

生成代码不等于任务完成。系统必须具备验证、测试、自查、失败处理和状态更新能力。

### 2.5 人机协作问题

凭证、账号验证、高风险确认和需求分支决策等事项，必须能明确转入人工处理，而不是强行自动化。

## 3. 总体工作流（冻结）

1. 用户输入项目设想。
2. `Planner` 生成项目摘要、任务清单和验收标准。
3. `Router` 基于优先级、依赖和风险选择可执行任务与模型。
4. `Context Builder` 组装最小必要上下文。
5. `Executor` 执行代码修改、命令执行或文档生成。
6. `Verifier` 运行验证步骤并给出结果。
7. 成功时写回任务状态、成本和项目记忆。
8. 失败时进入重试、升级模型、挂起或人工介入。
9. `UI Console` 持续展示队列、成本、日志和阻塞项。

### 3.1 核心模块冻结清单

| 模块 | 职责 |
| --- | --- |
| `Planner` | 任务拆解、摘要生成、验收标准生成 |
| `Router` | 任务选取、模型选择、失败升级决策 |
| `Context Builder` | 组装任务所需的最小上下文包 |
| `Executor` | 执行代码修改、命令、文档和脚本 |
| `Verifier` | 测试、构建、静态检查、自查 |
| `Memory` | 沉淀项目摘要、任务摘要、执行记录 |
| `Budget Guard` | 控制日预算、会话预算和失败重试成本 |
| `Intervention Inbox` | 承接人工介入与高风险确认 |
| `UI Console` | 透明展示运行状态与阻塞点 |

## 5. V1 范围冻结

### 5.1 核心定位

`V1` 不追求“最强自治”，而是追求“核心闭环成立”。

### 5.2 范围

1. 单用户
2. 单项目
3. 单工作区
4. 单仓库
5. 有限并发
6. 手动启动冲刺
7. 支持暂停与恢复

### 5.3 必须完成的能力

1. 输入项目 brief 并生成结构化任务计划。
2. 任务支持状态流转。
3. 可以按规则路由模型。
4. 可以构建任务级最小上下文。
5. 可以执行代码类任务。
6. 可以跑最小验证步骤。
7. 可以记录 token 使用与成本估算。
8. 可以将失败任务转入人工处理。
9. 有最小控制台展示系统状态。

### 5.4 `V1` 明确不做

1. 复杂多代理协商
2. 浏览器自动化闭环
3. 自动部署生产环境
4. 多用户权限系统
5. 高风险密钥托管
6. 自由形式插件生态

---

## 6. V2 / V3 进入前提（冻结）

### 6.1 进入 `V2` 的前提

1. `V1` 已形成可运行的最小闭环。
2. 任务状态流转、失败处理和恢复能力已具备基本稳定性。
3. 已出现真实的成本、复盘或观测痛点。

### 6.2 进入 `V3` 的前提

1. `V2` 已经带来真实收益。
2. 核心对象模型、状态机边界和 Provider 抽象已较清晰。
3. 已出现扩展性需求，而不是纯粹的技术好奇心。

---

## 7. 全局成功标准（冻结）

### 7.1 产品级成功标准

1. 用户可以输入项目设想并得到结构化任务计划。
2. 系统可以自主推进多轮开发任务。
3. 用户可以清晰看到系统当前在做什么。
4. 用户可以看到模型使用、成本、失败原因与阻塞点。
5. 系统在中断后可以恢复，而不是从零开始。

### 7.2 工程级成功标准

1. 模块边界清晰，可替换模型供应商。
2. 任务状态可追踪。
3. 上下文构建机制可控。
4. 每轮执行都有记录和结果。
5. 失败任务可重试、升级或挂起。

### 7.3 文档级成功标准

1. 文档体系能支撑阶段冻结和每日执行。
2. 每个阶段都有清晰的范围、边界、输出和完成标准。
3. 每日计划可直接转化为可验收的交付动作。

---

## 8. 对后续文档与开发的约束

1. 每个阶段文档都应明确“范围定位、边界决策、阶段路线、DoD”。
2. 每份每日计划都应明确“今日冻结范围、Step、输出、DoD、状态、证据”。
3. `V2 / V3` 只能作为长期约束来源，不能直接变成当前每日任务清单。
4. 技术选型优先服务于闭环、恢复、观测和成本控制。
5. 任何新增复杂度，都必须说明它解决了什么真实问题。

---

## 9. 最终结论

本项目的主线已经冻结为：**先建立 `V1` 最小闭环，再进入 `V2` 调度与观测增强，最后视真实收益进入 `V3` 平台化演进。**

只要后续文档与实现始终围绕“任务驱动、短上下文、可验证、可恢复、可观测、可人工接管”这六条主线推进，项目方向就不会跑偏。
