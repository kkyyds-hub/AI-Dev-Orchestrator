## 一、决策回放模型

### 1.1 设计目标

让历史路由、执行、验证和收口过程具备回放能力，使用户无需阅读完整原始日志也能理解任务运行的主线决策过程。

### 1.2 核心数据结构

#### DecisionTraceItem

单个决策时间线节点，表示一次关键决策事件：

```python
@dataclass(slots=True, frozen=True)
class DecisionTraceItem:
    timestamp: str          # ISO 格式时间戳
    stage: str              # 决策阶段：routing/claim/context/guard/execution/verification/cost/parallel/finalize/recovery
    title: str              # 中文标题，如"任务路由"、"执行结果"
    event: str              # 原始事件名，如"task_routed"、"execution_finished"
    level: str              # 日志级别：INFO/WARNING/ERROR
    summary: str            # 决策摘要，一句话说明发生了什么
    data: dict[str, Any]    # 附加结构化数据
```

#### DecisionTrace

完整的单次运行决策回放：

```python
@dataclass(slots=True, frozen=True)
class DecisionTrace:
    run_id: UUID
    task_id: UUID
    run_status: RunStatus
    failure_category: RunFailureCategory | None
    quality_gate_passed: bool | None
    trace_items: list[DecisionTraceItem]      # 按时间排序的决策节点
    failure_review: FailureReviewRecord | None # 失败复盘记录（如有）
```

#### DecisionHistoryItem

任务级别的历史决策摘要（轻量级）：

```python
@dataclass(slots=True, frozen=True)
class DecisionHistoryItem:
    run_id: UUID
    status: RunStatus
    failure_category: RunFailureCategory | None
    quality_gate_passed: bool | None
    created_at: str
    headline: str           # 一句话总结这次运行
    stages: list[str]       # 经历的决策阶段列表
```

### 1.3 事件映射表

从结构化日志事件到决策阶段的映射：

| 原始事件 | 决策阶段 | 中文标题 |
|---------|---------|---------|
| task_routed | routing | 任务路由 |
| run_claimed | claim | 任务领取 |
| context_built | context | 上下文构建 |
| guard_blocked | guard | 预算/重试守卫 |
| execution_finished | execution | 执行结果 |
| verification_finished | verification | 验证结果 |
| verification_skipped | verification | 验证跳过 |
| cost_estimated | cost | 成本估算 |
| run_finalized | finalize | 最终收口 |
| worker_slot_assigned | parallel | 槽位分配 |
| worker_slot_released | parallel | 槽位释放 |
| worker_rolled_back | recovery | 异常回滚 |

## 二、失败聚类模型

### 2.1 设计目标

将存储的失败复盘记录按失败类别聚合，支持快速识别同类问题的分布和频率。

### 2.2 核心数据结构

#### FailureReviewCluster

失败聚类结果：

```python
@dataclass(slots=True, frozen=True)
class FailureReviewCluster:
    cluster_key: str                    # 聚类键，通常是 failure_category 值
    failure_category: str               # 失败类别
    count: int                          # 该类别的失败次数
    latest_run_created_at: datetime     # 最近一次失败时间
    route_reason_excerpt: str | None    # 路由原因摘要（截断到160字符）
    sample_task_titles: list[str]       # 样本任务标题（最多5个）
    run_ids: list[UUID]                 # 样本运行ID（最多5个）
```

### 2.3 聚类规则

- 按 `failure_category` 分组，如果为空则按 `run_status` 分组
- 每个聚类保留最近5个样本
- 按最近失败时间倒序排列
- 聚类口径与失败分类保持一致

## 三、接口设计

### 3.1 GET /runs/{run_id}/decision-trace

获取单次运行的完整决策回放。

**响应结构：**

```typescript
{
  run_id: string;
  task_id: string;
  run_status: string;
  failure_category: string | null;
  quality_gate_passed: boolean | null;
  trace_items: Array<{
    timestamp: string;
    stage: string;
    title: string;
    event: string;
    level: string;
    summary: string;
    data: Record<string, any>;
  }>;
  failure_review: {
    review_id: string;
    conclusion: string;
    action_summary: string;
    // ... 其他字段
  } | null;
}
```

**使用场景：**
- 任务详情页展示完整决策时间线
- 失败分析时查看关键决策节点
- 调试时理解任务执行路径

### 3.2 GET /tasks/{task_id}/decision-history

获取任务的所有运行历史摘要。

**响应结构：**

```typescript
Array<{
  run_id: string;
  status: string;
  failure_category: string | null;
  quality_gate_passed: boolean | null;
  created_at: string;
  headline: string;
  stages: string[];
}>
```

**使用场景：**
- 任务详情页展示历史运行列表
- 对比多次运行的决策路径差异
- 识别重复失败模式

### 3.3 GET /console/review-clusters

获取全局失败聚类概览。

**响应结构：**

```typescript
Array<{
  cluster_key: string;
  failure_category: string;
  count: number;
  latest_run_created_at: string;
  route_reason_excerpt: string | null;
  sample_task_titles: string[];
  run_ids: string[];
}>
```

**使用场景：**
- 控制台首页展示失败分布
- 识别高频失败类别
- 快速定位需要人工介入的问题

## 五、实现要点

### 5.1 回放结构设计原则

1. **足够表达关键过程**：覆盖路由、执行、验证、收口全流程
2. **无需阅读原始日志**：摘要信息足以理解主线
3. **接口名称自解释**：decision-trace、decision-history、review-clusters
4. **前端可直接消费**：返回结构与前端展示需求对齐

### 5.2 聚类口径一致性

- 失败分类与 `RunFailureCategory` 枚举保持一致
- 聚类键与失败复盘记录的 `failure_category` 字段对齐
- 确保前端展示的失败类别在各处含义相同

### 5.3 性能考虑

- 决策回放限制读取最近200条日志事件
- 决策历史限制每次运行读取50条事件
- 聚类接口默认返回最多20个聚类
- 所有接口支持分页或限制参数

## 八、有限并行补充设计（Day12 / Day13）

### 8.1 槽位模型

V2-C 的并行能力限定为**单机固定槽位**，不做分布式调度。

核心对象：

- `WorkerSlotService`：维护进程内固定数量的槽位状态
- `WorkerPool`：一次启动多个保守 worker cycle，但数量不超过槽位上限
- `WorkerSlotLease`：槽位租约，确保 acquire / bind / release 成对出现
- `WorkerSlotSnapshot`：控制台消费的全局槽位快照

最小模型如下：

```text
MAX_CONCURRENT_WORKERS = N

WorkerPool.run_once(requested_workers)
  -> acquire 最多 N 个 slot
  -> 每个 slot 启动一个独立 Session 的 TaskWorker.run_once()
  -> 运行中 bind 当前 task_id / run_id
  -> 结束后记录 worker_slot_assigned / worker_slot_released
  -> release slot，回到 idle
```

### 8.2 配置字段

- `MAX_CONCURRENT_WORKERS`
  - 含义：单机允许并行执行的最大 worker 数
  - 默认值：`2`
  - 约束：最小为 `1`
  - 目的：把并行能力限制在固定上界内，避免无限扩张并发

### 8.3 安全边界

| 边界 | 设计口径 | 当前落实方式 |
| --- | --- | --- |
| 并发范围 | 仅限单进程、本机固定槽位 | `WorkerSlotService` 在进程内维护固定槽位 |
| 任务领取 | 不允许跳过状态机直接并发改状态 | 先 route，再通过 `claim_pending_task` 原子领取 |
| 竞争处理 | 多 worker 同时命中同一任务时必须自动重试 | `TaskWorker.run_once()` 在 claim 失败后重路由重试 |
| 会话隔离 | 每个并发 worker 使用独立数据库会话 | `WorkerPool._run_slot()` 内部创建独立 `SessionLocal()` |
| 预算控制 | 并行不允许绕过预算守卫 | 每个 worker 在执行前单独调用 `BudgetGuardService.evaluate_before_execution()` |
| 回放观测 | 并行执行后仍必须可追溯 | 每个 run 写独立日志，并追加 `worker_slot_assigned` / `worker_slot_released` |
| 回退路径 | 必须保留单轮执行能力 | `/workers/run-once` 继续保留并可单独调用 |

### 8.4 并发风险与控制

1. **重复选中同一任务**
   - 风险：多个 worker 同时 route 到同一待处理任务
   - 控制：`claim_pending_task` 负责原子领取；若 claim 失败，worker 自动重路由重试

2. **状态一致性被破坏**
   - 风险：并发执行时 task / run 状态错乱
   - 控制：所有状态落库仍走 repository 与状态机服务，不新增旁路写入

3. **预算守卫被并行稀释**
   - 风险：多个 worker 同时起跑导致预算判断失真
   - 控制：V2-C 采用保守固定槽位 + 每次执行前单独预算检查；当前目标是“不绕过守卫”，不是做分布式预算预留

4. **日志与复盘链路丢失并行信息**
   - 风险：只能看到执行结果，看不到哪个槽位处理了 run
   - 控制：run 日志中补写并行槽位事件，`DecisionTrace` 映射到 `parallel` 阶段

### 8.5 验证口径

Day13 收口时，最少要验证以下闭环：

1. `/workers/run-pool-once` 可在固定槽位下完成多 worker 周期
2. `/console/worker-slots` 能返回槽位状态和等待压力
3. 并行运行后的 `/runs/{run_id}/logs` 仍能看到槽位事件
4. 并行运行后的 `/runs/{run_id}/decision-trace` 仍能看到 `parallel` 阶段
5. `/workers/run-once` 仍可作为单轮回退路径
