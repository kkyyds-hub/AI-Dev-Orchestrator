- 阶段：`V4`
- 文档版本：`v1.1`
- 冻结日期：`2026-03-21`
- 文档性质：**V4 正式版总纲**
- 目标：定义 V4 的冻结范围、模块主线、Day 级推进边界和验收口径，并按冻结文档逐日回填仓库接入实现；当前已完成 Day01-Day07，后续 Day 仍按计划推进。

---

## 0. 范围定位（冻结）

1. `V4` 不是继续补“更花哨的老板界面”，而是把 V3 已有的项目级编排、角色协作、交付件审批能力真正落到**本地代码仓库执行闭环**上。
2. `V4` 的重点不是立即自动写完所有代码，而是先把“仓库接入 -> 文件定位 -> 变更计划 -> 验证证据 -> 提交候选”这条主线冻结清楚。
3. `V4` 继续坚持本地优先、单用户、可人工接管、高风险动作先闸门后执行的底线，不为了追求自动化速度牺牲可控性。
4. 所有 V4 Day 文档、测试文档和后续实现都必须服从本总纲；若出现冲突，以本文件为准。
5. 当前除 Day01-Day07 外，其余 Day 文档、关键产物路径和测试骨架仍只是后续回填目标，不代表仓库中已经存在对应实现。

## 1. 启动前提（已冻结）

1. `V1` 已提供任务、执行、验证、日志、预算守卫与控制台基础：**YES**
2. `V2` 已提供状态机、失败复盘、路由与观测增强：**YES**
3. `V3` 已提供 `Project`、角色协作、交付件、审批、Skill、项目记忆与策略入口：**YES**
4. 当前缺口已经从“有没有项目视角”转为“项目如何真正落到仓库与代码变更执行上”：**YES**
5. 在仓库边界、安全边界和人工闸门未冻结前直接进入大规模仓库自动化：**NO**

---

## 2. V4 为什么做（冻结）

### 2.1 项目视角已经成形，但仓库仍然是黑盒

V3 已能把项目、角色、交付件和审批串起来，但系统仍缺少“这个项目对应哪个仓库、当前仓库长什么样、这一轮改动准备基于哪条分支推进”的正式模型。

### 2.2 任务与交付件还没有稳定的文件级落点

当前任务和交付件可以被规划、审批和复盘，但还缺少“候选文件集合、变更计划、变更批次”这些仓库执行前的中间层，导致后续仓库接入实现容易范围漂移。

### 2.3 验证、差异与审批证据尚未围绕代码变更归档

V1 / V2 已有运行日志、验证结果和失败复盘，V3 已有交付件与审批，但这些信息还没有围绕“本轮仓库变更”形成统一证据包，老板很难直观看到一次代码改动为什么可放行。

### 2.4 高风险仓库动作需要显式人工闸门

仓库路径、敏感文件、危险命令、大范围差异和提交动作都比普通任务规划更高风险，必须在正式实现前先冻结人工确认边界，避免后续开发时 scope 失控。

---

## 3. V4 冻结目标（冻结）

1. 让每个 `Project` 能绑定一个本地主仓库，并具备仓库快照与变更会话入口。
2. 让任务可以落到候选文件、代码上下文包、变更计划和变更批次，而不是停留在抽象描述层。
3. 让仓库级验证命令、验证运行记录、差异摘要和证据包成为审批链路的一部分。
4. 让提交草案与放行检查单在真正执行远程仓库动作之前先形成可审阅、可解释、可追责的最小闭环。
5. 让 V4 最终形成一套可以逐 Day 推进并回填的正式文档体系，而不是一份模糊路线图。

---

## 4. V4 冻结范围（冻结）

1. **仓库入口**：项目绑定本地主仓库、记录仓库路径、安全边界、默认基线分支和工作区摘要。
2. **工作区快照**：支持目录结构、语言分布、忽略目录、仓库状态和变更会话的最小快照。
3. **文件级落点**：支持候选文件定位、代码上下文包、任务与交付件到变更计划的映射。
4. **变更准备**：支持变更批次、文件重叠提醒、执行前风险预检和人工确认链路。
5. **验证基线**：支持仓库级命令模板、验证运行记录、失败归因和最小回写接口。
6. **证据归档**：支持差异摘要、证据包、回退重做和仓库复盘收口。
7. **放行闭环**：支持提交草案、放行检查单、老板审批闸门与一轮 V4 最小演示。
8. **正式文档**：支持总纲、总览、模块说明、Day 计划与 Day 测试验证骨架，且后续按 Day 回填。

## 6. V4 Day 编排原则（冻结）

1. 每个 Day 仍然必须同时维护计划文档与测试验证文档，后续实施时两者同步回填。
2. Day 之间严格按边界推进：前一日只打地基，不提前挪用后一日范围做“顺手功能”。
3. 任何涉及路径安全、敏感文件、危险命令、分支状态变化或提交动作的能力，都必须先经过 Day08 / Day14 定义的人工闸门口径。
4. Day15 只做最小闭环演示，Day16 只做验收和文档收口，不在收口日继续扩展产品范围。
5. Day03 / Day13 / Day15 需要额外使用“只读记录 / 草案候选 / 闭环演示”的口径收敛，避免被误读为真实 Git 自动化。

### 6.1 仓库安全规则与人工闸门底线（冻结）

1. 仅允许在项目显式绑定的本地仓库根路径内进行读取、扫描、索引、计划和证据归档；不跨根目录、不穿透未绑定路径、不把系统目录或临时目录误当成工作区。
2. Day01-Day07 默认以“读仓库、记状态、做计划”为主；即使识别到脏工作区、异常分支或候选风险，也只记录风险，不自动清理、不自动切换、不自动修复。
3. 以下统一视为高风险 Git 写操作：`checkout`、创建 / 删除分支、`reset`、`clean`、`stash`、`rebase`、`cherry-pick`、`commit`、`tag`、`push`、创建 PR / merge。
4. 高风险 Git 写操作不属于 V4 当前冻结版的默认执行范围；若未来进入实施，也必须先过 Day08 预检、Day14 审批，并由人工显式放行。
5. “审批通过”只表示“允许人工进入下一步”，不等于系统自动执行 Git 写动作，更不等于远程仓库放行。

### 6.2 易误解 Day 的硬边界（冻结）

| Day | 本日只交付 | 明确不做 | 为后续 Day 留下什么 |
| --- | --- | --- | --- |
| Day03 | `ChangeSession` 记录 + 分支 / 工作区状态快照 | 不做 `checkout`、建分支、`stash`、`reset`、`merge`、`commit` | 给 Day05-Day08 提供稳定的会话基线与风险判断入口 |
| Day13 | `CommitCandidate` 草案 + 草案修订历史 | 不写入 `.git`、不生成真实 commit hash、不做 `git commit` / `push` | 给 Day14 提供可审批的提交候选 |
| Day15 | 串联 Day01-Day14 的最小演示与缺口记录 | 不把演示扩成新开发日，不要求真实 commit / push / PR 结果 | 给 Day16 提供验收输入与文档收口依据 |

## 7. V4 明确不做（冻结）

1. 远程仓库克隆、账号鉴权、自动 push、自动创建 PR、自动 merge。
2. 多人协作权限系统、在线协同编辑、跨团队工作区隔离。
3. 分布式 Worker、远程沙箱集群、复杂调度基础设施升级。
4. 完整 AST 索引、复杂符号级重构引擎、全量语义代码图谱。
5. 向量数据库、复杂 RAG、跨项目仓库记忆搜索后台。
6. 在 V4 文档阶段就承诺所有仓库动作自动执行；V4 首先冻结的是**可执行前的结构化准备与证据闭环**。

---

## 8. V4 成功标准（冻结）

1. 用户看完 V4 文档后，能一眼知道“Day01-Day07 已完成、Day08+ 尚未开始”，以及为什么要做仓库接入、分几块做、每天做什么、每天验什么。
2. V4 Day01-Day16 可以像 V3 一样逐天开工并回填，不需要再重新发明一套目录和文档口径。
3. V4 的实现边界明确，不把远程仓库自动化、多人协作和复杂平台化能力混进当前冻结范围。
4. 后续仓库接入实现能围绕本地仓库、文件落点、验证证据和审批闸门推进，而不是直接跳到高风险自动化，或把 Day03 / Day13 / Day15 误读为真实 Git 自动执行。

---

## 9. 最终结论

`V4` 已冻结为：**仓库接入与代码变更闭环版本。**

它的目标不是立刻把系统变成全自动 Git 平台，而是先把 `Project` 视角正式推进到仓库、文件、验证和提交候选这一层，并给后续开发一套可直接执行的正式文档体系。

当前已完成的是 **V4 Day01-Day07：仓库实体与项目绑定模型 + 工作区扫描与目录快照基线 + 分支会话与仓库状态快照 + 仓库首页 + 文件定位与代码上下文包 + ChangePlan 草案 + ChangeBatch 执行准备**，其余 Day 仍按冻结文档等待后续实现回填。
