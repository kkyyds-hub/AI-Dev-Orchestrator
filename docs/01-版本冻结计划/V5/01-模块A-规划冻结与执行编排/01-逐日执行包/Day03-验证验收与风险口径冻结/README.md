## 1. 本日定位

Day03 已完成的不是 verify / accept 实操，而是把 Day05-Day16 共用的**最低验证要求、证据口径、状态回填规则、风险分级、验收输入边界**一次冻结成后续线程必须共用的正式口径。

## 2. 本轮正式冻结结论

1. Day05-Day16 的 day-level 最低验证仍以各 Day README 与 `00-V5正式冻结执行计划.md` 合同表为准，Day03 不重开工作包命名、owner、主要改动面。
2. Day03 额外冻结了统一状态口径：
   - `已规划`：尚未启动，只存在冻结后的方案与入口。
   - `进行中`：已经启动，但本日最低验证尚未闭合，或正式回填尚未完成。
   - `已实现待验证`：实现或联调已落地，但还缺本日要求的关键验证证据，不能写成 `已完成`。
   - `已完成`：本日正式产物、最低验证、缺证说明、交接回填都已落到正式文档。
   - `阻塞`：存在 R1 阻塞级风险，无法满足本日最低验证，且已明确阻塞原因与解除前提。
3. Day03 额外冻结了统一证据口径：
   - `E0 缺证`：只有口头说明或聊天描述，没有正式文档 / 日志 / 记录入口。
   - `E1 文档证据`：正式 README、状态回填、验证记录、交接记录已落盘；只足够支撑 Day03/Day04 这类纯文档线程写 `已完成`。
   - `E2 单链路证据`：至少存在 1 条 compile / build / API / smoke / 页面回显 / 日志证据；Day05-Day14 要写 `已完成`，最低不能低于本日要求的 E2 级别。
   - `E3 闭环证据`：除 E2 外，还包含回归结果、缺证说明、风险归类与可供下游消费的材料包；Day08、Day15、Day16 的正式收口必须达到对应 E3 要求。
4. Day03 额外冻结了统一风险分级：
   - `R1 阻塞级`：直接阻断下一 Day 或 gate 输入，当前状态只能写 `阻塞`，Day16 不得据此给 Pass。
   - `R2 高风险级`：允许继续推进，但当前 Day 最高只能写 `已实现待验证` 或在 verify/accept 线程写 `部分通过`，且必须附补证入口、责任 owner、回退口径。
   - `R3 观察级`：不阻断当前 handoff，但必须进入正式风险记录，不能省略。
5. Day03 额外冻结了验收输入边界：
   - Day08 只能消费 Day05-Day07 的正式状态回填、验证记录、README/交接件和真实 smoke / 回归证据。
   - Day15 只能消费 Day05-Day14 的正式验证材料与真实运行证据，不接受“代码看起来差不多”式口头结论。
   - Day16 只能基于 Day15 verify 输出、review 风险输出（若有）与 `docs/01-版本冻结计划/V5/` 正式文档裁定；**没有 verify 材料不得写 Pass**。

## 4. 状态回填规则

1. Day05-Day14 线程想写 `已完成`，必须同时满足：本日完成定义成立、本日最低验证达到对应 E2、正式状态回填与验证记录已落盘。
2. Day08 / Day15 / Day16 想写 `已完成`，必须额外具备可供下游消费的材料包；只有单条日志或局部截图不足以写 `已完成`。
3. 任何 Day 只要还缺关键证据，但实现或文档已落地，应优先写 `已实现待验证`，禁止跳写 `已完成`。
4. `进行中` 只用于已经启动、但连最低验证都还没收齐的线程；不能用来掩盖阻塞。
5. 一旦进入 `阻塞`，必须同步写明：阻塞原因、上游依赖、解除条件、下一个 owner。

## 5. 风险分级与回填规则

1. `R1 阻塞级` 必须落到 Day 当日 `02-状态回填模板.md` 与 `03-验证记录模板.md`，并在交接件中标明“是否阻塞下一日 / gate”。
2. `R2 高风险级` 允许交接，但必须同时写明补证入口、责任 owner、预计回填位置；否则不能交给 accept 线程消费。
3. `R3 观察级` 允许继续推进，但仍要留在正式文档，防止 Day15 风险汇总失真。
4. Day15 的风险汇总必须把 Day05-Day14 的 R1/R2/R3 重新归并，不允许只写“有风险”。

## 6. Day08 / Day15 / Day16 验收输入边界

### 6.1 Day08
1. 输入来源只认 Day05-Day07 的正式 README、状态回填、验证记录、交接件与真实 smoke / 回归证据。
2. 不接受“代码已写完但日志还没整理”式口头输入。
3. 若 Phase 1 闭环只覆盖局部链路，Day08 只能输出 partial / gap 输入，不能伪装成 M1 通过材料。

### 6.2 Day15
1. 输入来源只认 Day05-Day14 的正式验证材料与真实运行证据。
2. 若缺 build / API / 闭环 smoke 任一关键入口，Day15 只能写缺证、R1 / R2 风险与 verify 输出缺口，不能偷写“已验证通过”。
3. Day15 输出必须可直接被 Day16 消费，至少包含：证据清单、风险分级、缺证说明、遗留项。

### 6.3 Day16
1. 输入来源只认 Day15 verify 输出、review 风险输出（若存在）与 V5 正式文档。
2. 没有 verify 输出材料，不得写 Pass。
3. 有 verify 输出但仍存在 `R1 阻塞级` 未收敛项，只能 Blocked。
4. Partial 必须同时给出：缺口项、责任 owner、回填位置、下一轮入口。
