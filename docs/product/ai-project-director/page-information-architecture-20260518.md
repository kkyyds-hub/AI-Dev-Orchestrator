## 0. 设计背景

当前项目已经完成运行摘要主链路、真实 DeepSeek 运行摘要联调、2C-B 运行时联调和 2C-B-R1 文档追溯修正。项目接下来不应继续零散小修页面，而应围绕“AI 项目主管 / AI 自动调度 / 多 Agent 协作”重新收口页面职责。

本阶段先确定六个核心页面 / 页面组，并补充跨页面自动摘要机制和成本治理机制：

1. 工作台
2. 项目页
3. 执行中心（任务队列、运行观测、仓库工作区）
4. 成果中心（交付物、审批）
5. 治理中心
6. 设置页
7. 全局自动摘要机制

本设计遵循以下固定边界：

- 不重建项目。
- 不换技术栈。
- 不随便改后端接口协议。
- 不做无边界大重写。
- 不做只好看但没有真实闭环的 UI。
- 不把多个页面的数据重复堆在一个页面上。
- 不把工作台做成统计大全页。
- 不把项目页做成其他页面的缩略导航页。
- 后续每个 Codex 阶段只处理一个明确目标，并有清晰验收标准。
- 页面按钮必须有真实闭环：要么调用真实后端，要么进行真实跳转，要么明确禁用并说明原因。
- 页面设计与变更需要回填到 `docs/product` 对应文档。

## 1. 当前产品风格定位

AI-Dev-Orchestrator 不是普通的项目管理系统，也不是单纯的任务列表工具。它的目标形态是：

> 用户提出目标，AI 项目主管负责生成计划、拆分任务、分配 Agent、绑定 Skill、监督运行、识别阻塞、提出调整建议，并把需要人工确认的高风险决策汇总给用户审批。

因此，项目整体 UI 风格应从“调试型后台”逐步转向“AI 项目编排工作台”。

### 1.1 页面风格关键词

| 风格关键词 | 说明 |
|---|---|
| 简洁 | 页面默认不堆大量卡片、日志、表格和技术字段。 |
| 指挥感 | 用户能明确知道当前 AI 项目主管建议做什么。 |
| 工作台感 | 页面像正式后台系统，不像调试页。 |
| 弹窗收纳 | 复杂信息通过弹窗、抽屉或独立页面承接。 |
| 状态明确 | 项目、任务、运行、审批、阻塞都要有清楚状态。 |
| 中文优先 | 面向用户的主链路文案优先中文化。 |
| 真实闭环 | 页面按钮不做假功能，不制造能力错觉。 |
| 页面分工清楚 | 工作台、项目页、任务页、运行页、仓库页、交付物页、审批页、治理页、设置页各司其职。 |

### 1.2 视觉与交互原则

1. 页面首屏要轻，不要一眼看到大量模块。
2. 复杂信息默认折叠或放入弹窗。
3. 每个页面只回答它最该回答的问题。
4. 同一类数据不要在多个页面重复完整展示。
5. 侧边栏负责全局导航，页面内部不重复做侧边栏。
6. 页面跳转要带上下文，例如当前 `projectId`。
7. 技术细节可以保留，但不应成为普通用户默认看到的内容。
8. 工作台和项目页都不直接承载完整任务列表、完整运行日志、完整仓库内容、完整审批列表。

### 1.3 当前前端风格声明

后续所有页面改造必须延续并强化当前项目已经形成的前端风格，不能突然换成另一套视觉体系。

当前前端风格应明确为：

> **简洁后台工作台风格 + AI 编排产品气质 + 中文业务摘要优先 + 技术细节弹窗收纳。**

具体表现如下：

| 风格项 | 要求 |
|---|---|
| 页面气质 | 像正式后台工作台，不像调试页、Demo 页、数据库表格页。 |
| 首屏密度 | 首屏保持轻量，默认只展示用户判断当前状态所需的信息。 |
| 信息承载 | 摘要常驻，详情弹窗 / 抽屉 / 独立页面承接。 |
| 卡片风格 | 使用轻量卡片、浅边框、适度圆角、清晰留白，不堆厚重阴影。 |
| 表格风格 | 表格要弱化，优先采用轻列表、分组列表、卡片化行，不做密集数据库表。 |
| 状态表达 | 状态 badge、短句摘要、风险提示要清楚，避免只显示英文枚举。 |
| 文案风格 | 面向用户的主链路文案中文优先，技术字段只在必要区域展示。 |
| 操作按钮 | 常驻按钮少而准，危险 / 低频操作放到更多菜单或确认弹窗。 |
| 技术细节 | runId、taskId、日志、仓库树、payload、诊断栈等默认收纳。 |
| 页面边界 | 每个页面只回答一个主问题，不重复展示其他页面的完整数据。 |

### 1.4 前端设计禁忌

后续 Codex 不能把页面改成以下形态：

- 一屏堆满统计卡片。
- 大面积传统表格铺满页面。
- 左中右三栏同时塞任务、运行、仓库三套重信息。
- 页面内再做复杂二级导航，和侧边栏重复。
- 把技术日志、仓库树、长 ID 默认展开。
- 用假按钮制造“已经自动执行 / 已经提交代码 / 已经发布”的错觉。
- 为了显得完整，把其他页面的数据缩略版复制到当前页面。


## 1.5 宽屏布局与有效空间利用原则

从当前前端截图可以看出，部分页面存在明显的“内容区过窄、右侧大面积空白、主内容没有充分利用宽屏”的问题。后续页面改造必须把这条写入整体风格：

> **页面可以轻量，但不能窄小；信息可以收纳，但不能让可用画布长期空着。**

### 1.5.1 当前问题判断

当前运行观测类页面的典型问题包括：

| 问题 | 影响 |
|---|---|
| 内容主体被限制在偏窄区域 | 宽屏下显得页面空、弱、像半成品。 |
| 左侧列表占用不大，但右侧详情没有充分展开 | 用户选择记录后仍看不到足够诊断信息。 |
| 空状态居中但周围大面积无信息 | 页面像调试状态，而不是正式后台。 |
| 顶部指标、列表、详情缺少宽屏网格关系 | 信息分布不均衡，视觉重心偏左。 |

### 1.5.2 后续布局要求

后续页面应采用“充分利用可用宽度 + 轻量信息密度”的布局，而不是把页面做窄。

| 布局规则 | 要求 |
|---|---|
| 主内容宽度 | 不要使用过窄固定 `max-width`；应根据侧边栏后的可用宽度自适应。 |
| 宽屏利用 | 在 1440px / 1600px / 更宽屏幕下，任务、运行、仓库、成果、治理页面应形成清晰的左右或 2x2 网格关系。 |
| 信息分布 | 左侧可放轻列表，右侧应承接摘要、诊断、态势或当前步骤面板，不能长期空白。 |
| 空状态 | 空状态要附带下一步说明和上下文按钮，不能只有一句“请选择”。 |
| 表格宽度 | 表格不应铺成密集数据库，但也不应窄到长字段折行；应采用轻列表 / 卡片化行。 |
| 详情承载 | 常驻详情只展示摘要；完整正文、日志、仓库树、证据链仍用弹窗 / 抽屉。 |
| 响应式 | 小屏可以单列；宽屏必须利用横向空间。 |

### 1.5.3 推荐宽屏结构

不同页面可按以下比例使用宽屏空间：

| 页面 | 推荐结构 |
|---|---|
| 工作台 | 主体对话区 + 右侧轻态势 / 弹窗入口。 |
| 项目页 | 左侧项目档案与计划摘要 + 右侧风险 / 最近事件 / 小状态卡。 |
| 任务队列 | 左侧轻量任务队列约 55%，右侧执行态势约 45%。 |
| 运行观测 | 左侧运行轻列表约 30%~35%，右侧诊断详情约 65%~70%。 |
| 仓库工作区 | 顶部状态条 + 横向步骤条 + 当前步骤面板，宽屏下当前步骤面板可拆成摘要区和证据区。 |
| 成果中心 | 左侧成果 / 审批轻列表，右侧摘要或决策面板。 |
| 治理中心 | 左侧资产轻列表，右侧治理摘要 / 消费证据。 |
| 设置页 | 顶部系统状态 + 2x2 设置卡片，编辑通过弹窗。 |

### 1.5.4 禁止的宽屏处理

后续 Codex 不允许：

- 把所有页面主体固定在很窄的中间容器里。
- 宽屏下右侧长期空白。
- 为了填满页面而重复展示其他页面的完整数据。
- 把长 ID、日志、仓库树直接铺满宽屏。
- 用三栏大屏同时塞任务、运行、仓库三套重信息。
- 只靠增加统计卡片填补空白。

正确做法是：**用合理的摘要面板、状态面板、当前步骤面板和弹窗入口填充宽屏，而不是堆信息。**

## 1.6 全局 AI 摘要机制声明

本次补充后，AI 摘要不再只属于运行页或成果中心，而是贯穿工作台、项目页、执行中心和成果中心的全局能力。

定稿原则：

```text
AI 产出内容以后，系统应在关键生命周期节点自动生成摘要并缓存。
用户进入页面时默认读取已生成摘要，不需要为了第一次理解结果而手动点击“生成摘要”等待。
用户手动按钮只用于重新生成、修正、展开、复制或生成更详细版本。
```

### 1.6.1 为什么要做自动摘要

当前项目目标是“AI 项目主管 / AI 自动调度 / 多 Agent 协作”。如果每个页面都要求用户手动点击生成摘要，用户会被迫等待模型调用，体验会变成半自动工具，而不是 AI 主管系统。

因此，摘要应跟随任务、运行、交付物、审批、计划变更等事件自动生成。页面只负责读取摘要、展示状态、提供重新生成入口。

### 1.6.2 全局摘要来源

| 来源 | 含义 | 页面展示要求 |
|---|---|---|
| `source=ai` | 真实模型生成 | 正常展示，并显示生成时间和基于对象。 |
| `source=rule_fallback` | Provider 失败、超时或未配置时由规则生成 | 必须明确标记，不能伪装成 AI 摘要。 |
| `source=manual_override` | 用户或人工审批意见覆盖 | 必须标明人工覆盖。 |
| `source=stale` | 摘要基于旧版本，可能已过期 | 必须提示可重新生成。 |

### 1.6.3 全局摘要触发点

| 触发事件 | 自动生成内容 | 主要展示页面 |
|---|---|---|
| Run 结束 | 运行摘要、失败分类说明、下一步建议 | 运行观测、任务队列、工作台 |
| Task 状态变化 | 任务当前结论、最近运行结果、是否需要人工、是否建议重新入队 | 任务队列、项目页、工作台 |
| Deliverable 生成 / 更新 | 交付物摘要、版本变化、验证结果、风险说明、是否建议送审 | 交付物、审批、项目页 |
| Approval request 创建 | 审批建议、证据充分性、通过 / 修改 / 驳回理由 | 审批、工作台待确认 |
| Project plan 变更 | 计划版本变化、新增 / 调整任务、风险变化、待确认事项 | 项目页、工作台 |
| Repository change request 评估完成 | 变更价值、影响文件、风险、验收建议 | 仓库工作区、任务队列 |

### 1.6.4 各页面摘要职责

| 页面 | 摘要职责 | 不能做什么 |
|---|---|---|
| 工作台 | 展示 AI 主管当前建议、待确认摘要、阻塞摘要 | 不现场生成所有页面摘要，不展示完整明细。 |
| 项目页 | 展示项目作战计划摘要、阶段摘要、风险摘要、最近事件摘要 | 不展示完整任务 / 运行 / 交付物列表。 |
| 任务队列 | 展示任务状态摘要、最近运行摘要、阻塞摘要 | 不展开完整运行日志。 |
| 运行观测 | 展示单次运行 AI 摘要和诊断摘要 | 不承担任务管理和审批决策。 |
| 仓库工作区 | 展示变更需求评估摘要、文件定位摘要、预检摘要、提交草案摘要 | 不伪装成真实 git commit / push。 |
| 交付物 | 展示交付摘要、验证摘要、证据摘要、风险摘要 | 不做审批决策。 |
| 审批 | 展示审批建议、证据充分性和决策后果 | 不管理完整成果库。 |

## 2. 全局导航方案

本阶段确认采用“收敛分组导航”。

```text
工作台 / 项目 / 执行中心 / 成果中心 / 治理 / 设置
```

### 2.1 导航职责

| 一级导航 | 职责 | 包含内容 |
|---|---|---|
| 工作台 | AI 项目主管主对接入口 | AI 主管对话、当前态势、待确认事项、轻量弹窗入口 |
| 项目 | 项目档案与 AI 作战计划总账本 | 项目目标、阶段、计划摘要、风险、最近事件 |
| 执行中心 | AI 执行过程 | 任务队列、运行观测、仓库工作区 |
| 成果中心 | 结果与放行 | 交付物、审批 |
| 治理 | Agent 能力与规则配置 | 角色、Skill、策略、记忆、成本、Agent 线程 |
| 设置 | 系统级技术配置 | Provider、模型、基础运行配置 |

### 2.2 为什么采用收敛分组导航

不采用以下长侧边栏：

```text
工作台 / 项目 / 任务 / 运行 / 仓库 / 交付物 / 审批 / 治理 / 设置
```

原因：

1. 入口过多，初次使用者会迷失。
2. 工作台和项目页容易被迫重复展示所有入口。
3. 页面会继续向“所有信息都在一个地方展示”的方向膨胀。
4. 不符合当前希望“页面简化、信息分工明确”的目标。

采用收敛分组后：

- 任务、运行、仓库统一进入“执行中心”。
- 交付物、审批统一进入“成果中心”。
- 角色、Skill、策略、记忆、成本统一进入“治理”。
- 工作台和项目页不再承担全局导航职责。

# 第一部分：工作台设计

---

## 3. 工作台最终定位

工作台正式定位为：

> AI 项目主管轻量指挥室。

工作台不再是浅层统计页，不再是任务列表页，也不再是运行诊断页。它是用户和 AI 项目主管的主对接入口。

用户进入系统后，默认面对 AI 项目主管，可以通过对话提出目标、询问进度、调整计划、处理阻塞、请求总结、发起重新规划。

AI 项目主管负责掌握当前项目计划、任务队列、Agent 编队、Skill 使用、运行结果、交付物和审批状态。它可以自动拆解任务、调整优先级、分配 Agent、总结运行反馈、识别阻塞并提出计划变更。

---

## 4. 工作台页面核心原则

### 4.1 只保留三个常驻区域

工作台页面本体只保留：

| 区域 | 作用 |
|---|---|
| AI 项目主管对话框 | 主入口，用户和 AI 主管沟通。 |
| 当前项目态势条 | 极简展示当前项目是否正常。 |
| 轻量入口区 | 作战计划、Agent 动向、项目流程、待确认、阻塞处理等弹窗入口。 |

### 4.2 不做信息大全页

工作台不常驻展示：

- 完整任务列表。
- 完整运行列表。
- 完整技术日志。
- 完整 Agent 配置。
- 完整 Skill 绑定。
- 完整交付物列表。
- 完整审批列表。
- 完整仓库树。
- 大量统计卡片。
- 多层折叠模块。

### 4.3 多用弹窗，不做多级模块

工作台允许放少量弹窗按钮，但不在页面本体展开复杂内容。

推荐弹窗：

| 弹窗 | 内容边界 |
|---|---|
| 作战计划 | 当前计划摘要、任务拆解摘要、计划变更建议。 |
| Agent 动向 | 各 Agent 当前状态、当前任务、最近反馈。 |
| 项目流程 | 可点击流程图，展示计划到交付审批的主流程。 |
| 待确认事项 | AI 主管汇总给用户确认的事项。 |
| 阻塞处理 | 当前阻塞原因和 AI 主管建议。 |

每个弹窗只处理一个问题，不能变成一个小型全功能页面。

## 5. 工作台建议布局

### 5.1 首屏布局草图

```text
┌──────────────────────────────────────────────────────────────┐
│ 工作台 / AI 项目主管                                          │
│ 当前项目：AI-Dev-Orchestrator    阶段：AI 辅助体验层          │
│ 状态：2 个进行中 / 1 个阻塞 / 1 个待确认                     │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  ┌──────────────────────────────────────────────────────┐    │
│  │ AI 项目主管对话框                                    │    │
```

## 6. 工作台弹窗设计

### 6.1 作战计划弹窗

用途：查看当前 AI 项目主管掌握的计划摘要。

展示内容：

- 当前计划版本。
- 当前阶段目标。
- 任务拆解摘要。
- 最近一次计划变更。
- AI 主管建议。
- 是否存在待用户确认的计划调整。

允许操作：

| 按钮 | 闭环要求 |
|---|---|
| 查看项目页 | 跳转项目页。 |
| 请求重新评估 | 生成计划评估建议，进入待确认，不直接应用。 |
| 查看计划变更 | 打开计划变更记录或项目页相关区域。 |

不允许：

- 在弹窗里编辑完整任务列表。
- 在弹窗里展开完整运行日志。
- 在弹窗里直接大规模改计划并自动应用。

### 6.2 Agent 动向弹窗

用途：让用户快速知道 Agent 正在做什么。

展示内容：

| 字段 | 说明 |
|---|---|
| Agent 名称 | 例如前端 Agent、后端 Agent、测试 Agent。 |
| 当前状态 | 空闲、执行中、等待、阻塞、失败。 |
| 当前任务 | 当前负责的任务标题。 |
| 最近反馈 | 最近一次运行或判断摘要。 |
| 风险提示 | 如果连续失败或等待人工，需要提示。 |

不允许：

- 展开完整 Agent 配置。
- 展开完整 Skill 绑定。
- 替代治理页。

### 6.3 项目流程弹窗

用途：展示当前项目从计划到交付审批的进度。

流程建议：

```text
目标输入 → 计划生成 → 角色编队 → 任务拆解 → 调度执行 → 运行验证 → 交付物 → 审批 / 放行
```

节点状态：

| 状态 | 含义 |
|---|---|
| 未开始 | 当前阶段还没进入。 |
| 进行中 | 当前正在处理。 |
| 已完成 | 已产生结果。 |
| 阻塞 | 需要用户或系统处理。 |
| 待确认 | 需要用户确认。 |
| 失败 | 需要重试或重新规划。 |

点击节点的行为：

| 节点 | 行为 |
|---|---|
| 计划生成 | 打开作战计划弹窗或项目页。 |
| 任务拆解 | 进入执行中心的任务队列。 |
| 调度执行 | 进入执行中心。 |
| 运行验证 | 进入执行中心的运行观测。 |
| 交付物 | 进入成果中心的交付物。 |
| 审批 / 放行 | 进入成果中心的审批。 |

### 6.4 待确认事项弹窗

用途：承接 AI 项目主管汇总给用户的决策点。

示例：

```text
待你确认：
1. AI 项目主管建议新增 2 个测试任务。
2. 前端 Agent 连续失败 2 次，建议切换为人工检查。
3. 当前交付物已生成，等待审批。
4. Provider 配置未测试，可能影响后续 AI 摘要。
```

按钮：

| 按钮 | 闭环要求 |
|---|---|
| 同意 | 真实写入确认结果或进入相应执行链路。 |
| 驳回 | 真实记录驳回意见。 |
| 修改后同意 | 用户补充意见后提交给 AI 项目主管重新判断。 |
| 查看证据 | 跳转运行、任务、交付物或审批页面。 |
| 稍后处理 | 只改变提醒状态，不丢失事项。 |

### 6.5 阻塞处理弹窗

用途：快速看到当前阻塞和 AI 主管建议。

展示内容：

- 阻塞任务数量。
- 阻塞原因分类。
- 关联任务。
- 最近失败运行。
- AI 主管建议。
- 下一步处理按钮。

按钮：

| 按钮 | 闭环要求 |
|---|---|
| 查看任务 | 跳转执行中心任务队列并定位任务。 |
| 查看运行证据 | 跳转执行中心运行观测并定位 run。 |
| 请求人工处理 | 调用真实任务人工处理接口。 |
| 生成重新规划建议 | 生成建议，进入待确认，不直接应用。 |

# 第二部分：项目页设计

---

## 9. 项目页最终定位

项目页正式定位为：

> 项目档案和 AI 作战计划总账本。

项目页不承担调度对话，不承担任务队列，不承担运行诊断，不承担仓库工作区，不承担交付物管理，不承担审批决策，不承担治理配置。

项目页只回答几个问题：

1. 这个项目是什么？
2. 当前项目目标是什么？
3. 当前项目处于什么阶段？
4. AI 项目主管当前作战计划是什么？
5. 当前项目有什么关键风险？
6. 最近发生了什么关键事件？
7. 是否需要用户进一步处理？

---

## 10. 项目页和工作台的区别

| 页面 | 核心问题 | 不做什么 |
|---|---|---|
| 工作台 | AI 项目主管现在怎么指挥？我现在要做什么？ | 不做完整项目档案，不做完整任务/运行/仓库展示。 |
| 项目页 | 这个项目是什么？计划是什么？进展到哪里？ | 不做对话调度，不做全局导航，不做其他页面缩略版。 |

工作台是动态指挥入口。  
项目页是稳定项目档案。

## 11. 项目页页面原则

### 11.1 不重复侧边栏

项目页不放以下入口矩阵：

```text
查看任务队列 / 查看运行观测 / 查看交付物 / 查看审批 / 查看仓库工作区 / 查看治理配置 / 查看项目时间线
```

原因：这会和全局侧边栏冲突，也会把项目页变成另一个导航页。

项目页只在出现具体状态或问题时，提供自然上下文跳转。例如：

```text
当前有 2 个阻塞任务。查看详情
最近一次运行失败。查看运行证据
1 个交付物等待审批。前往审批
```

这种跳转是问题驱动入口，不是导航栏。

### 11.2 不做其他页面缩略版

项目页不展示：

- 完整任务列表。
- 完整运行列表。
- 完整仓库工作区。
- 完整交付物列表。
- 完整审批列表。
- 完整治理配置。
- 完整项目时间线。

### 11.3 摘要是必要的

项目页需要保留摘要，因为用户要快速理解当前项目进展。

摘要包括：

- AI 作战计划摘要。
- 项目阶段摘要。
- 当前风险摘要。
- 进度摘要。
- 最近项目事件摘要。

但摘要不能变成完整功能区。

## 13. 项目页允许的操作

### 13.1 查看完整作战计划

行为：打开作战计划弹窗或进入作战计划详情。

内容：

- 当前计划版本。
- 计划目标。
- 任务拆解摘要。
- Agent 编队摘要。
- Skill 使用摘要。
- 风险与验收标准。
- 最近计划变更。

边界：

- 只查看计划，不在项目页直接大规模编辑任务。
- 完整任务管理仍属于执行中心任务队列。

### 13.2 请求 AI 重新评估计划

行为：用户给出建议，由 AI 项目主管判断价值，生成评估建议。

示例：

```text
我认为当前任务拆得太粗，请评估是否需要新增测试和文档审查任务。
```

AI 项目主管应该输出：

- 是否建议重新规划。
- 建议新增 / 修改 / 删除哪些任务。
- 影响范围。
- 风险。
- 是否需要用户确认。

重要边界：

> 项目页发起评估，不直接应用重新规划。  
> 评估结果应回到工作台的“待确认事项”，由用户最终决定是否应用。

### 13.3 查看完整时间线

行为：打开完整项目时间线。

边界：

- 项目页默认只显示最近 5 条事件。
- 完整时间线不常驻在项目页。
- 项目时间线不作为侧边栏一级页面。

### 13.4 更多操作

更多操作中可以放低频或危险操作，例如：

- 修改项目基础信息。
- 归档项目。
- 删除项目。
- 强制阶段推进。

危险操作必须二次确认，不能常驻首屏。

# 第三部分：执行中心设计

## 18. 执行中心最终定位

执行中心正式定位为：

> AI 项目主管调度后的执行工作区。

执行中心承接三类执行相关能力：

1. 任务队列
2. 运行观测
3. 仓库工作区

执行中心不是新的总览仪表盘，不重复工作台和项目页的信息。进入执行中心后，默认显示任务队列；用户通过顶部轻量页签切换到运行观测和仓库工作区。

### 18.1 执行中心和其他页面的边界

| 页面 | 核心问题 | 不做什么 |
|---|---|---|
| 工作台 | AI 项目主管现在怎么指挥？我现在要做什么？ | 不展示完整任务 / 运行 / 仓库。 |
| 项目页 | 项目是什么？作战计划是什么？项目进展到哪里？ | 不承担执行操作。 |
| 执行中心 | 任务如何调度？运行结果如何？仓库证据是否完整？ | 不做全局项目档案，不做成果审批。 |
| 成果中心 | 产出了什么？能否审批放行？ | 不做任务调度和运行诊断。 |
| 治理 | Agent、Skill、策略、记忆、成本如何配置和治理？ | 不做日常执行主入口。 |

执行中心三块的关系固定为：

```text
任务队列：要做什么
运行观测：做得怎么样
仓库工作区：基于哪些代码证据做
```

## 19. 执行中心整体布局

### 19.1 顶部结构

执行中心采用顶部轻量页签，不做复杂多级模块。

```text
执行中心
────────────────────────────────────────────
当前项目：AI-Dev-Orchestrator
执行状态：3 个待调度 / 2 个执行中 / 1 个阻塞 / 仓库快照正常

[任务队列] [运行观测] [仓库工作区]
```

默认页签：

```text
任务队列
```

### 19.2 执行中心不做事项

执行中心不应被做成以下形态：

- 新的执行概览大屏。
- 任务、运行、仓库三栏同时铺满。
- 统计卡片堆叠页。
- 工作台的复制版。
- 项目页的复制版。
- 仓库文件浏览器。
- 日志查看器。

执行中心常驻顶部只保留一行执行摘要，不铺大面积统计卡片。

## 20. 任务队列设计

任务队列是执行中心默认页，也是 AI 项目主管调度任务后的主视角。

### 20.1 任务队列定位

> 任务队列是 AI 项目主管拆出来的任务调度队列。

任务队列负责：

- 展示任务状态。
- 展示优先级。
- 展示依赖关系。
- 展示负责人 Agent。
- 展示阻塞原因。
- 展示任务是否可调度。
- 展示最近一次运行结果摘要。
- 提供任务级真实操作：暂停、恢复、请求人工、人工已处理、重新入队。

任务队列不负责：

- 展示完整运行日志。
- 展示完整运行诊断。
- 展示仓库目录树。
- 展示交付物正文。
- 展示审批表单。
- 展示 Agent / Skill 完整配置。

### 20.2 任务队列布局

任务队列采用“轻列表 + 执行态势面板 + 详情抽屉”。

```text
┌──────────────────────────────────────────────────────────────┐
│ 执行中心 / 任务队列                                           │
│ 当前项目：AI-Dev-Orchestrator                                 │
│ 3 个待调度 / 2 个执行中 / 1 个阻塞 / 1 个待人工                │
├───────────────────────────────┬──────────────────────────────┤
│ 轻量任务队列                    │ 执行态势面板                  │
│                               │                              │
│ 需要你处理                     │ Agent 负载                    │
│ - T-18 测试任务失败 2 次        │ 阻塞原因分布                  │
│                               │ 最近运行结果                  │
│ 执行中                         │ AI 主管下一步建议             │
│ - T-15 前端日志弹窗适配         │                              │
│                               │                              │
│ 可调度                         │                              │
│ - T-16 Provider 测试接口        │                              │
│                               │                              │
│ 等待依赖                       │                              │
│ - T-17 回归验证任务             │                              │
└───────────────────────────────┴──────────────────────────────┘
```

页面比例建议：

| 区域 | 比例 | 说明 |
|---|---:|---|
| 轻量任务队列 | 55% | 展示调度队列，不做传统大表格。 |
| 执行态势面板 | 45% | 展示 Agent 负载、阻塞原因、最近运行、AI 建议。 |

### 20.3 任务排列规则

任务不按创建时间简单排列，固定按调度优先级分组：

```text
1. 待人工 / 阻塞
2. 执行中
3. 可调度
4. 等待依赖
5. 已完成 / 已关闭（默认折叠）
```

原因：用户进入执行中心首先要看到“哪里需要处理”，而不是看到一张按 ID 排序的任务表。

### 20.4 任务行展示字段

每个任务行只常驻展示 6 个信息：

| 字段 | 说明 |
|---|---|
| 任务标题 | 清楚说明任务目标。 |
| 状态 | pending / running / blocked / waiting_human / completed 等要中文化展示。 |
| 负责人 Agent | 显示负责角色，例如前端 Agent、后端 Agent、测试 Agent。 |
| 优先级 | P0 / P1 / P2 或等价优先级。 |
| 阻塞 / 依赖 | 只显示关键原因，不展开完整依赖树。 |
| 最近运行结果 | succeeded / failed / blocked 的一句摘要。 |

示例：

```text
T-15  运行页日志弹窗前端适配
前端 Agent · 执行中 · P1 · 最近运行 succeeded
[详情] [查看运行] [更多]
```

### 20.5 执行态势面板

右侧不放任务详情，放任务队列级态势。

常驻内容：

#### A. Agent 当前负载

```text
前端 Agent：执行中 1 / 待调度 2 / 失败 1
后端 Agent：待调度 1
测试 Agent：等待依赖 2
文档 Agent：空闲
```

#### B. 阻塞原因分布

```text
等待人工：1
依赖未完成：2
运行失败：1
Provider 未验证：1
```

#### C. 最近运行结果

只显示最近 3 条：

```text
R-28 succeeded：运行摘要生成成功
R-27 failed：技术日志接口异常
R-26 blocked：等待人工确认
```

点击进入运行观测并定位对应 run。

#### D. AI 项目主管下一步建议

```text
AI 主管建议：先处理 T-18 的测试阻塞，再继续调度前端 Agent。
```

注意：这里是任务队列级建议，不是工作台全局建议。

### 20.6 任务详情抽屉

点击任务后打开右侧抽屉，不做常驻大右栏。

抽屉展示：

- 任务目标。
- 验收标准。
- 负责人 Agent。
- 依赖关系。
- 当前状态。
- 阻塞原因。
- 最近一次运行摘要。
- 任务级操作按钮。

抽屉不展示：

- 完整运行日志。
- 完整仓库树。
- 交付物正文。
- 审批表单。

### 20.7 任务队列按钮闭环

| 按钮 | 闭环要求 |
|---|---|
| 详情 | 打开任务详情抽屉。 |
| 暂停 | 调用真实 pause 接口。 |
| 恢复 | 调用真实 resume 接口。 |
| 请求人工 | 调用真实 request-human 接口。 |
| 人工已处理 | 调用真实 resolve-human 接口。 |
| 重新入队 | 调用真实 retry 接口，但文案必须说明“等待下一次调度”。 |
| 查看运行 | 跳转运行观测并定位 run。 |
| 查看仓库上下文 | 跳转仓库工作区并带 taskId / projectId。 |

不建议常驻以下按钮：

- 立即执行单个任务。
- 自动修复。
- 生成代码。
- 一键完成。

除非后端有明确真实闭环，否则不能展示这些动作按钮。

## 21. 运行观测设计

运行观测是执行中心的第二个页签。

### 21.1 运行观测定位

> 运行观测是单次执行的诊断证据页。

它回答：

1. 这一次执行发生了什么？
2. 为什么成功 / 失败 / 阻塞？
3. 有什么证据？
4. 下一步应该去哪里处理？

它不承担任务队列管理，不承担仓库变更，不承担交付物审批。

### 21.2 运行观测布局

运行观测采用“左侧运行轻列表 + 右侧诊断详情”。

```text
┌──────────────────────────────────────────────────────────────┐
│ 执行中心 / 运行观测                                           │
│ 当前项目 / 状态筛选 / Agent 筛选 / 任务筛选                   │
├───────────────────┬──────────────────────────────────────────┤
│ 运行轻列表          │ 运行诊断详情                              │
│                   │                                          │
│ R-28 succeeded    │ AI 运行摘要                               │
│ 前端 Agent · T-15 │ 失败原因 / 质量闸门                         │
│                   │ 验证结果 / token / cost                     │
│ R-27 failed       │ 技术日志按钮                               │
│ 后端 Agent · T-16 │                                          │
│                   │                                          │
│ R-26 blocked      │                                          │
│ 测试 Agent · T-18 │                                          │
└───────────────────┴──────────────────────────────────────────┘
```

左侧约 32%，右侧约 68%。

### 21.3 运行轻列表

左侧不是大表格，而是时间线式轻列表。

每条展示：

- run 状态。
- Agent。
- 关联任务。
- 时间。
- 摘要短句。

示例：

```text
R-28  succeeded
前端 Agent · T-15 · 2 分钟前
AI 摘要生成成功

R-27  failed
后端 Agent · T-16 · 10 分钟前
Provider 测试接口返回异常
```

### 21.4 运行诊断详情

右侧分为四块。

#### A. AI 摘要

给普通用户看的第一层摘要。

```text
这次运行已完成。AI 成功生成运行摘要，并通过基础质量检查。
```

#### B. 诊断结果

包括：

- 运行状态。
- 失败分类。
- 质量闸门。
- 验证摘要。
- 重试建议。

#### C. 证据字段

默认折叠或弱化展示：

- 关联任务。
- Agent。
- 模型。
- source=ai / fallback。
- token。
- cost。
- 耗时。
- runId / taskId。

#### D. 技术日志弹窗

页面只常驻一个按钮：

```text
查看技术日志
```

点击后打开弹窗，展示：

- 结构化日志。
- stdout / stderr。
- 错误堆栈。
- 复制日志。

技术日志不默认铺在页面上。

### 21.5 运行观测按钮闭环

| 按钮 | 闭环要求 |
|---|---|
| 查看技术日志 | 打开真实 run log 弹窗。 |
| 复制日志 | 真实复制当前日志内容。 |
| 查看任务 | 跳转任务队列并定位任务。 |
| 重新入队 | 调用关联任务 retry，但说明等待下一次调度。 |
| 查看交付物 | 跳转成果中心交付物并定位。 |
| 前往审批 | 跳转成果中心审批并定位。 |

运行观测不提供无后端支撑的“立即重跑”“自动修复”“自动提交代码”等按钮。

## 22. 仓库工作区设计

仓库工作区是执行中心的第三个页签，也是内容最重的页签。必须用步骤化布局控制复杂度。

### 22.1 仓库工作区定位

> 仓库工作区是代码变更需求、上下文证据和提交草案的受控工作区。

仓库工作区不是在线 IDE，不是文件管理器，也不是项目页的一部分。它负责把用户的仓库变更需求交给 AI 项目主管评估，再形成受控变更链路。

### 22.2 核心边界

当前仓库链路必须明确：

```text
当前主要是 read-only + 提交草案，不是真实 git commit / push。
```

因此页面上不能出现会误导用户的按钮：

- 直接提交代码。
- 推送分支。
- 合并 PR。
- 发布生产。

除非后端补齐真实写入能力，并通过审批和验证闭环。

### 22.3 仓库工作区主流程

原始流程：

```text
仓库绑定 → 快照 → 文件定位 → 上下文包 → 变更批次 → 预检 → 提交草案 → 放行判断
```

升级为：

```text
变更需求 → AI 主管评估 → 文件定位 → 上下文包 → 变更方案 → 变更批次 → 预检 → 提交草案 → 放行判断
```

新增前两步的目的：

- 用户不是直接编辑文件，而是提出仓库变更需求。
- AI 项目主管先判断需求价值、影响范围、风险和是否值得改。
- 用户根据 AI 评估决定接受、修改后接受、驳回或暂存。

### 22.4 仓库工作区整体布局

```text
┌──────────────────────────────────────────────────────────────┐
│ 仓库工作区                                                     │
│ 当前项目：AI-Dev-Orchestrator                                  │
│ 仓库：已绑定 ｜ 快照：正常 ｜ 当前变更：待评估 ｜ 草案：未生成     │
│                                  [提出变更需求] [刷新快照]      │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  ● 变更需求  →  ◐ AI 主管评估  →  ○ 文件定位  →  ○ 上下文包      │
│                                              ↓               │
│  ○ 变更方案  →  ○ 变更批次  →  ○ 预检  →  ○ 提交草案  →  ○ 放行判断 │
│                                                              │
├──────────────────────────────────────────────────────────────┤
│ 当前步骤：AI 主管评估                                          │
│                                                              │
│ ┌──────────────────────────────────────────────────────────┐ │
│ │ 用户需求：                                                │ │
│ │ 优化运行页技术日志弹窗，不要默认铺日志。                  │ │
│ │                                                          │ │
│ │ AI 判断：建议执行                                         │ │
│ │ 风险等级：中低                                            │ │
│ │ 涉及范围：运行观测页，不影响任务页                         │ │
│ │                                                          │ │
│ │ 推荐文件：                                                │ │
│ │ - RunsPage.tsx                                            │ │
│ │ - RunLogModal.tsx                                         │ │
│ │                                                          │ │
│ │ 建议验收：                                                │ │
│ │ - 日志默认隐藏                                            │ │
│ │ - 点击按钮打开弹窗                                        │ │
│ │ - 支持复制日志                                            │ │
│ │ - apps/web build 通过                                     │ │
│ │                                                          │ │
│ │ [接受方案] [修改需求] [驳回] [生成上下文包]                 │ │
│ └──────────────────────────────────────────────────────────┘ │
└──────────────────────────────────────────────────────────────┘
```

### 22.5 顶部仓库状态条

顶部只显示 4 个核心状态：

```text
仓库：已绑定 ｜ 快照：正常 ｜ 当前变更：待评估 ｜ 草案：未生成
```

右侧按钮：

```text
[提出变更需求] [刷新快照] [查看绑定]
```

不常驻展示文件总数、目录总数、语言比例、完整路径、所有扫描记录。

### 22.6 变更链路步骤条

步骤条是仓库工作区的核心视觉。

```text
● 变更需求 → ◐ AI 主管评估 → ○ 文件定位 → ○ 上下文包
                                      ↓
○ 变更方案 → ○ 变更批次 → ○ 预检 → ○ 提交草案 → ○ 放行判断
```

状态含义：

| 符号 | 含义 |
|---|---|
| ● | 已完成 |
| ◐ | 进行中 |
| ○ | 未开始 |
| ✕ | 阻塞 |

如果预检失败：

```text
● 变更需求 → ● AI 主管评估 → ● 文件定位 → ● 上下文包
                                      ↓
● 变更方案 → ● 变更批次 → ✕ 预检 → ○ 提交草案 → ○ 放行判断
```

用户马上知道代码链路卡在哪一步。

### 22.7 提出仓库变更需求弹窗

顶部主按钮：

```text
[提出变更需求]
```


这是需求级输入，不是直接编辑文件。

### 22.9 仓库工作区美观原则

仓库工作区不能把所有能力堆成一屏。

必须坚持：

1. 仓库树默认隐藏到弹窗 / 抽屉。
2. 步骤条代替模块堆叠。
3. 每次只展示当前步骤。
4. 文件定位结果用卡片，不用传统大表格。
5. 技术细节默认折叠。
6. 提交草案必须明确不是 git commit / push。
7. 用户可以提出变更需求，但不能绕过 AI 主管直接编辑并生效。

### 22.10 仓库工作区按钮闭环

| 按钮 | 闭环要求 |
|---|---|
| 提出变更需求 | 创建真实变更需求或 AI 主管评估请求。 |
| 提交给 AI 主管评估 | 必须生成结构化评估结果，不只是前端假文案。 |
| 接受方案 | 进入下一步变更链路，记录用户确认。 |
| 修改需求 | 回到变更需求弹窗，保留上下文。 |
| 驳回 | 记录驳回原因。 |
| 开始定位 | 调用真实文件定位接口。 |
| 加入上下文包 | 写入真实上下文包选择。 |
| 生成上下文包 | 调用真实 context pack 能力。 |
| 执行预检 | 调用真实 preflight 能力。 |
| 生成提交草案 | 调用真实 commit candidate 草案能力。 |
| 送审 | 创建或跳转真实审批链路。 |
| 前往审批 | 跳转成果中心审批并带上下文。 |

## 24. 执行中心当前定稿结论

```text
执行中心是 AI 项目主管调度后的执行工作区，由任务队列、运行观测、仓库工作区三个同级页面组成。

执行中心默认进入任务队列。任务队列采用轻表格化列表，不做传统大表格，不做常驻任务详情大右栏。任务按调度优先级分组：待人工 / 阻塞、执行中、可调度、等待依赖、已完成。右侧展示任务级执行态势，包括 Agent 负载、阻塞原因、最近运行和 AI 主管建议。

运行观测采用运行轻列表 + 诊断详情布局，核心展示 AI 摘要、失败分类、质量闸门、验证摘要和技术日志弹窗，不承担任务管理职责。

仓库工作区采用仓库状态条 + 变更链路步骤条 + 单步骤工作面板布局。仓库树、技术细节和大块证据通过弹窗或抽屉展示，页面常驻信息保持轻量。仓库页允许用户向 AI 项目主管提出仓库变更需求，但不能直接编辑文件并生效。提交草案必须明确为审阅草案，不伪装成真实 git commit / push。
```

## 25. 成果中心最终定位

成果中心是执行中心的下游页面组，用于承接 AI 项目主管、任务队列、运行观测和仓库工作区产生的正式结果。

成果中心不是新的总览仪表盘，也不是交付物和审批的混合大页面。它只承担两个职责：

1. 管理 AI 产出的正式成果。
2. 承接用户对成果的人工 Gate 决策。

最终定义：

```text
成果中心是 AI 项目执行结果的交付与审批工作区，由交付物和审批两个同级页面组成。

交付物页负责管理 AI 产出的正式成果，包括成果状态、版本、来源任务、来源运行、关联仓库变更、AI 交付摘要、验证状态和证据链。

审批页负责处理需要用户决策的 Gate，包括通过、要求修改、驳回、查看证据和记录审批意见。

成果中心不承担任务调度、运行诊断、仓库上下文、治理配置和全局对话职责。交付物页看成果，审批页做决定。所有高风险放行动作必须说明后果并保留证据。
```

---

## 26. 成果中心整体结构

成果中心采用两个同级页签：

```text
成果中心
------------------------------------------------
[交付物] [审批]
```

默认打开：

```text
交付物
```

原因：审批必须依附于具体成果，先有交付物，再有审批请求。

### 26.1 不做成果总览页

成果中心不额外做“成果总览”。

原因：

- 工作台已经承接全局态势。
- 项目页已经承接项目进展摘要。
- 成果中心如果再做统计总览，会变成第三个概览页。
- 用户进入成果中心时，主要目的是看成果或做审批，不是再看一套卡片统计。

### 26.2 成果中心只保留最小状态条

顶部只允许保留一行轻量状态，例如：

```text
当前项目：AI-Dev-Orchestrator ｜ 交付物：3 个 ｜ 待审批：1 个 ｜ 需要返工：1 个
```

不允许在成果中心顶部堆放大量统计卡片。

## 27. 交付物页设计

### 27.1 交付物页定位

交付物页是 AI 产出成果的正式仓库。

它回答以下问题：

```text
AI 到底交付了什么？
这个成果来自哪个任务 / 运行 / 仓库变更？
当前成果是草稿、待审、已通过，还是需要返工？
是否有版本记录？
是否有证据链？
是否建议送审？
```

交付物页不回答：

```text
任务为什么失败？
运行日志是什么？
仓库文件在哪里？
审批是否通过？
Agent 和 Skill 怎么配置？
```

这些问题分别进入运行观测、仓库工作区、审批页和治理页。

### 27.2 交付物页布局

交付物页采用：

```text
左侧：交付物轻列表
右侧：交付物摘要面板
正文 / 证据 / 版本：弹窗或抽屉
```

示意：

```text
┌────────────────────────────────────────────────────────────┐
│ 成果中心 / 交付物                                           │
│ 当前项目：AI-Dev-Orchestrator ｜ 3 个成果 ｜ 1 个待审           │
├──────────────────────┬─────────────────────────────────────┤
│ 交付物轻列表           │ 交付物摘要                            │
│                      │                                     │
│ D-01 运行摘要设计文档   │ 标题：运行摘要设计文档                 │
│      待审 · v2         │ 状态：待审批                           │
│      来源：T-15 / R-28 │ 版本：v2                               │
│                      │ 来源任务：T-15                          │
│ D-02 技术日志弹窗方案   │ 来源运行：R-28                          │
│      草稿 · v1         │ 关联仓库变更：CB-03                     │
│                      │ 验证：build 通过 / 预检通过              │
│ D-03 提交草案           │                                     │
│      需要返工 · v1      │ AI 交付摘要：                          │
│                      │ 本成果完成运行页技术日志弹窗方案收口，     │
│                      │ 证据链基本完整，建议进入审批。             │
│                      │                                     │
│                      │ [查看正文] [查看证据] [版本记录] [发起审批] │
└──────────────────────┴─────────────────────────────────────┘
```

### 27.3 交付物排序规则

交付物列表不按创建时间简单排序，而按用户处理优先级排序：

```text
1. 待审批
2. 需要返工
3. 草稿
4. 已通过
5. 已归档
```

用户进入交付物页时，首先看到需要处理的成果，而不是历史记录。

### 27.4 交付物轻列表展示字段

每个交付物条目只展示：

```text
交付物标题
状态
版本
来源任务 / 来源运行
最近处理结果
```

不展示完整正文、不展示完整运行日志、不展示完整证据包。

### 27.5 交付物摘要面板展示字段

右侧摘要面板只展示：

```text
标题
状态
版本
来源任务
来源运行
关联仓库变更
AI 交付摘要
验证状态
审批状态
主要操作
```

正文、证据链、版本记录必须通过弹窗或抽屉承接。

### 27.6 交付物弹窗

交付物页允许以下弹窗：

| 弹窗 | 内容 |
|---|---|
| 查看正文 | 展示交付物正文或生成结果。 |
| 查看证据 | 展示来源任务、来源运行、仓库变更、预检、build/test 证据。 |
| 版本记录 | 展示交付物历史版本和差异摘要。 |
| 发起审批 | 创建审批请求前的确认弹窗。 |
| 要求返工 | 填写返工原因，并进入任务 / AI 主管确认链路。 |

正文、证据、版本记录不得常驻铺在页面主体里。

### 27.7 交付物按钮闭环

| 按钮 | 闭环要求 |
|---|---|
| 查看正文 | 打开真实交付物内容弹窗。 |
| 查看证据 | 打开真实证据链弹窗。 |
| 版本记录 | 打开真实版本记录。 |
| 发起审批 | 创建真实 approval request。 |
| 要求返工 | 创建返工请求或进入 AI 主管待确认，不是假按钮。 |
| 查看来源运行 | 跳转运行观测并定位 runId。 |
| 查看来源任务 | 跳转任务队列并定位 taskId。 |
| 重新生成摘要 | 调用真实摘要生成接口，生成后缓存。 |

不允许常驻以下按钮：

| 禁止按钮 | 原因 |
|---|---|
| 一键发布 | 当前没有真实发布闭环。 |
| 直接通过 | 通过属于审批页职责。 |
| 删除交付物 | 高风险动作，不应常驻。 |
| 自动修改 | 应回到任务 / 仓库变更流程。 |

## 28. 审批页设计

### 28.1 审批页定位

审批页是用户对 AI 产出进行人工 Gate 决策的地方。

它回答以下问题：

```text
现在有什么需要我决定？
AI 建议通过、要求修改，还是驳回？
证据是否充分？
通过以后会发生什么？
要求修改以后会发生什么？
驳回以后会发生什么？
```

审批页不管理完整成果库，也不展示完整运行日志。

### 28.2 审批页布局

审批页采用：

```text
左侧：待审批轻列表
```

```text
│ A-02 提交草案           │ 证据充分性：基本充分                   │
```

### 28.3 审批列表排序规则

审批列表按处理优先级排列：

```text
1. 待我审批
2. 需要补证据
3. AI 建议驳回
4. 已通过
5. 已驳回
```

用户进入审批页后，应优先看到需要自己决策的事项。

### 28.4 审批决策面板展示字段

审批决策面板只展示：

```text
审批对象
来源交付物
来源任务 / 来源运行
AI 审批建议
风险等级
证据充分性
通过后果
要求修改后果
驳回后果
审批意见输入框
决策按钮
```

### 28.5 审批按钮必须说明后果

审批按钮不能只显示“通过 / 驳回”。

每个按钮必须让用户知道后果：

```text
通过：该交付物版本标记为已通过，可进入下一阶段。
要求修改：生成返工请求，回到任务队列或 AI 主管待确认。
驳回：关闭本次交付物版本，保留审批证据和意见。
```

高风险动作必须二次确认。

### 28.6 审批按钮闭环

| 按钮 | 闭环要求 |
|---|---|
| 通过 | 写入真实审批决策。 |
| 要求修改 | 写入真实审批决策，并创建返工请求或回到任务队列。 |
| 驳回 | 写入真实审批决策，关闭当前版本或标记驳回。 |
| 查看证据 | 打开真实证据链。 |
| 查看交付物 | 跳转或弹窗展示对应交付物。 |
| 查看来源运行 | 跳转运行观测并定位 runId。 |
| 查看来源任务 | 跳转任务队列并定位 taskId。 |
| 重新评估 | 调用真实 AI 审批建议生成接口，不能只改前端文案。 |

## 29. 全局自动摘要机制（覆盖工作台 / 项目页 / 执行中心 / 成果中心）

### 29.1 自动摘要原则

本节不是成果中心专属规则，而是覆盖工作台、项目页、执行中心和成果中心的全局摘要机制。

摘要不应该主要依赖用户手动点击生成。

定稿原则：

```text
AI 产出内容以后，系统应在关键生命周期节点自动生成摘要并缓存。
用户进入页面时默认读取已生成摘要，不需要手动点击生成并等待。
用户手动按钮只用于重新生成、修正、展开、复制或生成更详细版本。
```

### 29.2 摘要由谁生成

摘要不是由用户写，也不是所有摘要都由单个执行 Agent 自说自话。

建议分层：

| 摘要类型 | 生成者 | 校正 / 归纳者 |
|---|---|---|
| 运行摘要 | Run Summary 服务在 run 结束后自动生成 | AI 项目主管读取并归纳 |
| 任务摘要 | 任务状态变化后自动生成 | AI 项目主管归纳任务状态 |
| 交付物摘要 | 交付物版本生成时自动生成 | AI 项目主管 / 审查 Agent 判断质量 |
| 证据摘要 | 交付物生成或审批请求创建时自动生成 | 审查 Agent / AI 项目主管判断证据充分性 |
| 审批建议 | 审批请求创建时自动生成 | AI 项目主管给通过 / 修改 / 驳回建议 |
| 项目摘要 | 阶段变化、计划变化、关键事件后生成 | AI 项目主管维护总摘要 |
| 工作台态势摘要 | 根据项目关键事件更新缓存 | AI 项目主管汇总当前建议 |

一句话：

```text
执行 Agent 产生事实，AI 项目主管负责总结、判断和给建议。
```

### 29.3 自动摘要触发点

摘要按事件触发，不按页面打开触发。

#### Run 结束时

触发：

```text
运行摘要
失败分类说明
下一步建议
```

适用状态：

```text
succeeded / failed / blocked / cancelled
```

#### Task 状态变化时

当任务变为以下状态时，触发任务摘要更新：

```text
completed / failed / waiting_human / blocked / paused
```

生成：

```text
任务当前结论
最近运行结果
是否需要人工
是否建议重新入队
```

#### Deliverable 生成或更新时

触发交付物摘要：

```text
成果说明
版本变化
来源任务
来源运行
验证结果
风险说明
是否建议送审
```

#### Approval request 创建时

触发审批建议：

```text
AI 建议：通过 / 要求修改 / 驳回
理由
证据充分性
通过后的后果
驳回后的建议
```

#### Project plan 变更时

触发项目摘要：

```text
计划版本变化
新增任务
删除 / 调整任务
风险变化
需要用户确认的事项
```

### 29.4 自动摘要缓存规则

摘要必须保存为可读取的结果，不应每次打开页面都重新调用模型。

要求：

```text
1. 自动摘要生成后写入数据库或 runtime data 中的明确存储位置。
2. 页面默认读取缓存摘要。
3. 页面展示摘要生成时间和来源。
4. 相关对象更新后，摘要标记为可能过期。
5. 用户可以手动重新生成，但不能反复无提示消耗模型。
```

### 29.5 摘要来源标记

每条摘要都必须有来源标记。

建议状态：

| 来源 | 含义 |
|---|---|
| source=ai | 由真实模型生成。 |
| source=rule_fallback | Provider 失败或无配置时由规则生成。 |
| source=manual_override | 用户或人工审批意见覆盖。 |
| source=stale | 摘要基于旧版本，可能已过期。 |

页面必须展示来源，不允许让用户误以为所有摘要都是 AI 实时生成。

### 29.6 fallback 机制

必须支持 fallback。

当 DeepSeek / OpenAI Provider 调用失败、超时或未配置时，页面不能空白。

应生成规则摘要，例如：

```text
该交付物来自任务 T-15，当前状态为待审批，关联运行 R-28，验证结果未记录。
```

并标记：

```text
source=rule_fallback
```

Provider 恢复后，允许用户点击：

```text
重新生成 AI 摘要
```

### 29.7 摘要状态展示

页面应展示摘要状态，例如：

```text
AI 摘要：已生成
来源：source=ai
生成时间：2026-05-18 20:42
基于：Run R-28 / Task T-15 / Deliverable D-03
```

失败时展示：

```text
AI 摘要生成失败，已使用规则摘要。
原因：Provider 调用失败。
[重新生成]
```

过期时展示：

```text
摘要可能已过期：交付物版本已更新。
[重新生成摘要]
```

### 29.8 对前面页面设计的修订

由于自动摘要是全局机制，前面已经定稿的页面需要同步调整，避免把摘要生成设计成用户手动等待。

#### 工作台修订

工作台的“当前项目态势条”“AI 主管当前建议”“待确认事项”“阻塞处理”应读取已缓存的项目态势摘要和主管建议。

工作台允许快捷指令：

```text
总结进度 / 生成汇报 / 重新评估计划
```

但这些按钮不应承担第一次摘要生成。第一次摘要应在项目关键事件发生后自动生成。

#### 项目页修订

项目页的 AI 作战计划摘要、阶段摘要、风险摘要、最近事件摘要，应在计划变更、阶段变化、关键任务状态变化后自动更新。

项目页可以提供：

```text
重新生成项目摘要
查看摘要来源
查看计划变更详情
```

但不能让用户进入项目页后才手动生成第一版项目摘要。

#### 任务队列修订

任务行里的“最近运行摘要”“阻塞摘要”“下一步建议”应来自任务状态变化和 run 结束后的自动摘要，不由任务列表前端临时拼接。

任务详情抽屉可以展示摘要状态：

```text
摘要来源：source=ai / source=rule_fallback
摘要时间：xxxx
基于：Run R-xx / Task T-xx
```

#### 运行观测修订

运行观测页默认读取 run 结束时生成的 AI 摘要。

如果 `source=rule_fallback`，应明确提示：

```text
真实 AI 摘要生成失败，当前为规则摘要。
```

用户可以重新生成，但不应每次打开运行页都重新调用模型。

#### 仓库工作区修订

仓库变更需求提交后，应自动生成 AI 主管评估摘要，包括：

```text
需求价值
涉及文件
风险等级
建议验收
是否建议进入变更批次
```

文件定位、上下文包、预检、提交草案都应有对应摘要，不要求用户手动等待第一次总结。

#### 成果中心修订

交付物摘要、证据摘要、风险摘要、审批建议应在交付物版本生成或审批请求创建时自动生成并缓存。

交付物页和审批页只提供重新生成、修正、复制、展开功能。

---

### 29.9 成果中心按钮调整

交付物页不应默认展示：

```text
[生成 AI 摘要]
```

应展示：

```text
AI 摘要已生成
[重新生成摘要]
[查看详细摘要]
[复制摘要]
[修正摘要]
```

审批页不应默认展示：

```text
[生成审批建议]
```

应展示：

```text
AI 审批建议已生成
[重新评估]
[查看证据]
[通过]
[要求修改]
[驳回]
```

## 30. 成果中心信息不重复规则

成果中心必须避免和执行中心、项目页、工作台重复。

| 信息 | 成果中心怎么展示 | 详情去哪里 |
|---|---|---|
| 任务 | 只展示来源任务和跳转 | 任务队列 |
| 运行 | 只展示来源运行和摘要 | 运行观测 |
| 仓库变更 | 只展示关联批次 / 草案摘要 | 仓库工作区 |
| 项目状态 | 只展示当前项目轻状态 | 项目页 |
| Agent 状态 | 不展示完整 Agent 面板 | 工作台 / 治理 |
| 审批 | 只在审批页做决策 | 审批页 |
| 交付物正文 | 弹窗展示 | 交付物正文弹窗 |
| 证据链 | 弹窗展示摘要和证据明细 | 证据弹窗 / 来源页面 |

核心边界：

```text
交付物页：看成果。
审批页：做决定。
执行中心：看怎么做。
项目页：看项目总账。
工作台：和 AI 项目主管对接。
```

## 31. 成果中心不做事项

成果中心不做以下内容：

- 不做成果总览大屏。
- 不展示完整任务队列。
- 不展示完整运行日志。
- 不展示仓库目录树。
- 不展示完整 Agent 配置。
- 不展示 Skill 绑定编辑。
- 不把审批按钮放到交付物页直接完成审批。
- 不把交付物正文铺满页面。
- 不让用户为了第一次理解成果而手动等待摘要生成。
- 不做一键发布、真实发布、git push、合并 PR 等未闭环危险动作。

## 32. 成果中心当前定稿结论

```text
成果中心由交付物和审批两个同级页面组成，默认进入交付物。

交付物页采用交付物轻列表 + 交付物摘要面板布局，正文、证据链、版本记录全部通过弹窗或抽屉展示。

审批页采用待审批轻列表 + 审批决策面板布局，所有审批按钮必须说明后果并写入真实审批决策。

成果中心的摘要应在运行完成、任务状态变化、交付物版本生成、审批请求创建等关键节点自动生成并缓存。用户进入页面时默认读取摘要，不应手动等待第一次摘要生成。Provider 失败时使用 rule_fallback 摘要，并明确标记来源。
```

# 第五部分：治理中心设计

## 33. 治理中心最终定位

治理中心不是普通设置页，也不是当前项目的简单配置表。

治理中心应升级为：

> **AI 团队资产治理中心。**

它负责管理 AI 项目主管系统中的可复用团队资产与运行规则，包括：

| 治理对象 | 说明 |
|---|---|
| 本项目 AI 团队 | 当前项目主管生成的 Agent 编队、角色实例、职责与绑定 Skill |
| 角色治理 | 可复用角色模板、项目角色实例、角色沉淀与角色版本演化 |
| Skill 治理 | 正式 Skill、项目临时 Skill、Skill 迭代候选、Skill 版本与消费证据 |
| 策略与权限 | AI 项目主管和 Agent 的自动执行边界、用户确认边界、禁止动作 |
| 成本与记忆 | AI 生成预算、模型分级、摘要缓存、强模型调用理由、记忆治理、成本可信度 |
| 治理审计 | 谁修改了角色、Skill、策略、预算，哪些治理变更被运行消费 |

治理中心不承担：

```text
项目调度
任务执行
运行诊断
仓库变更
交付物审批
Provider 基础配置
全局 AI 项目主管对话
```

治理中心的职责是：

```text
让 AI 团队能力越用越好，同时越用越省。
```

---

## 34. 治理中心和设置页的边界

设置页和治理中心必须分开。

| 页面 | 管理对象 | 典型内容 |
|---|---|---|
| 设置页 | 系统连接和基础配置 | Provider、API Key、Base URL、Timeout、系统级连接配置 |
| 治理中心 | AI 团队资产和运行规则 | Agent、角色、Skill、权限、预算、记忆、成本、审计 |

硬边界：

```text
Provider Key 不进入治理中心。
AI 角色和 Skill 不进入设置页。
成本策略和模型分级可以放治理中心。
API Key、Base URL、连接测试放设置页。
```

---

## 35. 治理中心整体布局

治理中心采用轻量区块，不做大面积配置表，不做复杂多级菜单。

建议布局：

```text
┌────────────────────────────────────────────────────────────┐
│ 治理中心                                                     │
│ 当前项目：AI-Dev-Orchestrator                                │
│ AI 团队：6 个 Agent ｜ Skill：12 个 ｜ 策略：已生效 ｜ 成本：估算 │
├────────────────────────────────────────────────────────────┤
│ [本项目 AI 团队] [角色治理] [Skill 治理] [策略与权限] [成本与记忆] │
├──────────────────────┬─────────────────────────────────────┤
│ 轻列表 / 资产列表       │ 当前选中项摘要面板                     │
│                      │                                     │
│ AI 项目主管            │ 角色：项目主管                         │
│ 前端体验 Agent         │ 职责：统筹计划、分配任务、汇总风险       │
│ 后端闭环 Agent         │ Skill：计划拆解、风险判断、调度建议      │
│ 测试验收 Agent         │ 权限：可建议重规划，不可直接写仓库       │
│ 文档收口 Agent         │ 最近证据：参与计划 v3 评估              │
│ 审查 Agent             │                                     │
│                      │ [编辑职责] [配置 Skill] [查看证据]       │
└──────────────────────┴─────────────────────────────────────┘
```

交互原则：

```text
主页面只显示摘要。
详细配置放抽屉。
治理审计放弹窗或详情页。
不在治理中心展示完整任务列表、完整运行日志、完整审批列表。
```

---

## 36. 本项目 AI 团队

### 36.1 定位

本项目 AI 团队是治理中心的默认入口。

它展示当前项目主管为本项目生成的 Agent 编队。

第一阶段采用：

```text
AI 项目主管
├─ 前端体验 Agent
├─ 后端闭环 Agent
├─ 测试验收 Agent
├─ 文档收口 Agent
└─ 审查 Agent
```

第一阶段暂不做复杂子主管树。

后续大项目可以升级为：

```text
AI 项目主管
├─ 前端子主管
│  ├─ 页面 Agent
│  └─ 交互 Agent
├─ 后端子主管
│  ├─ API Agent
│  └─ Runtime Agent
└─ 测试子主管
   ├─ 单测 Agent
   └─ 回归 Agent
```

### 36.2 展示字段

每个 Agent 只展示关键摘要：

| 字段 | 说明 |
|---|---|
| Agent 名称 | 当前项目中的角色实例名称 |
| 来源 | 来自角色模板 / 项目主管临时生成 |
| 职责摘要 | 做什么、不做什么 |
| 绑定 Skill 数量 | 当前角色绑定的 Skill 数 |
| 当前任务状态 | 空闲 / 执行中 / 等待依赖 / 阻塞 |
| 最近消费证据 | 最近一次被运行消费的 run / task |
| 是否建议沉淀 | 项目角色是否值得沉淀为角色模板 |

### 36.3 允许操作

| 按钮 | 行为 |
|---|---|
| 查看职责 | 打开 Agent 职责抽屉 |
| 查看绑定 Skill | 跳到 Skill 治理或打开抽屉 |
| 查看运行证据 | 跳运行观测并带 runId / roleCode |
| 查看当前任务 | 跳任务队列并带 roleCode |
| 建议沉淀为角色模板 | 生成沉淀建议，不直接永久保存 |
| 请求 AI 重新组队 | 生成重组建议，进入工作台待确认 |

危险操作：

```text
停用 Agent
删除角色实例
更换关键 Agent
大幅修改职责
```

必须进入确认流程，不允许静默执行。

---

## 37. 角色治理

### 37.1 角色治理定位

角色治理管理两类对象：

```text
角色模板
项目角色实例
```

角色模板是可复用资产；项目角色实例是当前项目内由 AI 项目主管生成的具体角色。

### 37.2 角色模板

角色模板保存：

| 字段 | 说明 |
|---|---|
| 角色名称 | 例如：前端页面治理 Agent、后端接口闭环 Agent |
| 职责 | 负责什么 |
| 输入边界 | 接收什么上下文 |
| 输出边界 | 产出什么结果 |
| 禁止动作 | 不能做什么 |
| 适用项目类型 | 前端、后端、文档、测试、审查等 |
| 常用 Skill | 推荐绑定的 Skill |
| 历史使用次数 | 被多少项目复用 |
| 最近使用项目 | 追溯模板使用场景 |
| 效果反馈 | 成功率、返工率、用户采纳情况 |

角色模板不是固定套壳。

新项目开始时，AI 项目主管应读取角色模板，但必须根据项目目标重新判断是否适用。

### 37.3 项目角色实例

项目角色实例保存：

| 字段 | 说明 |
|---|---|
| 实例名称 | 当前项目中的具体 Agent 名称 |
| 来源 | 来自模板 / AI 临时生成 |
| 和模板差异 | 本项目对职责、边界、Skill 的调整 |
| 执行反馈 | 项目中是否有效 |
| 是否候选模板 | 是否值得沉淀为长期模板 |

### 37.4 角色生命周期

角色生命周期：

```text
AI 主管生成项目角色
  ↓
项目中使用
  ↓
执行结果反馈
  ↓
AI 主管提出沉淀建议
  ↓
用户确认
  ↓
保存为角色模板 / 继续保留项目记录 / 不沉淀
  ↓
后续项目复用
```

角色状态建议：

| 状态 | 含义 |
|---|---|
| project_local | 项目本地角色 |
| template_candidate | 候选角色模板 |
| template_stable | 稳定角色模板 |
| deprecated | 不推荐继续使用 |
| archived | 归档 |

### 37.5 关键边界

AI 项目主管可以建议沉淀角色，但不能自动永久保存角色模板。

必须由用户确认。

原因：

```text
角色模板会影响未来项目
角色职责可能携带项目特定假设
角色可能过拟合当前仓库
角色可能引入权限风险
```

---

## 38. Skill 治理

### 38.1 Skill 治理定位

Skill 治理是治理中心的核心能力之一。

它管理：

```text
正式 Skill
项目临时 Skill
Skill 迭代候选
Skill 版本
Skill 消费证据
Skill 清理策略
```

### 38.2 正式 Skill

正式 Skill 是可长期复用的能力。

例如：

```text
前端页面职责边界审查
按钮闭环检查
运行日志摘要
Git 提交草案生成
文档口径统一检查
构建失败诊断
```

正式 Skill 展示字段：

| 字段 | 说明 |
|---|---|
| Skill 名称 | 能力名称 |
| 能力说明 | 能解决什么问题 |
| 适用角色 | 哪些 Agent 常用 |
| 适用项目类型 | 前端 / 后端 / 文档 / 测试 / 审查 |
| 当前版本 | v1 / v2 / v3 |
| 使用次数 | 被项目调用次数 |
| 最近消费证据 | 最近被哪个 run / task 使用 |
| 效果反馈 | 采纳率、失败率、返工率 |
| 是否推荐 | stable / deprecated |

### 38.3 项目临时 Skill

项目临时 Skill 是高度依赖某个项目或阶段的一次性能力。

例如：

```text
AI-Dev-Orchestrator 运行摘要 2C-B-R1 文档追溯检查
某个特定页面的临时验收规则
某次阶段 gate 的临时审计规则
```

临时 Skill 字段：

| 字段 | 说明 |
|---|---|
| Skill 名称 | 临时能力名称 |
| 所属项目 | 来源项目 |
| 创建原因 | 为什么生成 |
| 使用次数 | 是否被实际使用 |
| 最后使用时间 | 清理判断依据 |
| 过期时间 | 生命周期限制 |
| 是否绑定角色 | 是否仍被项目角色引用 |
| 是否建议提升 | 是否值得沉淀为正式 Skill |

### 38.4 Skill 迭代候选

当某个 Skill 有复用价值但能力不足时，不应该直接废弃。

应进入 Skill 迭代候选。

例子：

```text
按钮闭环检查 Skill 在 3 次项目中都漏掉了后端状态机判断。
AI 项目主管建议升级为 v2，增加“后端状态机验证”步骤。
```

候选操作：

| 按钮 | 行为 |
|---|---|
| 生成新版本 | 创建 Skill 版本草案 |
| 合并到已有 Skill | 合并能力，不新增重复 Skill |
| 提升为正式 Skill | 需要用户确认 |
| 继续暂存 | 留在临时池 |
| 删除临时 Skill | 按清理策略删除 |

### 38.5 Skill 生命周期

Skill 生命周期：

```text
临时生成
  ↓
项目内使用
  ↓
AI 项目主管评估
  ↓
用户确认
  ↓
正式保存 / 继续暂存 / 删除
  ↓
后续项目复用
  ↓
根据执行反馈迭代版本
```

Skill 状态建议：

| 状态 | 含义 |
|---|---|
| draft | 草案 |
| temporary | 临时 |
| candidate | 候选沉淀 |
| stable | 稳定可复用 |
| deprecated | 不推荐使用 |
| archived | 归档 |

### 38.6 Skill 清理机制

项目临时 Skill 不能无限增长。

清理不能只按时间顺序 FIFO，还要综合判断：

```text
最后使用时间
使用次数
是否被角色绑定
是否有关联项目仍在进行
是否有成功运行证据
是否被正式 Skill 覆盖
是否被更高版本替代
是否被用户收藏 / 固化
```

优先清理：

```text
长期未使用
只用过一次
没有任何角色绑定
没有成功运行证据
已有更高版本替代
已有正式 Skill 覆盖
```

### 38.7 Skill 沉淀边界

AI 可以建议沉淀 Skill，但不能自动永久保存 Skill。

必须用户确认。

原因：

```text
Skill 会影响未来项目
Skill 可能携带项目特定假设
Skill 可能过拟合当前仓库
Skill 可能带来错误工作流
Skill 可能产生权限风险
```

## 39. 策略与权限

### 39.1 定位

策略与权限用于控制 AI 项目主管和 Agent 能自动做什么，哪些必须用户确认，哪些禁止自动执行。

### 39.2 三类权限

治理中心必须用三类展示权限：

```text
可自动执行
需要用户确认
禁止自动执行
```

#### 可自动执行

```text
拆分任务
调整任务优先级
分配 Agent
生成短摘要
生成变更建议
生成审批建议
记录运行消费证据
生成临时 Skill 草案
```

#### 需要用户确认

```text
应用重规划
新增大量任务
删除任务
关闭任务
修改验收标准
发起正式审批
更换关键 Agent
调整预算策略
提升临时 Skill 为正式 Skill
保存项目角色为角色模板
```

#### 禁止自动执行

```text
真实修改仓库文件
git commit
git push
发布生产环境
删除项目
删除交付物
覆盖 Provider Key
删除运行证据
永久删除稳定 Skill / 角色模板
```

### 39.3 权限展示方式

不要做大权限表。

建议使用三张卡片：

```text
┌────────────────────────────┐
│ 可自动执行                  │

│ - 分配 Agent                │
│ - 生成短摘要                │
└────────────────────────────┘

┌────────────────────────────┐
│ 需要用户确认                │
│ - 应用重规划                │
│ - 删除任务                  │
│ - 提升 Skill                │
└────────────────────────────┘

┌────────────────────────────┐
│ 禁止自动执行                │
│ - git push                  │
│ - 发布生产                  │
│ - 删除运行证据              │
└────────────────────────────┘
```

## 40. 成本与记忆

### 40.1 成本治理最终定位

成本治理不是单纯成本看板。

它是：

> **AI 生成行为的预算、复用、缓存和模型分级控制机制。**

它要回答：

```text
哪些内容必须 AI 生成？
哪些内容可以规则生成？
哪些内容可以复用？
哪些内容只能用户主动问时再生成？
哪些内容需要便宜模型？
哪些内容值得用强模型？
哪些内容生成一次后必须缓存？
```

核心原则：

```text
事件触发、结果缓存、分层摘要、按需展开、可复用资产优先、强模型只用于高价值判断。
```

## 41. AI 生成成本治理规则

### 41.1 页面打开不触发 AI 生成

硬原则：

```text
页面打开不触发 AI 生成。
页面只读取已有缓存。
```

禁止设计：

```text
用户打开成果中心 → 现场生成交付物摘要
用户打开项目页 → 现场生成项目摘要
用户打开治理页 → 现场生成 Skill 总结
用户打开运行页 → 现场生成运行摘要
```

正确设计：

```text
运行结束 → 自动生成运行摘要并缓存
任务状态变化 → 自动更新任务短摘要
交付物生成 → 自动生成交付摘要并缓存
审批请求创建 → 自动生成审批建议并缓存
Skill 被使用后 → 记录消费证据和效果摘要
角色参与项目后 → 记录角色效果摘要
用户打开页面 → 读取已有摘要
```

用户手动按钮只允许：

```text
重新生成
生成详细版
生成汇报版
要求 AI 项目主管解释
修正摘要
```

### 41.2 AI 生成分级

所有 AI 生成内容分成 4 个等级。

| 等级 | 名称 | 是否调用模型 | 适用场景 |
|---|---|---|---|
| L0 | 规则摘要 | 不调用模型 | 状态、数量、是否阻塞、是否待审批 |
| L1 | 短摘要 | 便宜模型 / 快模型 | 运行短摘要、任务短摘要、交付物短摘要 |
| L2 | 结构化判断 | 中等模型 | 是否重试、是否返工、是否送审、Skill 是否沉淀 |
| L3 | 高价值决策 | 强模型 | 重规划、复杂根因分析、高风险仓库变更、审批放行建议 |

L0 示例：

```text
当前项目有 3 个待调度任务、1 个阻塞任务、1 个待审批交付物。
```

L1 示例：

```text
该任务最近一次运行失败，失败集中在日志弹窗接口调用，建议先检查 run-log API 和空状态处理。
```

L2 示例：

```text
该 Skill 在本项目中被复用 3 次，其中 2 次被采纳，建议进入候选沉淀。
```

L3 示例：

```text
当前阶段连续失败 2 次，且阻塞涉及任务拆解不足，建议触发项目重规划评估。
```

### 41.3 强模型调用理由

每次使用强模型必须记录理由。

示例：

```text
使用强模型原因：连续 2 次运行失败，需要复杂根因分析。
```

```text
使用强模型原因：用户请求重新规划当前阶段任务。
```

如果没有明确理由，不允许默认使用强模型。

### 41.4 省钱模式 / 标准模式 / 深度模式

治理中心应支持 AI 成本模式。

| 模式 | 行为 |
|---|---|
| 省钱模式 | 优先规则摘要，自动摘要更短，强模型只在用户确认后使用，非关键 Skill 迭代建议延后 |
| 标准模式 | 自动生成短摘要，关键节点生成审批建议，必要时使用中等模型，强模型用于重规划和复杂诊断 |
| 深度模式 | 更主动分析，更完整摘要，更多风险评估，更多 Skill 迭代建议，成本更高 |

模式位置：

```text
治理中心 → 成本与记忆
```

不放在设置页。

### 41.5 成本预算闸门

AI 项目主管不能无限生成。

预算策略应包括：

```text
每个项目每日 AI 生成预算
每个项目总预算
强模型调用次数上限
自动摘要最大长度
自动摘要每日上限
重规划每日上限
Skill 迭代建议每日上限
```

超过预算时：

```text
自动降级为规则摘要
等待用户确认
改用便宜模型
推迟非必要生成
```

用户可见提示：

```text
今日强模型预算已接近上限，AI 项目主管将只生成短摘要和必要审批建议。
```

## 42. AI 生成资产台账

### 42.1 台账定位

所有 AI 生成内容都应记录进入 AI 生成资产台账。

台账不是普通日志，而是成本治理和复用评估的依据。

### 42.2 台账字段

| 字段 | 说明 |
|---|---|
| 生成对象 | run_summary / task_summary / deliverable_summary / approval_advice / skill_revision |
| 触发事件 | run_completed / deliverable_created / user_requested / skill_used |
| 使用模型 | deepseek-v4-pro / flash / fallback |
| 生成来源 | ai / rule_fallback / reused / inherited |
| token 成本 | prompt / completion / total |
| 是否缓存 | 是 / 否 |
| 是否复用 | 是 / 否 |
| 复用来源 | 哪个历史摘要、角色模板、Skill |
| 是否被用户采纳 | 是 / 否 |
| 是否可复用 | 是 / 否 |
| 强模型理由 | 如果使用强模型，必须填写 |

### 42.3 台账回答的问题

```text
钱花在哪里？
哪些 AI 生成最贵？
哪些摘要重复生成了？
哪些 Skill 复用节省了成本？
哪些角色模板提升了效率？
哪些模型调用没有价值？
```

## 43. 复用优先级

每次系统准备生成内容前，必须按这个顺序判断：

```text
1. 是否已有完全匹配的缓存？
2. 是否已有同一对象的最新摘要？
3. 是否可以用规则摘要？
4. 是否可以复用历史角色 / Skill / 模板？
5. 是否只需要短摘要？
6. 是否必须调用 AI？
7. 是否需要强模型？
```

如果前面任一步能解决，就不应继续调用更贵模型。

## 44. 摘要过期判断

缓存不是永久有效。

以下事件会导致摘要过期：

```text
关联 run 变了
任务状态变了
交付物版本变了
审批状态变了
计划版本变了
Skill 版本变了
用户手动修改了内容
```

页面展示：

```text
AI 摘要：已生成
来源：source=ai
生成时间：2026-05-18 20:42
状态：有效
```

过期提示：

```text
摘要可能已过期：交付物版本已更新。
[重新生成]
```

## 45. 角色和 Skill 复用收益

角色和 Skill 复用必须进入成本治理。

原因：

```text
角色模板减少重复角色设计
Skill 复用减少重复能力生成
历史消费证据减少重新验证成本
高质量 Skill 迭代减少任务失败和返工
```

页面展示示例：

```text
本项目复用了 4 个角色模板、7 个正式 Skill。
其中 3 个 Skill 有历史成功消费证据。
预计减少 8 次重复生成。
```

注意：

```text
这是成本治理提示，不是精确账单。
```

## 46. 记忆治理和成本治理的关系

记忆治理服务于成本治理。

它应该帮助系统少问、少生成、少重复判断。

记忆治理应保留：

```text
项目经验摘要
角色使用反馈
Skill 消费证据
摘要生成记录
用户采纳 / 驳回记录
高价值失败案例
```

不应该无限保留：

```text
低价值中间输出
过期临时 Skill
重复摘要
未被采纳的低质量建议
无证据支撑的临时推断
```

记忆操作：

| 按钮 | 行为 |
|---|---|
| 查看记忆快照 | 弹窗 |
| compact | 压缩低价值记忆 |
| rehydrate | 为当前任务恢复相关上下文 |
| reset 治理产物 | 高风险确认 |
| 查看成本来源 | 弹窗 |

## 48. 治理中心按钮闭环

| 按钮 | 闭环要求 |
|---|---|
| 请求 AI 重新组队 | 生成团队重组建议，进入工作台待确认 |
| 保存为角色模板 | 必须用户确认，记录来源项目和版本 |
| 更新角色模板 | 必须记录版本差异 |
| 提升为正式 Skill | 必须用户确认，记录适用范围 |
| 生成 Skill 新版本 | 创建版本草案，不直接覆盖稳定版本 |
| 清理临时 Skill | 按清理策略执行，支持确认 |
| 切换成本模式 | 保存治理策略，影响后续 AI 生成行为 |
| 查看强模型调用 | 打开台账弹窗 |
| 查看复用收益 | 打开成本治理摘要 |
| compact 记忆 | 调真实治理接口 |
| rehydrate 记忆 | 调真实治理接口 |

不能做假按钮：

```text
一键优化所有 Skill
自动永久保存所有角色
自动删除低质量 Skill
自动重组团队并立即生效
```

这些必须经过建议、确认、执行三个阶段。

## 49. 治理中心当前定稿结论

```text
治理中心是 AI 团队资产治理中心，不是普通设置页。

它负责管理本项目 AI 团队、角色模板、项目角色实例、正式 Skill、项目临时 Skill、Skill 迭代候选、策略权限、成本预算、模型分级、记忆治理和治理审计。

AI 项目主管每个项目都应根据项目需求重新生成团队方案，可以复用历史角色和 Skill，但不能机械套用模板。

角色和 Skill 可以沉淀、复用、暂存、清理和迭代，但 AI 只能建议沉淀，永久保存和高风险变更必须由用户确认。

成本治理不是成本看板，而是 AI 生成行为的预算、复用、缓存和模型分级控制机制。页面打开不得触发 AI 生成，所有摘要应在生命周期事件中生成并缓存。强模型调用必须记录理由，角色和 Skill 复用收益也应进入成本治理。
```

## 50. 设置页设计

设置页是本轮页面信息架构的最后一个基础页面。它必须和治理中心严格拆开。

最终边界：

```text
治理中心管 AI 怎么工作。
设置页管系统怎么连接、怎么运行、是否健康。
```

### 50.1 设置页最终定位

```text
设置页是 AI-Dev-Orchestrator 的系统基础配置中心，负责 Provider 与模型连接、运行环境状态、安全配置和系统诊断。

设置页不负责 Agent 编队、角色模板、Skill 治理、成本模式、记忆治理、任务调度、运行诊断、仓库变更、交付物审批和项目计划。
```

设置页回答的问题是：

```text
系统是否能连上模型？
后端是否正常？
Worker 是否可用？
运行数据目录是否可写？
关键安全配置是否完整？
出现问题时，我能复制什么诊断信息给开发者？
```

它不回答：

```text
AI 项目主管应该怎么调度？
哪个 Skill 应该沉淀？
哪个 Agent 应该执行任务？
某个交付物是否应该审批通过？
某次运行为什么失败？
```

这些问题分别由工作台、治理中心、执行中心和成果中心负责。

---

### 50.2 设置页只保留 4 个区块

设置页采用轻量 2x2 卡片布局：

```text
Provider 与模型
运行环境
安全与权限
系统诊断
```

不要继续扩展成大杂烩设置页。

| 区块 | 职责 |
|---|---|
| Provider 与模型 | API Key、Base URL、默认模型、轻量模型、Timeout、测试连接。 |
| 运行环境 | 后端健康、数据库、Worker、Event Stream、runtime data、日志目录状态。 |
| 安全与权限 | 密钥隐藏、仓库安全根目录、危险能力底层开关。 |
| 系统诊断 | 版本、提交哈希、连接状态、最近错误、复制诊断信息。 |

---

### 50.3 设置页推荐布局

设置页不能做成窄表单，也不能做成日志大屏。推荐结构：

```text
┌────────────────────────────────────────────────────────────┐
│ 设置                                                         │
│ 系统状态：Provider 已配置，后端正常，Worker 空闲              │
├──────────────────────────┬─────────────────────────────────┤
│ Provider 与模型            │ 运行环境                         │
│ 状态：已配置               │ 后端：正常                        │
│ 默认模型：deepseek-v4-pro  │ 数据库：正常                      │
│ 轻量模型：deepseek-flash   │ Worker：空闲                      │
│ [测试连接] [编辑配置]       │ [运行诊断] [复制诊断]              │
├──────────────────────────┬─────────────────────────────────┤
│ 安全与权限                 │ 系统诊断                         │
│ Key：已隐藏                │ 前端版本：xxx                     │
│ Git 写入：未启用           │ 后端版本：xxx                     │
│ 仓库根路径：已配置          │ 最近错误：无                      │
│ [编辑安全配置]              │ [复制诊断信息]                    │
└────────────────────────────────────────────────────────────┘
```

布局要求：

- 顶部只放一句系统状态，不做大量统计。
- 主体用 2x2 卡片充分利用宽屏空间。
- 编辑配置通过弹窗 / 抽屉完成，不在页面常驻大表单。
- 诊断结果可以复制，不默认铺完整日志。
- 危险配置必须二次确认。

### 50.4 Provider 与模型

这是设置页最重要的区块。

常驻展示：

```text
当前 Provider：DeepSeek / OpenAI Compatible
状态：已配置 / 未配置 / 测试失败
Base URL：已配置
默认模型：deepseek-v4-pro
轻量模型：deepseek-v4-flash
Timeout：300s
最近测试：成功 / 失败 / 未测试
```

API Key 不常驻明文显示，只能显示脱敏状态：

```text
API Key：已配置，sk-****abcd
```

按钮：

| 按钮 | 闭环要求 |
|---|---|
| 测试连接 | 必须调用真实后端测试接口；没有接口时不能假装测试成功。 |
| 编辑配置 | 打开弹窗 / 抽屉，保存后重新读取配置状态。 |
| 查看最近调用 | 读取已有调用记录，不现场生成 AI 分析。 |
| 清除 Key | 高风险操作，必须二次确认。 |

建议后续补齐后端接口：

```text
POST /provider-settings/openai/test
```

返回至少包括：

```text
configured
base_url reachable
auth valid
model reachable
latency
error summary
```

如果该接口尚未实现，前端只能展示“待接入测试连接”，不能做假按钮。

### 50.5 运行环境

运行环境区块用于快速判断系统是否能正常工作。

常驻展示：

```text
后端健康状态
数据库连接状态
Worker 状态
Event Stream 状态
runtime_data_dir 是否可写
日志目录是否可访问
```

推荐文案：

```text
后端：正常
数据库：正常
Worker：空闲
Event Stream：正常
Runtime Data：可写
```

按钮：

| 按钮 | 闭环要求 |
|---|---|
| 运行诊断 | 调用健康检查 / 诊断接口，返回真实状态。 |
| 复制诊断信息 | 复制当前真实状态，不生成长篇 AI 总结。 |
| 查看健康检查 | 跳转或弹窗展示健康检查结果。 |

注意：设置页不展示完整运行日志。完整日志属于运行观测页。

---

### 50.6 安全与权限

设置页的安全与权限只处理系统底层开关，不处理 AI 行为治理。

设置页可以展示：

```text
API Key 是否已配置
Key 最后更新时间
敏感配置是否隐藏
仓库安全根目录
允许访问的本地路径
Git 写入底层能力是否启用
```

但是：

| 内容 | 应放页面 |
|---|---|
| AI 是否可以自动 git commit / push | 治理中心 |
| 哪些危险动作必须审批 | 治理中心 |
| 仓库变更需求、文件定位、预检、提交草案 | 执行中心 / 仓库工作区 |
| Provider Key / Base URL / Timeout | 设置页 |

一句话：

```text
设置页决定底层能力是否存在。
治理中心决定 AI 是否允许请求使用该能力。
```

### 50.7 系统诊断

系统诊断是为开发、联调、排错准备的轻量状态汇总。

应展示：

```text
前端版本
后端版本
当前 main commit
Provider configured
Repository bound count
Task count
Run count
Latest run status
Runtime data path
最近错误摘要
```

按钮：

```text
复制诊断信息
查看健康检查
导出诊断包（后续能力，未接后端前不展示或禁用）
```

系统诊断不能做成大日志页，也不能在页面打开时自动调用 AI 生成诊断报告。需要深度排查时，用户可以复制诊断信息到工作台询问 AI 项目主管。

### 50.9 设置页按钮闭环

| 按钮 | 是否允许 | 要求 |
|---|---|---|
| 测试连接 | 允许 | 必须真实请求后端测试接口。 |
| 保存 Provider | 允许 | 保存后重新读取 provider 状态。 |
| 清除 API Key | 允许 | 必须二次确认。 |
| 复制诊断信息 | 允许 | 复制真实状态。 |

| 开启 Git 写入 | 后续 | 必须有后端能力和权限闭环。 |
| 修改成本模式 | 不允许放设置页 | 应进入治理中心。 |

### 50.10 设置页最终定稿结论

```text
设置页是系统基础连接和运行环境配置中心，只负责 Provider 与模型连接、运行环境状态、安全配置和系统诊断。

设置页不承担 AI 团队治理、角色 Skill 资产管理、成本策略、记忆治理、任务执行、运行观测、仓库变更、交付物审批和项目计划职责。

设置页页面常驻信息应保持轻量，但要充分利用宽屏空间，采用顶部状态条 + 2x2 设置卡片布局。配置编辑通过弹窗或抽屉完成，危险配置必须二次确认。所有测试连接、运行诊断和保存配置按钮必须有真实后端闭环。
```

## 53. 当前定稿结论

### 52.1 工作台定稿

```text
工作台是 AI 项目主管轻量指挥室。

它以 AI 项目主管对话框为主体，只展示当前项目最小态势、AI 主管当前建议，以及作战计划、Agent 动向、项目流程、待确认、阻塞处理等少量弹窗入口。

工作台不展示完整任务列表、完整运行列表、完整仓库内容、完整交付物列表、完整审批列表和完整治理配置。
```

### 52.2 项目页定稿

```text
项目页是项目档案和 AI 作战计划总账本。

它只展示项目目标、当前阶段、AI 作战计划摘要、当前风险、进度摘要和最近项目事件。

项目页不承担全局导航职责，不常驻展示任务、运行、仓库、交付物、审批、治理等入口矩阵，避免和侧边栏重复。

项目页允许问题驱动的上下文跳转，但不做其他页面的缩略版。
```

### 52.3 执行中心定稿

```text
执行中心是 AI 项目主管调度后的执行工作区。

它由任务队列、运行观测、仓库工作区三个同级页面组成，默认进入任务队列。

任务队列负责调度状态，运行观测负责单次执行诊断，仓库工作区负责代码变更需求、上下文证据和提交草案。

执行中心不做新的总览大屏，不重复工作台和项目页，不把任务、运行、仓库三套重信息堆在同一个页面。
```

### 52.4 成果中心定稿

```text
成果中心是 AI 项目执行结果的交付与审批工作区。

它由交付物和审批两个同级页面组成，默认进入交付物。

交付物页负责看成果，审批页负责做决定。交付物正文、证据链、版本记录通过弹窗或抽屉收纳。审批按钮必须说明后果并写入真实审批决策。

成果中心摘要应在关键生命周期节点自动生成并缓存。用户进入页面时默认读取已生成摘要，不应为了第一次理解成果而手动等待 AI 总结。Provider 失败时使用 rule_fallback 摘要并明确标记来源。
```

### 52.5 全局自动摘要机制定稿

```text
AI 摘要是全局能力，不只属于成果中心。

系统应在 run 结束、task 状态变化、deliverable 生成、approval request 创建、project plan 变更、repository change request 评估完成等关键节点自动生成并缓存摘要。

页面默认读取缓存摘要，用户手动按钮只用于重新生成、修正、复制、展开或生成更详细版本。

Provider 失败时必须生成 source=rule_fallback 摘要，页面不能空白，也不能伪装成真实 AI 摘要。
```

### 52.6 治理中心定稿

```text
治理中心是 AI 团队资产治理中心。

它管理本项目 AI 团队、角色模板、项目角色实例、正式 Skill、项目临时 Skill、Skill 迭代候选、策略权限、成本预算、模型分级、记忆治理和治理审计。

AI 项目主管每个项目都应根据项目需求重新生成团队方案，可以复用历史角色和 Skill，但不能机械套用模板。

角色和 Skill 可以沉淀、复用、暂存、清理和迭代，但永久保存和高风险变更必须由用户确认。

成本治理不是成本看板，而是 AI 生成行为的预算、复用、缓存和模型分级控制机制。页面打开不得触发 AI 生成，强模型调用必须记录理由，角色和 Skill 复用收益应进入成本治理。
```

### 52.7 全局导航定稿

```text
工作台 / 项目 / 执行中心 / 成果中心 / 治理 / 设置
```
