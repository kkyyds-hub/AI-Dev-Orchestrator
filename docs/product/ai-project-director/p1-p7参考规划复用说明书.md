### 2.1 明确结论

```text
当前路线"有偏移，但不是做错"。
```

### 2.2 偏移点

| 原 gap 文档 | 当前实际 | 偏移判断 |
|------------|---------|---------|
| P2: 前端只读展示 + Runtime 使用 workspace_path | 已完成 | 无偏移 |
| P3: AI runtime in worktree | 已完成（Runtime Lifecycle Evidence + Event Audit） | 无偏移 |
| **P4: 失败回流** | **当前 P4 实际在做 Git Delivery Preview / Operation Dry-run / Delivery Gate Evidence / Human Approval Gate** | **偏移** |

## 6. 当前 P4 为什么可以继续

### 6.1 不是做错

当前 P4 虽然偏离原 gap 文档编号，但它覆盖了原文 P5/P6 的一部分必要能力：

| 当前 P4 的实际工作 | 对应原 gap 文档中的阶段 |
|-------------------|----------------------|
| Git Diff Dry-run (P4-B) | 原 P5 worktree diff → commit candidate |
| Git Operation Dry-run Preview (P4-C) | 原 P5 commit candidate |
| Delivery Gate Evidence (P4-D) | 原 P6 审批 / delivery |
| Human Approval Gate (P4-F) | 原 P6 审批（人工确认部分） |

这些都是 Git 只读交付链路上的必要能力，当前实现是正确的、可测试的、可审计的。

### 6.2 不应推倒

推倒当前 P4 重新从 Failure Recovery 开始会造成：

- 已完成的 P4-B/C/D/F1/F2-A/F2-B 全部浪费。
- 已通过的 75 个 targeted tests 全部作废。
- Git delivery evidence 链断裂。

### 6.3 继续推进的前提

| 前提 | 说明 |
|------|------|
| P4 继续按当前路线收完 Git Delivery Preview + Human Approval Gate | 合理且必要的 |
| P4 完成后必须回补 P5 Failure Recovery | 不能跳过失败回流直接进入 P6/P7 |
| P4 的全线安全边界不变 | git add / commit / push / PR 仍然绝对禁止 |
| P4 的 gate_allows_write 永不为 True | approval_granted 只表示可进入下一 guardrail |
