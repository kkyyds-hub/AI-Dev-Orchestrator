## 2. 参考项目与可吸收经验（基于公开仓库）

### 2.1 `OpenHands`
1. 公开仓库显示其同时提供 `GUI / CLI / REST API` 入口，并持续投入评测和轨迹可见性。
2. 可吸收的经验：
   1. 不要把“后端能力”和“前端控制台”分裂成两套概念；同一套任务状态应同时服务 API、CLI、UI。
   2. 好用的代理系统必须让用户看到当前在做什么，而不是只看到最后成败。

### 2.2 `AutoGPT`
1. 公开仓库强调 `Agent Builder`、工作流管理、部署控制、监控与分析面板。
2. 可吸收的经验：
   1. 当系统开始具备自动执行能力后，用户最需要的是**控制入口、监控入口和回看入口**。
   2. “会跑”不是重点，“能看见、能干预、能复盘”才是产品体验分水岭。

### 2.3 `aider`
1. 公开仓库突出代码库感知、Git 工作流整合，以及自动运行 lint / test 的使用方式。
2. 可吸收的经验：
   1. 编码代理真正好用，必须把“改代码 -> 验证 -> 给出结果”连成一个闭环。
   2. 如果验证动作需要用户每次手工拼装，体验会迅速下降。

### 2.4 `mini-swe-agent`
1. 公开仓库强调最小实现、线性历史、子进程执行和可浏览轨迹。
2. 可吸收的经验：
   1. 在当前阶段，保持单 `Worker`、线性状态流是正确的，不要过早追求复杂编排。
   2. 简单、可追踪、容易调试，往往比架构上的“高级感”更重要。

### 2.5 `Agentless`
1. 公开仓库把问题求解拆成 `localization -> repair -> patch validation` 这类阶段化流程。
2. 可吸收的经验：
   1. 诊断、执行、验证最好在代码结构上显式分层，不要全部塞进一个巨大的执行器里。
   2. 一旦要提升成功率，就必须让失败阶段和失败类型可见。

## 3. 对本项目的冻结判断

1. 你当前已经完成了 Day 10 的最小闭环，下一阶段不该优先做“更强的自动化”，而该优先做“更顺手的使用体验”。
2. 从这些项目的共同模式看，接下来最值得投资的顺序应当是：
   1. **详情与历史可见**
   2. **日志可读与操作可控**
   3. **状态实时更新**
   4. **验证更加结构化**
   5. **预算与重试更安全**
3. 这一路径同时符合你现有总纲里“可验证、可恢复、可观测、可人工接管”的主线，不需要推翻当前 V1 方向。
