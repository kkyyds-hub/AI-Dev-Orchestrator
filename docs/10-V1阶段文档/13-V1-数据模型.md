# V1 数据模型冻结：Task / Run 最小模型

- 项目：`AI-Dev-Orchestrator`
- 阶段：`V1`
- 文档版本：`v1.0`
- 冻结日期：`2026-03-10`
- 文档性质：**数据模型冻结文档**
- 目标：冻结 `Task / Run` 的最小字段、状态语义和代码落位，为 Day 3 的 `SQLite` 持久化与 Day 4 的任务接口提供统一基础

---

## 0. 范围定位（冻结）

1. 本文档只解决 `Task / Run` 最小模型定义，不扩展到数据库建表、接口设计或 `Worker` 调度。
2. 本文档的目标是把 Day 2 的核心边界定准，而不是一次性设计完整领域体系。
3. 所有字段都以“够支撑 Day 3 / Day 4”为标准，避免过度设计。

---

## 1. 边界决策（已冻结）

1. `Task` 和 `Run` 必须拆成两个独立对象：**YES**
2. `Task` 负责“要做什么”和“任务整体状态”：**YES**
3. `Run` 负责“某次执行尝试”以及执行结果：**YES**
4. `Task -> Run` 先按一对多关系建模：**YES**
5. Day 2 只冻结最小字段，不提前设计完整事件流模型：**YES**
6. Day 2 引入数据库表、仓储层和接口 DTO：**NO**

## 4. `Task` 模型冻结

### 4.1 职责定义

`Task` 表示系统中的“一个待推进事项”，关注的是这件事本身，而不是某次执行尝试。

### 4.2 最小字段

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `id` | `UUID` | 是 | 任务唯一标识 |
| `title` | `str` | 是 | 任务标题，面向列表展示和人工识别 |
| `status` | `TaskStatus` | 是 | 任务整体推进状态 |
| `priority` | `TaskPriority` | 是 | 任务优先级 |
| `input_summary` | `str` | 是 | 任务输入摘要，供路由、执行和 UI 查看 |
| `created_at` | `datetime` | 是 | 创建时间 |
| `updated_at` | `datetime` | 是 | 最后更新时间 |

### 4.3 状态枚举冻结

| 枚举值 | 含义 |
| --- | --- |
| `pending` | 任务已创建，尚未开始执行 |
| `running` | 任务正在执行中 |
| `completed` | 任务已完成 |
| `failed` | 任务已失败，需要重试或人工处理 |
| `blocked` | 任务被阻塞，当前无法自动继续 |

### 4.4 优先级枚举冻结

| 枚举值 | 含义 |
| --- | --- |
| `low` | 低优先级 |
| `normal` | 默认优先级 |
| `high` | 高优先级 |
| `urgent` | 最高优先级 |

### 4.5 暂不纳入 Day 2 的字段

以下字段先不进入 Day 2 最小模型：

1. `description`
2. `acceptance_criteria`
3. `depends_on`
4. `assignee`
5. `budget_limit`
6. `latest_run_id`

这些字段是否需要引入，留待 Day 3 之后根据接口和调度需要再判断。

## 5. `Run` 模型冻结

### 5.1 职责定义

`Run` 表示对某个 `Task` 的一次执行尝试，关注的是这次执行的状态、模型和结果摘要。

### 5.2 最小字段

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `id` | `UUID` | 是 | 执行记录唯一标识 |
| `task_id` | `UUID` | 是 | 关联的任务标识 |
| `status` | `RunStatus` | 是 | 当前执行状态 |
| `model_name` | `str \| None` | 否 | 本次执行使用的模型名称 |
| `started_at` | `datetime \| None` | 否 | 实际开始执行时间 |
| `finished_at` | `datetime \| None` | 否 | 实际结束执行时间 |
| `result_summary` | `str \| None` | 否 | 本次执行结果摘要，可记录成功或失败信息 |
| `created_at` | `datetime` | 是 | 执行记录创建时间 |

### 5.3 状态枚举冻结

| 枚举值 | 含义 |
| --- | --- |
| `queued` | 运行记录已创建，等待执行 |
| `running` | 本次执行正在进行 |
| `succeeded` | 本次执行成功 |
| `failed` | 本次执行失败 |
| `cancelled` | 本次执行被取消 |

### 5.4 时间语义约束

1. `started_at` 和 `finished_at` 都可为空，因为运行记录可能还未真正开始。
2. 当两者都存在时，`finished_at` 不允许早于 `started_at`。
3. `created_at` 用于记录该次运行记录的生成时间，与实际开始执行时间分离。

### 5.5 暂不纳入 Day 2 的字段

以下字段先不进入 Day 2 最小模型：

1. `attempt_number`
2. `model_provider`
3. `prompt_tokens`
4. `completion_tokens`
5. `estimated_cost`
6. `artifact_path`
7. `error_code`

这些字段留待 Day 3 之后配合持久化、成本记录和日志系统再评估。

---

## 6. `Task -> Run` 关系冻结

1. 一个 `Task` 可以对应零次或多次 `Run`。
2. `Task` 表示“任务本体”，`Run` 表示“执行尝试历史”。
3. `Task.status` 和 `Run.status` 不能混为一谈：
   1. `Task.status` 描述任务整体推进状态。
   2. `Run.status` 描述某次执行尝试状态。
4. Day 3 落库时，应优先保证 `Run.task_id -> Task.id` 关系清晰。

## 8. 最终结论

Day 2 已冻结的最小领域模型为：**`Task` 管任务本体，`Run` 管执行尝试；二者分开建模，按最小字段推进。**

只要 Day 3 和 Day 4 继续围绕这套边界实现，就能避免在持久化和接口层反复返工。
