# V1 技术栈与工程基线冻结

- 项目：`AI-Dev-Orchestrator`
- 阶段：`V1`
- 文档版本：`v1.1`
- 冻结日期：`2026-03-08`
- 文档性质：**技术栈冻结文档**
- 目标：冻结 `V1` 的后端、前端、存储、通信、执行与运行策略，避免实现期反复摇摆

---

## 0. 范围定位（冻结）

1. `V1` 技术栈必须服务于“单用户、本地优先、低成本、可恢复、可观察”的目标。
2. `V1` 的重点不是追求分布式或平台级复杂度，而是尽快建立一个可演示的最小闭环。
3. 所有技术选型都应优先支持任务流转、状态持久化、执行验证和成本控制。

---

## 1. 边界决策（已冻结）

1. 后端语言采用 `Python 3.11+`：**YES**
2. 后端框架采用 `FastAPI`：**YES**
3. 数据校验以 `Pydantic v2` 为主：**YES**
4. 持久层采用 `SQLAlchemy 2.x + SQLite`：**YES**
5. 前端采用 `React + TypeScript + Vite + Tailwind CSS`：**YES**
6. 前后端通信采用 `REST + SSE`：**YES**
7. 执行方式采用本地子进程调用：**YES**
8. 日志与非结构化产物落本地文件系统：**YES**
9. 托管数据库、托管队列、托管日志平台：**NO**
10. 分布式调度、复杂消息系统、独立向量数据库：**NO**
11. 默认引入 WebSocket 作为实时主通道：**NO**
12. `V1` 阶段引入浏览器自动化闭环：**NO**

---

## 2. 工程基线总览（冻结）

| 层级 | 定案 |
| --- | --- |
| 后端语言 | `Python 3.11+` |
| 后端框架 | `FastAPI` |
| 数据校验 | `Pydantic v2` |
| 持久层 | `SQLAlchemy 2.x` |
| HTTP 服务 | `Uvicorn` |

| 非结构化产物 | 本地文件系统 |

| 执行方式 | 本地 `subprocess` |
| 调度方式 | 数据库状态机 + 单 `Worker` 循环 |

### 2.1 正式定案组合

- 后端：`Python + FastAPI + Pydantic + SQLAlchemy + SQLite`
- 前端：`React + TypeScript + Vite + Tailwind CSS + TanStack Query`
- 通信：`REST API + SSE`
- 执行：本地命令执行
- 存储：`SQLite + JSONL/Markdown/Artifacts`
- 部署：默认本机运行，前端构建产物由后端托管

## 3. 分层定案与原因

### 3.1 后端层：`Python + FastAPI`

- 选择原因：
  1. 任务编排、文件处理、命令执行和自动化脚本都更适合 `Python`。
  2. `FastAPI` 适合快速建立本地控制面 API，并天然带有较好的类型和文档体验。
  3. `Pydantic` 与 `SQLAlchemy` 的组合足以支撑 `V1` 的状态持久化需求。
- 冻结结论：
  1. 后端负责 `Planner / Router / Context Builder / Executor / Verifier / Budget Guard / Memory / Intervention Inbox`。
  2. `V1` 不追求服务拆分，不引入多进程、多节点或分布式调度。

### 3.2 前端层：`React + TypeScript + Vite`

- 选择原因：
  1. 控制台类界面需要任务队列、日志流、详情抽屉和状态面板。
  2. `React` 组件生态成熟，适合长期迭代。
  3. `TypeScript` 有利于前后端契约一致。
  4. `Vite` 和 `Tailwind CSS` 适合个人项目快速起步。
- 冻结结论：
  1. `V1` 前端聚焦状态展示，不追求复杂交互系统。
  2. `V1` UI 范围只保留项目概览、任务列表、运行状态、成本统计、阻塞项。

### 3.3 存储层：`SQLite + 本地文件系统`

- 结构化状态统一存入 `SQLite`：
  1. `Project`
  2. `Task`
  3. `Run`
  4. `Event`
  5. `BudgetSnapshot`
  6. `InterventionItem`
- 非结构化内容统一放本地文件系统：
  1. 执行日志
  2. 仓库摘要
  3. 任务摘要
  4. 调用快照
  5. 生成文档
  6. 构建产物
- 冻结结论：
  1. `SQLite` 开启 `WAL`。
  2. 核心状态进数据库，长文本和过程日志进文件。
  3. `V1` 不引入独立向量数据库。

### 3.4 通信层：`REST + SSE`

- 选择原因：
  1. `REST` 足以覆盖查询与操作。
  2. `SSE` 足以覆盖任务状态流、运行日志和阶段进度推送。
  3. 可避免过早引入 `WebSocket` 的连接管理复杂度。
- 冻结结论：
  1. `V1` 默认不做 `WebSocket`。
  2. 实时能力优先通过单向推送解决。

### 3.5 调度与执行层：数据库状态机 + 单 `Worker`

- 选择原因：
  1. 最简单可靠。
  2. 易恢复、易调试、易审计。
  3. 更适合单用户本地优先场景。
- 冻结结论：
  1. 任务状态持久化到数据库。
  2. 单 `Worker` 循环从数据库中取任务执行。
  3. 完成后写回状态、日志、成本和验证结果。

### 3.6 验证层：本地命令执行

- 允许的验证工具类型：
  1. `pytest`
  2. `npm test`
  3. `npm run build`
  4. `ruff`
  5. `mypy`
  6. `eslint`
- 冻结结论：
  1. `Verifier` 与 `Executor` 统一走本地命令能力。
  2. `V1` 不依赖外部 CI 平台作为闭环前提。

## 4. 仓库骨架冻结

### 4.1 后端目录约定

```text
runtime/orchestrator/
  app/
    api/
    core/
    domain/
    services/
    workers/
  pyproject.toml
  README.md
```

### 4.2 数据与运行目录约定

```text
runtime/
  data/
    db/
    logs/
    artifacts/
    summaries/
```

### 4.3 冻结结论

1. `app/domain/` 放核心对象模型。
2. `app/services/` 放业务服务与编排逻辑。
3. `app/workers/` 放后台循环与调度单元。
4. `runtime/data/` 放数据库、日志、产物和摘要。

---

## 5. 接口与契约策略（冻结）

1. 后端 `Pydantic` 模型应作为接口契约源头。
2. 前端类型应围绕后端接口收敛，而不是各自发散。
3. 路由命名、状态枚举、错误结构必须统一。
4. `Task / Run / Event` 的字段命名应优先稳定，为后续 UI 和数据库提供统一语义。

---

## 6. 运行与部署策略（冻结）

1. 默认运行方式为本机直接启动。
2. 后端使用 `Uvicorn` 启动。
3. 前端在 `V1` 默认作为独立开发环境运行，后期可由后端托管构建产物。
4. 远程部署、容器化和多机运行都不是 `V1` 前置条件。

---

## 7. 成本控制策略（冻结）

1. 优先减少无效模型调用，而不是过早优化服务器成本。
2. 优先投资 `Router / Context Builder / Budget Guard / Verifier`。
3. 默认应具备：
   1. 日预算
   2. 会话预算
   3. 贵模型使用配额
   4. 失败重试上限
   5. 超预算降级策略

---

## 8. 后续演进触发条件

只有当以下条件出现时，才考虑升级当前技术边界：

1. `SQLite` 成为明确瓶颈。
2. 单 `Worker` 已无法满足真实任务吞吐。
3. `REST + SSE` 已无法支撑必要的交互场景。
4. 现有存储与摘要机制已无法支持复盘或长期记忆需求。

---

## 9. 最终结论

`V1` 的技术栈已经冻结为：**`Python FastAPI` 后端 + `React/Vite` 前端 + `SQLite` + 本地文件系统 + 本地命令执行。**

这套技术栈的价值不在于“高级”，而在于它能以最低复杂度支撑 `V1` 最小闭环。
