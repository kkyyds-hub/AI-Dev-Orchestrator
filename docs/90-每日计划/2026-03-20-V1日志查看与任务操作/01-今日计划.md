# 2026-03-20 范围冻结：V1 日志查看与任务操作
- 项目：`AI-Dev-Orchestrator`
- 阶段：`V1`
- 模块：`Console / Worker / Run / Intervention`
- 文档版本：`v1.0`
- 冻结日期：`2026-03-20`
- 文档性质：**日计划冻结文档**
- 目标：让用户不仅能看任务，还能看日志、手动触发执行并对失败任务做最小操作。
---

## 0. 范围定位（冻结）

1. 今天的唯一主目标是：给控制台补齐最小操作能力和日志查看能力。
2. 今天优先解决的问题是：Day 10 / Day 11 的页面还偏“只读”，用户无法方便地继续推进任务。
3. 今天明确不做：复杂批量操作、任务编辑器、多用户审批流。

---

## 1. 边界决策（已冻结）
1. 日志查看优先读取结构化 `jsonl` 事件，而不是直接做原始文件下载：**YES**
2. 任务操作优先只支持“触发 Worker 一次”和“重试任务”两类核心动作：**YES**
3. 重试仍然围绕同一条 `Task` 增加新的 `Run` 尝试，不复制任务记录：**YES**

---

## 2. 当前基线（基于仓库事实）

1. 当前已经具备：日志文件落盘、日志路径字段、Worker 单轮执行接口、首页任务列表。
2. 当前仍然缺少：日志读取接口、前端日志查看器、控制台操作按钮、任务重试入口。
3. 今天的工作将基于：Day 9 的日志文件结构和 Day 10 的首页继续扩展。

---

## 4. Step `P12-S1`：日志读取接口

- 目标：让控制台可以读取结构化运行日志。
- 范围：
  1. 新增按 `run_id` 读取日志事件的接口
  2. 控制返回结构为事件列表，而不是原始文本
  3. 限制单次返回量，避免日志接口失控
- 输出：
  1. `runtime/orchestrator/app/services/run_logging_service.py`
  2. `runtime/orchestrator/app/api/routes/runs.py` 或等价文件
- DoD：
  1. 可以按 `run_id` 获取日志事件
  2. 结果中包含 `timestamp / level / event / message / data`
- 完成状态：**已完成**
- 完成交付证据：
  1. 已在 `runtime/orchestrator/app/services/run_logging_service.py` 增加结构化日志读取能力
  2. 已新增 `runtime/orchestrator/app/api/routes/runs.py` 并暴露 `GET /runs/{run_id}/logs`
  3. 日志接口已限制单次读取上限，并返回 `timestamp / level / event / message / data`

## 5. Step `P12-S2`：任务操作接口

- 目标：给用户一个继续推进任务的最小控制入口。
- 范围：
  1. 复用已有 `POST /workers/run-once`
  2. 新增 `POST /tasks/{task_id}/retry` 或等价接口
  3. 明确失败任务 / 阻塞任务的重试语义
- 输出：
  1. `runtime/orchestrator/app/api/routes/tasks.py`
  2. `runtime/orchestrator/app/services/task_service.py`
  3. `runtime/orchestrator/app/workers/task_worker.py`（如有必要）
- DoD：
  1. 控制台可以手动触发单轮执行
  2. 失败任务可以被重新推进到下一次尝试
- 完成状态：**已完成**
- 完成交付证据：
  1. 已复用现有 `POST /workers/run-once` 作为 Day 12 手动执行入口
  2. 已在 `runtime/orchestrator/app/services/task_service.py` 冻结失败 / 阻塞任务重试语义
  3. 已在 `runtime/orchestrator/app/api/routes/tasks.py` 新增 `POST /tasks/{task_id}/retry`

## 7. 风险与阻塞
1. 如果日志接口直接返回整文件原文，后续分页和实时化会很难扩展。
2. 如果重试语义今天不冻结，Day 14 接验证结果时会返工。
3. 如果前端按钮缺少明确 loading / error 状态，体验会很差。
