## 2. P7 核心定位

### 2.1 P7 是什么

P7 Project Director Conversation Hub + Governance 的目标是：

**让系统从"安全 Agent 编排器"升级为"可讨论、可质疑、可影响计划的 AI 项目主管"。**

引用 P1-P7 参考规划复用说明书第 5 节的核心判断：

> P1-P6 是理想 AI 主管的必要条件，但不是充分条件。
> 如果没有 P7 Conversation Hub，系统最终会更接近一个
> 能执行代码任务、能做工作区隔离、能做审批与交付治理的安全 Agent 编排器，
> 而不是理想中的 AI 项目主管。

### 2.2 P7 不是什么

| # | P7 不是 | 说明 |
|---|--------|------|
| 1 | 不是外部执行器接入 | P8/P9 才处理 executor config / runtime adapter |
| 2 | 不等于自动派发 | 不自动派发 Codex / Claude Code / DeepSeek |
| 3 | 不是 Git 写操作 | 产品运行时全线禁止 |
| 4 | 不是 P6 调度建议的自动执行 | P6 只出建议，P7 可以讨论建议，但不执行 |
| 5 | 不是 P8/P9 的实现 | P8/P9 是 P7 后的独立阶段 |

### 2.3 P7 与 P8/P9 的关系

| P7（本阶段） | P8/P9（后续） |
|-------------|-------------|
| 产出对话、质疑、建议、提案、审批治理 | 产出 executor config discovery + controlled launch |
| 设计 DirectorActionProposal | 设计 ExecutorLaunchPreview |
| 讨论"应该让 Codex 修这个 bug 吗？" | 展示"Codex CLI 可用，launch command 已预览" |
| 用户确认提案 | feature flag + 用户确认 + workspace 验证后真实启动 |
| 对话和提案可审计 | 执行器 session 可追踪 |

**P7 可以设计 DirectorActionProposal，但不能启动 Codex / Claude Code / DeepSeek。**

如果后续要把 Proposal 转成 Executor Launch Preview，必须进入 P8/P9，不能在 P7 偷做。

### 11.3 ProjectDirectorConversation 定义

#### 11.3.1 与 ProjectDirectorSession 的关系

**设计原则**：ProjectDirectorConversation 是 ProjectDirectorSession 的对话层叠加，不替换现有状态机。

| 维度 | ProjectDirectorSession（已有） | ProjectDirectorConversation（P7-C 设计新增） |
|------|------------------------------|------------------------------------------|
| 职责 | 目标摄入 → 澄清 → 确认 → 计划草案 → 审核 → 创建正式项目 | 自由对话、会话管理、多会话列表 |
| 主键 | `session.id` | 复用 `session.id` 作为 `conversation_id` |
| 状态机 | DRAFT → CLARIFYING → READY_TO_CONFIRM → CONFIRMED | active / idle / awaiting_user / archived |
| 消息 | 已有 `ProjectDirectorMessage`，按 `session_id` 关联 | 复用同一 message 表，不新增 `conversation_id` 列 |
| 与 project 关系 | `session.project_id`（可为 null） | 继承 `session.project_id` |
| 与 plan 关系 | 通过 `ProjectDirectorPlanVersion` 间接关联 | 通过 `latest_plan_version_id` 直读引用 |
| 与 task 关系 | 通过 `TaskCreationRecord` 间接关联 | 通过 `latest_task_creation` 直读引用 |

**结论**：P7-C 不新增 `conversation_id`。`session.id` 即 `conversation_id`。Conversation 是 session 的对话视图，不新建数据实体。

#### 11.3.2 Conversation 状态枚举

```text
ProjectDirectorConversationStatus
├── active: 有活跃对话，最近 N 分钟内有消息
├── idle: 无活跃对话，但 session 未确认/未完成
├── awaiting_user: assistant 最后一条消息 requires_confirmation=True
├── archived: 用户主动归档
├── completed: session 状态为 CONFIRMED 且已创建正式项目
```

#### 11.3.3 Conversation 与关键对象的关联边界

| 关联对象 | 关系 | 访问方式 | 读写 |
|---------|------|---------|------|
| ProjectDirectorMessage | 1:N | 复用 `message_repository.list_by_session_id(session_id)` | 只读 |
| ProjectDirectorPlanVersion | N:1（最新） | 复用 `plan_version_repository.list_by_session_id()` 取最新 | 只读 |
| TaskCreationRecord | N:1（最新） | 复用 `task_creation_repository.get_by_plan_version_id()` | 只读 |
| P6 AgentDispatchDecision | N:1（引用） | conversation 可引用 dispatch_decision_id 只读展示 | 只读 |
| P8/P9 ExecutorLaunch | **禁止直接引用** | P7 只允许 DirectorActionProposal 建议启动，不允许真实 launch | 不读不写 |
| Run / Approval / Deliverable | N:M（通过 project_id 间接关联） | ContextBuilderService 已可拼装 | 只读 |
| Project | N:1 | `session.project_id` | 只读 |

#### 11.3.4 Conversation 与 P6 dispatch decision 的只读引用

P7 conversation 可以只读引用 P6 `AgentDispatchDecision`：
- 对话上下文中展示"AI 主管上次调度建议：建议由 Codex 修复此任务"
- 用户可在对话中讨论 dispatch suggestion
- 讨论结果可生成 DirectorActionProposal（P7-G）
- 但不自动派发，不自动创建任务

#### 11.3.5 Conversation 与 P8/P9 的关系

| 规则 | 说明 |
|------|------|
| P7 conversation 中可以讨论"要不要让 Codex 修这个 bug？" | 允许 |
| 讨论结果可生成 DirectorActionProposal | 允许（P7-G） |
| Proposal 可以引用 P6 dispatch decision | 允许 |
| Proposal 不能直接触发 executor launch | **禁止** |
| 从 Proposal 到真实 Launch 必须经过 P8/P9 guardrail | **强制** |

### 11.4 ConversationList contract

#### 11.4.1 列表项字段设计

```text
ConversationListItem
├── conversation_id: UUID              # session.id
├── project_id: UUID | None
├── title: str                         # goal_text 的前 80 字符
├── kind: ConversationKind             # project_onboarding | general_discussion | plan_review | follow_up
├── status: ConversationStatus         # active | idle | awaiting_user | archived | completed
├── session_status: str                # session.status.value（DRAFT/CLARIFYING/READY_TO_CONFIRM/CONFIRMED）
├── last_message_preview: str          # 最近一条消息的前 120 字符
├── last_message_at: datetime | None   # 最近一条消息时间
├── message_count: int                 # 消息总数
├── pending_challenge_count: int       # 待处理质疑数（P7-F 后启用）
├── pending_proposal_count: int        # 待审批提案数（P7-G 后启用）
├── requires_user_action: bool         # awaiting_user 或 pending_challenge > 0 或 pending_proposal > 0
├── owner_scope: str                   # "project" | "user"
├── created_at: datetime
└── updated_at: datetime
```

#### 11.4.2 ConversationKind 枚举

```text
ConversationKind
├── project_onboarding: 目标输入 → 澄清 → 确认 → 计划草案（当前主流）
├── general_discussion: 自由对话（P7-C 新能力）
├── plan_review: 计划审核讨论
├── follow_up: 执行后的后续讨论
```

#### 11.4.3 排序与筛选规则

| 维度 | 规则 |
|------|------|
| 默认排序 | `last_message_at DESC`（最近活跃的在前） |
| 全部 | 所有 non-archived conversation |
| 当前项目 | `project_id == selectedProjectId` |
| 未完成 | `session_status != CONFIRMED` 或 `status == active` |
| 有待确认 | `status == awaiting_user` |
| 有质疑 | `pending_challenge_count > 0`（P7-F 后） |
| 已归档 | `status == archived` |
| 空状态 | "还没有主管会话。点击「新建主管会话」开始。" |

#### 11.4.4 恢复规则

| 场景 | 行为 |
|------|------|
| 点击 ConversationList 中的 session | 恢复该 session → 加载 messageTimeline，**不创建新 session** |
| session 为 null + 用户输入自然语言文本 | **不得默默当成新 goal_text 创建 session**。必须先要求用户选择现有会话或明确新建 |
| 用户点击「新建主管会话」 | 进入 goal_text 输入流程 → POST `/project-director/sessions` |
| 用户点击「继续现有会话」 | 直接恢复 session，不触发 provider，不生成澄清问题 |

#### 11.4.5 新建规则（关键安全修正）

**这是 P7-C 最关键的设计决策之一**：修正历史审计 P0 缺口。

| 修正前（P7-B 现状） | 修正后（P7-C contract） |
|-------------------|---------------------|
| session 为 null + 任何文本输入 → `handleSubmit` 调用 `createSessionMutation` | session 为 null + 自然语言输入 → 前端提示"请先选择或新建主管会话" |
| 自由文本被当成新 goal_text | 只有用户明确进入"新建主管会话"流程才允许创建 session |
| `DirectorChatEntry` 的 `handleSubmit` 无条件创建 session | 拆分两个路径：`handleStartNewSession`（创建）vs `handleSendMessage`（发送对话消息） |

### 12.3 DirectorInbox 定义

#### 12.3.1 DirectorInbox 是什么

DirectorInbox 是 AI 项目主管的**跨页面收件箱**，汇总来自不同页面和对象的需要主管注意的事项：用户随手备注、计划质疑、调度问题、审批提醒、运行阻塞、失败回流信号、治理警告等。

| DirectorInbox 是 | DirectorInbox 不是 |
|-----------------|-------------------|
| 跨页面注意事项的汇总视图 | 任务队列（不取代 Task list） |
| 备注、质疑、待处理事项的 intake 容器 | 通知中心泛化（不覆盖所有系统事件） |
| 主管注意力的管理中心 | Worker 执行入口 |
| 结构化 challenge 的种子来源 | 自动派发入口 |

#### 12.3.2 DirectorInbox 与其他 P7 组件的关系

| 组件 | 关系 | 说明 |
|------|------|------|
| ProjectDirectorConversation | inbox item 可 link 到已有 conversation | 用户把 inbox 中的事项带到对话中讨论 |
| ConversationList | inbox 在 ConversationList 上方作为 badge/入口 | 提醒用户有待处理事项 |
| UserChallenge (P7-F) | inbox item 可 convert 为 challenge seed | P7-D 只设计 intake，完整 challenge domain 留给 P7-F |
| DirectorActionProposal (P7-G) | inbox item 可 convert 为 proposal seed | P7-D 只设计 intake，完整 proposal domain 留给 P7-G |
| P6 dispatch decision | inbox 可显示 dispatch_question 类型 | 用户质疑 dispatch suggestion → 进入 inbox |

#### 12.3.3 DirectorInbox 不是什么

| # | 不是 | 说明 |
|---|------|------|
| 1 | 不是任务队列 | 不取代 Task list / Run list |
| 2 | 不是通知中心泛化 | 不覆盖所有系统事件（token 用完、model switch 等不在此） |
| 3 | 不是 Worker 执行入口 | 不启动 Worker |
| 4 | 不是自动派发入口 | 不派发 Codex / Claude Code / DeepSeek |
| 5 | 不是 retry 入口 | run_blocker 项不触发 retry |
| 6 | 不是 approval 审批流 | approval_attention 只是提醒，审批在 P4/G 链路 |

| Deliverable page | 交付物质疑 | note / plan_question | false | 用户手动 |
| Governance page | 成本超限、角色冲突、Skill 配置风险 | governance_warning | true | 系统自动生成 |

### 12.6 API contract 设计

以下为 P7-D 冻结的 API contract 设计，**不实现代码**。

#### GET `/project-director/inbox`

| 属性 | 设计 |
|------|------|
| Query | `project_id?`, `status?`, `kind?`, `priority?`, `limit?`, `before?` |
| Response | `{ "items": [DirectorInboxItem], "has_more": bool, "unread_count": int, "needs_response_count": int }` |
| 是否只读 | **是** — 纯查询 |
| 默认排序 | `created_at DESC`（最新的在前） |
| 默认筛选 | `status != archived` |

#### POST `/project-director/inbox/items`

| 属性 | 设计 |
|------|------|
| Request | `{ "kind": InboxItemKind, "title": str, "summary": str, "source_page": str, "source_entity_type?": str, "source_entity_id?": UUID, "related_*_id?": UUID }` |
| Response | `{ "item": DirectorInboxItem }` |
| 是否只读 | **否** — 创建 inbox item（仅持久化 metadata） |
| 禁止 | 创建 task/run/worker、触发 provider、执行 suggested_actions |

#### PATCH `/project-director/inbox/items/{item_id}/read`

| 属性 | 设计 |
|------|------|
| Request | `{ "status": "read" }` |
| Response | `{ "item": DirectorInboxItem }` |
| 是否只读 | **否** — 仅更新 status |
| 禁止 | 修改 conversation / task / run / plan |

#### PATCH `/project-director/inbox/items/{item_id}/archive`

| 属性 | 设计 |
|------|------|
| Request | `{ "status": "archived" }` |
| Response | `{ "item": DirectorInboxItem }` |
| 是否只读 | **否** — 仅更新 status |
| 禁止 | 删除历史数据、删除关联 message、修改 conversation |

#### POST `/project-director/inbox/items/{item_id}/link-conversation`

| 属性 | 设计 |
|------|------|
| Request | `{ "conversation_id": UUID }` |
| Response | `{ "item": DirectorInboxItem, "conversation_id": UUID }` |
| 行为 | 设置 `item.conversation_id` + `status=linked_to_conversation` + `linked_conversation_at=now` |
| 禁止 | 自动新建 conversation（conversation 必须已存在）、移动历史消息、创建 task/run/worker |

#### POST `/project-director/inbox/items/{item_id}/convert-to-challenge`

| 属性 | 设计 |
|------|------|
| Request | `{ "challenge_type": str, "challenge_detail": str }` |
| Response | `{ "item": DirectorInboxItem, "challenge_seed_id": UUID }` |
| 行为 | 创建 challenge seed 记录（P7-F 的输入），设置 `item.status=converted_to_challenge` |
| 禁止 | 应用计划变更、修改 task/run、触发 provider |
| 说明 | convert-to-challenge 只是生成 challenge seed，完整 UserChallenge domain 留给 P7-F |

### 12.7 Conversation relation contract

| 规则 | 说明 |
|------|------|
| inbox item 可关联已有 conversation | `link-conversation` → `item.conversation_id = conversation.id` |
| inbox item **不能**自动新建 conversation | conversation 必须已存在 |
| link-conversation 只改变 item 归属 | 不移动历史消息 |
| item linked 后 conversation timeline 可显示 inbox-linked 事件 | timeline 聚合时包含 inbox item 摘要 |
| conversation archive **不删除** inbox item | inbox item 独立保留 |
| inbox item resolved/archived **不删除** conversation message | message 独立保留 |
| inbox item 可存在但不关联任何 conversation | `conversation_id = None` 是合法状态 |

#### 22.4.3 ConversationRouter 存在性

**不存在。** 当前 MessageService 没有消息路由逻辑。所有消息走同一路径：
- 不区分 ordinary chat / continue / new goal / challenge plan / ask task-run / ask inbox
- session 为 null 时任何文本输入被前端当成新 goal_text（P7-C3 已修正前端行为）
- 无跨 session 消息归属判断
- 无消息是否应创建新 conversation 的判断

#### 22.4.4 是否支持区分消息类型

| 消息类型 | 当前是否支持 | 附注 |
|---------|------------|------|
| ordinary chat | **是** | 默认行为；所有消息先经过 Provider 尝试生成回复 |
| continue existing conversation | **是** | 前端点击 ConversationList item 恢复 session，后端 POST 到已有 session |
| create new goal | **部分** | `POST /sessions` 创建新 session；但消息内容不分析是否为"新目标意图" |

#### 22.4.5 forbidden action 检测机制

| 机制 | 当前状态 |
|------|---------|
| 检测时机 | Provider 输出解析时（`_parse_provider_reply_contract`） |
| 检测内容 | AI 回复文本中是否包含"已启动 Worker"、"已创建 Run"、"已执行 planning/apply"、"已执行 apply-local"、"已写入仓库"、"已提交代码"、"git push 已完成" |
| 检测结果 | 检测到→拒绝 Provider 输出→降级 rule fallback |
| 强制追加 | `_FORBIDDEN_MESSAGE_ACTIONS` 始终追加到 `forbidden_actions_detected` |
| 输入端检测 | **否** — 不检测用户输入是否包含"执行"、"启动"、"apply"等指令 |
| 误确认风险 | **存在** — 用户说"好的，继续执行"时，系统不会区分这是确认执行还是普通对话 |

#### 22.4.6 用户普通输入可能被误当成执行确认的风险

| 风险 | 当前状态 | P7-E 需求 |
|------|---------|----------|
| 用户说"可以的，去做吧" | Provider 可能生成带 `create_formal_project` suggested_action 的回复但不执行 | 无明显风险 — 系统不自动执行 suggested_actions |
| 用户说"继续" | 被当成普通对话消息，生成助手回复 | 风险低 — 但 assistant 可能误解用户意图 |
| 用户说"批准" | 当成普通对话，不触发 approval | 风险中等 — 用户以为批准了，系统未感知 |
| 用户说"创建任务" | 当成普通对话 | 风险低 — 但用户期望与实际行为不一致 |
| session null + 用户输入自然语言 | P7-C3 已修正前端：显示"请先选择或新建主管会话" | **已修正** |

#### 22.5.2 统一 routing / context assembly 缺口

**当前各端点独立运作，无统一 routing 层、无统一 context assembly 层。**

具体缺口：

| # | 缺口 | 影响 |
|---|------|------|
| 1 | POST message 不感知其他 conversation 的存在 | 无法判断消息是否应归属到已有 conversation 或新建 |
| 2 | ContextBuilder 不纳入 ConversationList readback | AI 不知道用户有多少会话、最活跃的会话是哪些 |
| 3 | ContextBuilder 不纳入 DirectorInbox readback | AI 不知道收件箱中有哪些待处理事项 |
| 4 | ConversationList / Inbox 是独立 read model，不与 POST message 联动 | 用户发消息后 inbox badge 不变（inbox 是确定的 synthetic read model）；conversation status 只在 GET 时计算 |
| 5 | 无统一 `GET /sessions/{id}/messages` 与 `GET /conversations/{id}` 的 context merge | 两个端点的 response 结构不同，前端需要两个独立的数据流 |

### 22.7 ConversationRouter contract 草案

以下为 P7-E1 冻结的 ConversationRouter contract 设计，**不实现代码**。

#### 22.7.1 核心域模型

**ProjectDirectorConversationIntent**（用户输入分类）

```text
ProjectDirectorConversationIntent
├── ordinary_chat: 普通闲聊、无明确任务指向的自由对话
├── summarize_current_state: 请总结当前整体状态
├── ask_why: 问原因（例如"为什么选了 Codex？"）
├── ask_risk: 问风险（例如"当前最大的风险是什么？"）
├── challenge_plan: 质疑计划（例如"阶段 2 不应该放在前面"）
├── challenge_dispatch: 质疑调度建议（例如"不应该用 Codex，应该用 DeepSeek"）
├── ask_next_step: 问下一步建议
├── request_plan_revision: 请求修改计划
├── request_create_task_queue: 请求创建任务队列
├── ask_task_or_run_status: 询问任务/运行状态
├── ask_inbox_item: 提及收件箱中的某项
├── unknown_or_unsafe: 无法分类或疑似不安全输入
```

**ConversationRouteDecision**（路由决策）

```text
ConversationRouteDecision
├── intent: ProjectDirectorConversationIntent
├── risk_level: RouteRiskLevel
├── context_scope: ContextScope              # 决定 ContextAssembler 组装哪些 section
├── target_conversation_id: UUID | None       # 归属到哪个 conversation
├── target_inbox_item_id: UUID | None         # 引用哪个 inbox item
├── target_plan_version_id: UUID | None       # 引用哪个 plan version
├── target_task_id: UUID | None               # 引用哪个 task
├── target_run_id: UUID | None                # 引用哪个 run
├── allowed_actions: list[str]                # 允许生成的 assistant 动作
├── forbidden_actions: list[str]              # 禁止生成的 assistant 动作
├── requires_confirmation: bool               # 是否需要用户显式确认
├── resolve_strategy: str                     # 如何生成回复：provider_only / rule_only / provider_with_fallback
└── safety_policy: RouteSafetyPolicy
```

**RouteRiskLevel**

```text
RouteRiskLevel
├── safe: 纯信息问答，回复不含任何 suggested action
├── low: 回复可含 navigate/explain/summarize 类型 suggested action
├── medium: 回复可含 request_changes 类型 suggested action
├── high: 回复可含 create_formal_project/run_worker_once，强制 requires_confirmation
├── unsafe: 输入疑似越权/禁止执行声明 → 强制 rule fallback + 安全提示
```

**RouteSafetyPolicy**

```text
RouteSafetyPolicy
├── policy_id: str
├── max_suggested_actions: int                # 最多几条 suggested action（默认 3）
├── allowed_action_types: list[str]           # 允许的 action type 白名单
├── forbidden_action_types: list[str]         # 禁止的 action type 黑名单
├── require_confirmation_for_actions: list[str]  # 哪些 action type 强制 requires_confirmation
├── forbid_provider_for_intents: list[str]    # 哪些 intent 不准调用 Provider（只用 rule fallback）
├── only_answer_for_intents: list[str]        # 哪些 intent 只能生成文本回复，不生成 suggested_actions
├── max_context_items: int                    # ContextAssembler 最大项数
└── human_readable_boundary_cn: str           # 用户可理解的安全边界文案
```

**ContextScope**（决定 DirectorContextAssembler 组装哪些 section）

```text
ContextScope
├── include_current_session: bool             # 默认 true
├── include_recent_messages: bool             # 默认 true
├── include_latest_plan: bool                 # 默认 true
├── include_task_creation: bool               # 默认 true
├── include_conversation_list_snapshot: bool  # 默认 false（仅在跨 session 路由时 true）
├── include_inbox_snapshot: bool              # 默认 false（ask_inbox_item 时 true）
├── include_related_task: bool                # ask_task_or_run_status 时 true
├── include_related_run: bool                 # ask_task_or_run_status 时 true
├── include_dispatch_decision_snapshot: bool  # challenge_dispatch 时 true
├── include_cost_budget_snapshot: bool        # 可选，ask_risk 时 true
├── include_safety_boundaries: bool           # 默认 true
└── recent_message_limit: int                 # 默认 20，ask_task_or_run_status 时可扩至 50
```

#### 22.8.1 Context Sections 定义

| Section | 来源 | 只读 | 最大条数 / 截断策略 | 允许缺失 | 可进入 provider prompt | 只用于 UI readback |
|---------|------|------|-------------------|---------|----------------------|-------------------|

| `recent_messages` | `message_repository.list_by_session_id()` | 是 | 20-50 条，每条 content 截断 500 chars | 是 | 是 | 否 |
| `latest_plan` | `plan_version_repository.list_by_session_id()` | 是 | 最新 1 条 — phases 截断 8、proposed_tasks 截断 12、risks 截断 8 | 是 | 是 | 否 |

| `related_run` | `run_repository`（如已有） | 是 | 最近 3 条 run — status/result_summary 截断 500 chars | 是 | 是 | 否 |
| `project_snapshot` | `project_repository.get_by_id()` | 是 | 单条 — name/summary/status/stage/task_stats | 是 | 是 | 否 |
| `task_snapshot` | `task_repository.list_by_project_id()` | 是 | 总数 + status 分布 + 最近 10 条 | 是 | 是 | 否 |
| `safety_boundaries` | 硬编码 + RouteSafetyPolicy | 只读 | 固定 6 条 + policy 动态追加 | 否 | 是 | 否 |
| `forbidden_actions` | `_FORBIDDEN_MESSAGE_ACTIONS` + RouteSafetyPolicy | 只读 | 全部 | 否 | 是 | 否 |
| `cost_budget_snapshot` | cost tracking service（如果已有） | 是 | 最近 1 条 cost summary | 是 | 是 | 否 |
| `dispatch_decision_snapshot` | P6 AgentMessage dispatch timeline | 是 | 最近 1 条 dispatch decision — instruction_draft/reason 截断 500 chars | 是 | 是（challenge_dispatch 时） | 否 |

#### 22.8.2 section 数据流

```text
DirectorContextAssembler.build_context(intent, route_decision)
  ├── current_session ← session_repository
  ├── recent_messages ← message_repository
  ├── latest_plan ← plan_version_repository
  ├── task_creation ← task_creation_repository (conditional)
  ├── project_snapshot ← project_repository (conditional)
  ├── task_snapshot ← task_repository (conditional)
  ├── conversation_list_snapshot ← ConversationService (conditional: cross-session)
  ├── inbox_snapshot ← InboxService (conditional: ask_inbox_item)
  ├── related_task ← task_repository (conditional: ask_task_or_run_status)
  ├── related_run ← run_repository (conditional: ask_task_or_run_status)
  ├── dispatch_decision_snapshot ← AgentMessageTable (conditional: challenge_dispatch)
  ├── cost_budget_snapshot ← cost service (conditional: ask_risk)
  ├── safety_boundaries ← hardcoded + RouteSafetyPolicy
  └── forbidden_actions ← hardcoded + RouteSafetyPolicy
```

新增 `SafetyPolicy` domain model：

```text
risk_level: low | medium | high
requires_confirmation: bool
forbidden_actions: list[str]
safe_next_actions: list[str]
user_visible_warning: str
```

安全策略：

| intent | risk | requires_confirmation | 用户可见提醒 |
|--------|------|-----------------------|--------------|
| `request_action` | high | true | 这听起来像要执行操作。系统不会自动执行任务，也不会修改仓库。 |
| `request_plan_change` | medium | true | 这可能会修改项目草案，需要你确认。 |
| `challenge_plan` | medium | false | 这是对计划的质疑，系统会先记录和解释，不会直接修改草案。 |
| `restart_or_new_goal` | medium | true | 新目标会开始新的主管对话，不会自动创建正式项目或任务。 |
| 其他 | low | false | 系统只会基于当前已保存信息回答。 |

`forbidden_actions` 始终为简单中文：

```text
不会自动执行任务
不会自动创建任务
不会修改仓库
不会启动外部工具
不会直接应用草案修改
```

`safe_next_actions` 只包含继续提问、查看项目草案、查看提醒、查看任务状态、要求解释原因等非执行动作。
