## 1. 背景与问题定义

### 1.1 当前已有能力

AI Project Director 截至 `4f6d37f` 已具备完整流程骨架：

```
目标输入 → 澄清问题生成 → 回答澄清 → 目标确认
    → 计划草案生成 → 审核草案（approve/reject/request_changes）
    → 创建正式项目与任务队列 → Worker 执行入口
```

支持的关键端点：
- `POST /project-director/sessions` — 创建会话（Provider-first 澄清，rule fallback）
- `POST /sessions/{id}/answers` — 提交澄清回答
- `POST /sessions/{id}/confirm` — 确认目标
- `POST /sessions/{id}/plan-versions` — 生成计划草案
- `POST /plan-versions/{id}/review` — 审核草案（approve/reject/request_changes）
- `POST /plan-versions/{id}/create-formal-project` — 创建正式项目与任务
- `GET /workbench/resume` — 工作台恢复（session + plan + taskCreation readback）

### 1.2 当前核心缺口

**审计结论（P0）**: 系统是**固定流程状态机**，不具备 session-scoped 自由对话能力。

具体表现：
1. `DirectorChatEntry.tsx` 的 `handleSubmit` **永远调用 `createSessionMutation`** — 任何用户文本被当成新目标
2. 无 `project_director_messages` 表、领域模型或 API
3. 无 context builder — AI 无法围绕当前 session/plan/taskCreation 回答
4. 无 intent 识别 — "总结这个草案" 会触发新澄清流程
5. 工作台恢复不包含对话历史

### 1.3 为什么"固定流程状态机"不等于 AI 项目主管

`page-information-architecture-20260518.md §3` 明确定义工作台为：

> **AI 项目主管轻量指挥室**。用户进入系统后，默认面对 AI 项目主管，可以通过**对话**提出目标、询问进度、调整计划、处理阻塞、请求总结、发起重新规划。

固定流程状态机只能处理"目标→计划→创建任务"这条预设路径。它不能处理：
- 用户在草案生成后追问"为什么这么拆"
- 用户在任务创建后问"下一步该做什么"
- 用户犹豫时问"你觉得这个方案有什么风险"
- 用户在审批前问"改成三阶段会更好吗"

这些交互**正是 AI 项目主管区别于普通任务管理工具的核心价值**。

### 1.4 为什么是 P0

`closure-checklist-20260518.md §WB-09` 明确要求"聊天框是否能访问项目上下文"。

当前实现使工作台**不符合产品基线定位**。此缺口阻止 AI Project Director 成为"对话式编排入口"，因此判为 **P0**。

### 1.5 Total Closure 影响

**AI Project Director total closure 仍为 Partial** — 此 P0 缺口尚未解决。

## 2. 设计目标

### 2.1 必须实现

1. **自由对话**: 用户可以像和 GPT 一样围绕当前目标、草案、技术细节、下一步自由输入自然语言，并获得 AI 主管的结构化回复
2. **上下文感知**: AI 主管必须理解当前 session（目标、澄清、确认状态）、plan version（草案、阶段、风险）、taskCreation（项目、任务队列）、project 状态
3. **智能解释与推理**: AI 主管可以解释草案拆分逻辑、总结当前进度、分析风险、建议修改方向、生成导航建议、识别用户动作意图
4. **受控执行**: AI 主管不能擅自执行高风险动作。聊天自由，执行受控，证据可回放
5. **持久化与恢复**: 对话历史持久化到后端，刷新页面后可恢复

### 2.2 设计原则

| 原则 | 说明 |
|------|------|
| 聊天自由 | 任何自然语言输入都必须被正常回答，不因"不匹配流程"而丢弃 |
| 执行受控 | 高风险动作必须用户显式确认，AI 只能建议 |
| 事实优先 | AI 回答基于后端 readback 的事实上下文，不靠幻想记忆 |
| 来源可追溯 | 每条 AI 回复标注 source（ai/rule_fallback）、模型、receipt |
| 渐进增强 | 不替换现有状态机，而是补充对话大脑层 |
| 安全边界不改 | Worker/Run/repo 写入的现有安全边界不受影响 |

---

## 3. 非目标

以下**明确不属于本设计范围**：

- 不做完全自主 Agent — AI 主管不能自主决策并执行多步操作
- 不允许 AI 自动启动 Worker — Worker dispatch 必须用户显式触发
- 不允许 AI 自动创建 Run — Run 创建必须通过 Worker 调度
- 不允许 AI 自动写仓库、git commit、git push、apply-local
- 不把聊天能力完成等同 total closure Pass
- 不处理 CL-16 成本 Pass（成本闭环需要真实 provider-reported cost）
- 不替代现有 Project Director 状态机 — 本设计是补充对话大脑层，不是重写

## 4. 总体架构

### 4.1 概念模型

```
AI Project Director = Conversational Brain
                    + Project State Context
                    + Tool Router
                    + Governance Guardrails
```

### 4.2 四层职责

```
┌─────────────────────────────────────────────────────────┐
│                  Conversational Brain                    │
│  自由理解用户意图，生成自然语言回答                        │
│  不限于固定流程 — 任何问题都正常回答                       │
│  基于下方 Project State Context 的事实数据                │
└──────────────────────┬──────────────────────────────────┘
                       │ 读取上下文
┌──────────────────────▼──────────────────────────────────┐
│                Project State Context                     │
│  后端组装当前所有事实：session / plan / taskCreation       │
│  / project / provider / forbidden_actions / messages     │
│  每次用户消息时实时 readback，不做前端缓存假数据           │
└──────────────────────┬──────────────────────────────────┘
                       │ 基于意图
┌──────────────────────▼──────────────────────────────────┐
│                    Tool Router                           │
│  把用户意图映射为 suggested_actions                       │
│  区分：只读动作 / 中风险动作 / 高风险动作 / 禁止动作       │
│  AI 只建议，不自动执行                                    │
└──────────────────────┬──────────────────────────────────┘
                       │ 动作分类
┌──────────────────────▼──────────────────────────────────┐
│               Governance Guardrails                      │
│  高风险动作必须用户确认                                   │
│  禁止动作绝对不执行                                       │
│  AI 回复不得宣称已执行未执行动作                           │
│  证据可回放                                              │
└─────────────────────────────────────────────────────────┘
```

### 4.3 与现有状态机的关系

本设计**不替换**现有 Project Director 状态机（DRAFT → CLARIFYING → READY_TO_CONFIRM → CONFIRMED）。Conversational Brain 是**叠加在状态机之上**的对话层：

- 状态机负责**流程推进**（澄清→确认→计划→创建任务）
- Conversational Brain 负责**理解与解释**（"这个草案怎么样？""下一步做什么？"）
- Tool Router 负责**桥接**两者（用户说"创建正式项目"→ 识别为 request_action → 建议 create_formal_project → 确认后调用现有端点）

## 5. 数据模型设计

### 5.1 新表: `project_director_messages`

```sql
CREATE TABLE project_director_messages (
    id              UUID PRIMARY KEY,
    session_id      UUID NOT NULL REFERENCES project_director_sessions(id),
    role            VARCHAR(16) NOT NULL,  -- 'user' | 'assistant' | 'system'
    content         TEXT NOT NULL,

    -- Intent & routing (AI replies only)
    intent          VARCHAR(64),           -- NULL for user messages

    -- Related entities (for traceability)
    related_plan_version_id UUID,
    related_project_id      UUID,
    related_task_id         UUID,

    -- AI reply metadata
    source          VARCHAR(32) NOT NULL DEFAULT 'system',  -- 'ai' | 'rule_fallback' | 'system'
    source_detail   TEXT,
    suggested_actions_json TEXT,          -- JSON array of suggested actions
    requires_confirmation  BOOLEAN NOT NULL DEFAULT FALSE,
    risk_level      VARCHAR(16),          -- 'low' | 'medium' | 'high'
    forbidden_actions_detected_json TEXT, -- JSON array of strings

    -- Cost tracking
    token_count     INTEGER,
    estimated_cost  REAL,

    created_at      TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_pd_messages_session_id ON project_director_messages(session_id, created_at);
CREATE INDEX idx_pd_messages_plan_version ON project_director_messages(related_plan_version_id);
```

### 5.2 领域模型

```python
# runtime/orchestrator/app/domain/project_director_message.py

@dataclass(frozen=True, slots=True)
class ProjectDirectorMessage:
    id: UUID
    session_id: UUID
    role: str                        # "user" | "assistant" | "system"
    content: str

    # Intent (AI replies)
    intent: str | None               # general_discussion | ask_about_plan | ...

    # Related entities
    related_plan_version_id: UUID | None
    related_project_id: UUID | None
    related_task_id: UUID | None

    # AI metadata
    source: str                      # "ai" | "rule_fallback" | "system"
    source_detail: str | None
    suggested_actions: list[dict] | None
    requires_confirmation: bool
    risk_level: str | None           # "low" | "medium" | "high"
    forbidden_actions_detected: list[str] | None

    # Cost
    token_count: int | None
    estimated_cost: float | None

    created_at: datetime
```

### 5.3 为什么必须持久化消息

1. **刷新恢复**: 用户刷新页面后必须恢复对话历史（`WB-09` 要求）
2. **证据回溯**: 每条 AI 回复的 source/source_detail/token_count 必须可审计
3. **上下文连续性**: 后续消息需要最近 N 条历史作为 LLM 上下文
4. **禁止前端缓存**: 不允许只靠 localStorage — 多设备/清缓存后数据丢失

### 5.4 与现有模型的关系

```
ProjectDirectorSession (1) ──→ (N) ProjectDirectorMessage
ProjectDirectorSession (1) ──→ (N) ProjectDirectorPlanVersion
ProjectDirectorPlanVersion (1) ──→ (N) ProjectDirectorMessage (via related_plan_version_id)
ProjectDirectorPlanVersion (1) ──→ (1) ProjectDirectorTaskCreationRecord
```

### 5.5 刷新后恢复

`WorkbenchResumeResponse` 在 Stage 7-B1 后应新增：
```python
recent_messages: list[MessageResponse] | None = None  # 最近 50 条
```

## 6. API 设计

### 6.1 GET /project-director/sessions/{session_id}/messages

**用途**: 获取指定 session 的消息历史，用于页面刷新恢复和滚动加载。

**请求参数**:
| 参数 | 类型 | 必填 | 默认 | 说明 |
|------|------|------|------|------|
| session_id | UUID (path) | 是 | — | 会话 ID |
| limit | int (query) | 否 | 50 | 返回条数上限 |
| before | UUID (query) | 否 | — | 游标，获取此 ID 之前的消息 |

**响应体** (`200`):
```json
{
  "session_id": "uuid",
  "messages": [
    {
      "id": "uuid",
      "role": "user",
      "content": "总结一下这个草案",
      "created_at": "2026-06-03T10:00:00Z"
    },
    {
      "id": "uuid",
      "role": "assistant",
      "content": "当前草案包含 2 个阶段...",
      "intent": "ask_about_plan",
      "source": "ai",
      "source_detail": "provider=deepseek; model=deepseek-v4-pro; receipt=xxx",
      "suggested_actions": [...],
      "requires_confirmation": false,
      "risk_level": "low",
      "created_at": "2026-06-03T10:00:05Z"
    }
  ],
  "has_more": false
}
```

**安全边界**: 只读操作。不创建 session、不调用 Provider、不触发 Worker。

**错误情况**:
- `404`: session 不存在
- 空 session: 返回 `{"session_id": "...", "messages": [], "has_more": false}`

### 6.2 POST /project-director/sessions/{session_id}/messages

**用途**: 用户发送一条消息，后端组装上下文、调用 Provider 生成 AI 回复，持久化并返回两者。

**请求体**:
```json
{
  "content": "这个草案有什么风险？"
}
```

**响应体** (`201`):
```json
{
  "user_message": {
    "id": "uuid",
    "role": "user",
    "content": "这个草案有什么风险？",
    "created_at": "..."
  },
  "assistant_message": {
    "id": "uuid",
    "role": "assistant",
    "content": "根据当前草案，主要风险包括：1) 范围蔓延...",
    "intent": "ask_about_risks",
    "source": "ai",
    "source_detail": "provider=deepseek; model=deepseek-v4-pro; receipt=xxx",
    "suggested_actions": [
      {
        "type": "request_changes",
        "label": "根据风险调整草案",
        "requires_confirmation": false,
        "risk_level": "low"
      }
    ],
    "requires_confirmation": false,
    "risk_level": "low",
    "forbidden_actions_detected": [],
    "created_at": "..."
  }
}
```

**后端处理流程**:
```
1. 验证 session 存在且状态允许发消息（非 DRAFT）
2. Persist user message
3. Context Builder 组装当前事实（见 §7）
4. Provider Config → call AI → guardrail → fallback if needed
5. Persist AI reply
6. Return both
```

**安全边界**:
- **不会触发**: Worker dispatch、Run 创建、git write、planning/apply、apply-local
- **不会修改**: session status、plan version、task queue
- AI reply 中的 `suggested_actions` 仅建议，不自动执行

**错误情况**:
- `404`: session 不存在
- `422`: content 为空或超长（max 10000 chars）
- `502`: Provider 调用失败且 fallback 不可用

### 6.3 POST /project-director/sessions/{session_id}/messages/{message_id}/confirm-action（可选，Stage 7-B5）

**用途**: 用户确认 AI 建议的高风险动作，触发对应的现有端点。

**请求体**:
```json
{
  "action_type": "create_formal_project",
  "confirmed": true
}
```

**响应体**: 代理调用对应端点（如 `POST /plan-versions/{id}/create-formal-project`）的结果。

**安全边界**:
- 必须用户显式确认
- `action_type` 必须在 AI 回复的 `suggested_actions` 中存在且 `requires_confirmation=true`
- 禁止动作（write repo / git commit / git push / apply-local）不可通过此端点触发

## 7. Context Builder 设计

### 7.1 上下文组装清单

每次 `POST /sessions/{id}/messages` 时，后端必须从持久化存储 readback 以下事实：

```python
context_payload = {
    # Session core
    "session": {
        "id": session.id,
        "goal_text": session.goal_text,
        "constraints": session.constraints,
        "status": session.status,                   # clarifying|ready_to_confirm|confirmed
        "goal_summary": session.goal_summary,
        "clarifying_questions": [...],              # 仅当 status=clarifying
        "clarifying_answers": [...],                # 仅当已提交
        "confirmed_at": session.confirmed_at,
    },

    # Plan (latest, any status)
    "latest_plan_version": {
        "id": plan.id,
        "version_no": plan.version_no,
        "status": plan.status,                     # pending_confirmation|confirmed|rejected
        "plan_summary": plan.plan_summary,
        "phases": [{"sequence": n, "name": ..., "goal": ...}],
        "proposed_tasks": [{"title": ..., "suggested_role_code": ...}],
        "acceptance_criteria": [...],
        "risks": [...],
        "project_scope": {"in_scope": [...], "out_of_scope": [...], "assumptions": [...]},
        "complexity_assessment": {"level": ..., "score": ..., "drivers": [...]},
        "source": plan.source,                     # ai|rule_fallback
    } or None,

    # Task creation (if exists)
    "task_creation": {
        "project_id": tc.project_id,
        "project_name": tc.project_name,
        "task_count": tc.task_count,
        "status": tc.status,
    } or None,

    # Provider status
    "provider_settings_status": "configured" | "not_configured" | "tested",

    # Governance
    "forbidden_actions": [
        "不自动创建任务", "不自动调用 Worker", "不写仓库",
        "不调用 planning/apply", "不调用 apply-local", "不执行 git-commit",
    ],

    # Recent conversation
    "recent_messages": [...]   # 最近 20 条，供 LLM 理解对话上下文

    # Current user message
    "current_user_message": request.content,
}
```

### 7.2 Token 控制策略

| 策略 | 说明 |
|------|------|
| 最近消息窗口 | 仅取最近 20 条消息（约 4K-8K tokens） |
| Plan 摘要截断 | plan_summary 截断到 2000 chars，完整 plan 通过 `related_plan_version_id` 关联 |
| 澄清问答截断 | 仅取问题和答案文本，不取完整 source_detail |
| 总量硬限制 | context_payload 序列化后不超过 ~12K tokens |
| 超长策略 | 如果超限，优先截断消息历史而非当前事实 |

### 7.3 事实来源规则

- **AI 不靠幻想记忆** — 所有事实从后端实时 readback
- **不从前端传上下文** — 前端只传 `content`，后端自主组装
- **source 必须标注** — `rule_fallback` 生成的 plan 必须标注，AI 回复不能假装它是 AI 生成的
- **过期检测** — 如果 plan version 已被 superseded，context builder 应标注

## 8. Intent 识别设计

### 8.1 设计原则

- **开放自然语言** — 不设计成"只能问 N 类问题"
- **任何输入都能正常回答** — unknown/general_discussion 不是错误处理，是正常的回答模式
- **不匹配不应触发新 session** — 只有用户明确表达"重新开始"才新建
- **Intent 是软引导，不是硬路由** — LLM 在 system prompt 中接受 intent 引导，但可以自主判断

### 8.2 建议 Intent 列表

| Intent | 含义 | 示例 |
|--------|------|------|
| `general_discussion` | 一般性讨论 | "这个想法怎么样"、"帮我分析一下" |
| `ask_about_current_context` | 询问当前状态 | "我们现在在哪一步"、"当前进度怎么样" |
| `ask_about_plan` | 询问草案 | "总结这个草案"、"为什么分两个阶段" |
| `ask_about_risks` | 询问风险 | "有什么风险"、"最可能出问题的地方" |
| `ask_about_next_step` | 询问下一步 | "接下来该做什么"、"下一步是什么" |
| `request_plan_change` | 想修改草案 | "改成三阶段"、"加一个测试任务" |
| `request_action` | 想执行动作 | "创建正式项目"、"开始执行" |
| `navigation_help` | 不知道去哪 | "在哪看任务"、"怎么去审批" |
| `restart_or_new_goal` | 想重新开始 | "换个目标"、"重新来"、"新建项目" |
| `unknown` | 未识别 | 任何其他自然语言输入 |

### 8.3 核心规则

1. **任何自然语言都必须能落到 `general_discussion` / `unknown`** — 这是正常路径，不是异常
2. **不能因为意图不匹配就重新创建 session** — 只有 `restart_or_new_goal` 且用户明确确认后才进入新 session
3. **只有用户明确表达"新建/重新开始/换目标"才进入 restart 流程**，且 AI 应先确认再执行
4. **Intent 由 LLM 在回复中自行标注**，不增加单独的 intent classification API call

```json

  "suggested_actions": [

      "label": "调整草案范围",
      "requires_confirmation": false,
      "risk_level": "low"
    },
    {
      "type": "create_formal_project",
      "label": "创建正式项目与任务队列",

    },
    {
      "type": "navigate",
      "label": "查看项目页",
      "requires_confirmation": false,
      "risk_level": "low"
    }
  ],
  "forbidden_actions_detected": [],
  "risk_level": "low",
  "source": "ai",
  "source_detail": "provider=deepseek; model=deepseek-v4-pro; receipt=a1b2c3d4"
}
```

### 9.2 各字段说明

| 字段 | 说明 | 约束 |
|------|------|------|
| `intent` | AI 识别的用户意图 | 来自 §8 的 intent 列表 |
| `answer` | 面向用户的中文自然语言回复 | 必须可读、准确、基于事实 |
| `related_plan_version_id` | 回复主要涉及的 plan version | 如有，便于前端定位 |
| `suggested_actions` | AI 建议的下一步动作 | 仅建议，不等于执行 |
| `requires_confirmation` | 该动作是否需要用户二次确认 | 高风险动作必须 `true` |
| `risk_level` | 该动作的风险等级 | `low` / `medium` / `high` |
| `forbidden_actions_detected` | AI 检测到用户可能想做的禁止动作 | 如检测到则警告 |
| `source` | 回复来源 | `ai` / `rule_fallback` |
| `source_detail` | 来源详情 | provider + model + receipt |

### 9.3 硬约束

1. **`answer` 是用户可读回复** — 中文优先，技术术语适度
2. **`suggested_actions` 只是建议** — 绝不等于自动执行
3. **`requires_confirmation=true` 的动作必须二次确认** — 前端必须渲染确认弹窗
4. **高风险动作不能自动执行** — 包括 create_formal_project / run_worker_once / create_run
5. **AI 不得宣称已执行未执行动作** — 如不能说"已创建项目"，除非真实创建完成
6. **禁止动作在 `forbidden_actions_detected` 中列出** — 如用户说"帮我 commit"，AI 必须说明不能执行

## 10. 前端交互设计

### 10.1 输入框语义按状态切换

| Session 状态 | 输入框 Placeholder | 行为 |
|---|---|---|
| `null` | "输入你的项目目标..." | 输入文本 → 创建新 session → 进入 clarifying |
| `clarifying` | "回答澄清问题，或继续描述你的需求..." | 主路径：在每个问题旁 textarea 填写答案；也允许在输入框发送自由消息 |
| `ready_to_confirm` | "继续和 AI 项目主管讨论..." | 自由对话；也可点击"确认目标"按钮 |
| `confirmed`（无 plan） | "讨论目标，或生成项目草案..." | 自由对话 + 快捷按钮"生成草案" |
| `confirmed`（有 plan） | "讨论草案、询问细节、或继续下一步..." | 自由对话 + suggested_actions 按钮 |
| taskCreation 存在 | "讨论执行计划、询问进度、或启动执行..." | 自由对话 + 执行入口 |

### 10.2 核心交互改动

1. **输入框始终可见** — 不再在 session 存在时隐藏
2. **输入框不再是"新目标"专属** — session 存在时调 `POST /sessions/{id}/messages`
3. **AI 回复列表** — 渲染 message 历史（用户气泡 + AI 气泡交替）
4. **suggested_actions 按钮** — 在 AI 回复下方以内联按钮展示
   - 低风险（`risk_level=low`）：直接渲染，无额外确认
   - 中/高风险（`risk_level=medium/high`）：渲染为带确认的按钮
5. **"新建会话"按钮** — 页面顶部显式入口，不与输入框混淆
6. **消息历史恢复** — 页面加载时调用 `GET /sessions/{id}/messages` 恢复
7. **source 标记** — AI 回复旁显示 source badge（"AI 生成" / "规则生成"）

### 10.3 确认弹窗规则

| 动作类型 | 确认弹窗需求 |
|----------|------------|
| `request_changes` | 直接打开 review modal（已有），不额外确认 |
| `create_formal_project` | 确认弹窗："将创建正式项目与 N 个任务，是否继续？" |
| `run_worker_once` | 确认弹窗："将启动一次 Worker 执行，可能产生费用，是否继续？" |
| `navigate` | 直接跳转，不确认 |
| `none` | 不渲染按钮 |

### 10.4 刷新恢复

页面刷新后：
1. `GET /workbench/resume` → 获取 session + plan + taskCreation
2. 如果 session 存在 → `GET /sessions/{id}/messages` → 恢复消息历史
3. 前端渲染完整的对话历史 + 当前状态

## 11. 工具动作与确认机制

### 11.1 动作分级

| 级别 | 动作 | 说明 |
|------|------|------|
| **只读** | `navigate`（导航到页面）、`summarize`（总结）、`explain`（解释）、`show_evidence`（展示证据） | 不修改任何状态，无需确认 |
| **中风险** | `request_changes`（请求修改草案）、`regenerate_plan`（重新生成草案） | 创建新 plan version，但不创建任务、不触发 Worker |
| **高风险** | `create_formal_project`（创建正式项目）、`run_worker_once`（启动 Worker）、`create_run`（创建 Run） | 修改项目/任务状态，可能产生费用 |
| **禁止或审批** | `write_repo`（写仓库）、`git_commit`、`git_push`、`apply_local` | 当前阶段不允许，或需要额外审批链路 |

### 11.2 确认规则

1. **AI 只能建议** — suggested_actions 是建议，不是决定
2. **用户确认后才调用现有接口** — 不能绕过已有的 create-formal-project / worker dispatch 端点
3. **Worker / Run / repo 写入必须保留现有安全边界** — `WORKER_SIMULATE_EXECUTION_OVERRIDE`、forbidden_actions 合约等
4. **任何动作结果必须可回放** — 动作执行后，结果应通过 message 回复或状态更新可见
5. **idempotency** — `create_formal_project` 已有 `already_created` 保护，重复调用安全

### 11.3 confirm-action 流程

```
用户点击 suggested_action 按钮
    ↓
前端检查 requires_confirmation
    ├── false → 直接触发（如 navigate / request_changes modal）
    └── true  → 弹出确认弹窗
                  ↓
              用户确认
                  ↓
         POST /sessions/{id}/messages/{msg_id}/confirm-action
                  ↓
              后端调用对应端点
              （如 POST /plan-versions/{id}/create-formal-project）
                  ↓
              返回结果
                  ↓
              前端更新状态 + 刷新消息列表
```

## 12. Provider 与 Fallback 策略

### 12.1 Provider 策略

结合当前系统已有的 Provider 策略（`project_director_service.py` 和 `project_director_plan_service.py`）：

| 场景 | 策略 |
|------|------|
| Provider 已配置且可用 | 调用 → guardrail 验证 → `source=ai` |
| Provider 未配置 | 使用 rule_fallback → `source=rule_fallback`、提示"AI 模型未配置" |
| Provider 调用失败/超时 | 使用 rule_fallback → `source=rule_fallback`、提示失败原因 |
| Provider 返回但 guardrail 不通过 | 使用 rule_fallback → `source=rule_fallback`、不展示不合格输出 |

### 12.2 对话 Fallback 特殊处理

对话回复的 rule_fallback 与 plan generation 不同：

- **Plan generation rule_fallback**: 生成确定性草案（已有 `_generate_rule_fallback_plan`）
- **对话 rule_fallback**: 生成规则化回复，例如：

```
我是 AI 项目主管。当前 AI 模型未配置或调用失败，我只能提供基础信息。

根据当前会话状态：
- 目标：Build a full-stack task management app
- 阶段：草案已生成，状态 pending_confirmation
- 草案包含 2 个阶段、6 个拟议任务

建议操作：
1. 审核草案：[查看草案详情]
2. 配置 AI 模型以获得智能回复：[前往设置]

source=rule_fallback，reason=provider_not_configured
```

### 12.3 不允许的行为

- **不允许把 rule_fallback 冒充 AI** — source badge 必须明确标注
- **不允许因为 Provider 失败就拒绝回答** — 必须给出 rule_fallback 回复
- **不允许展示未通过 guardrail 的 Provider 输出** — 安全优先

## 15. 风险与未决问题

| 风险/问题 | 影响 | 缓解措施 |
|-----------|------|---------|
| 上下文过长 | LLM token 超限导致回复质量下降 | 窗口截断策略（最近 20 条消息 + 摘要化的 plan/task context） |
| 多 session 并行 | 用户可能同时有多个未完成 session | `restart_or_new_goal` 时提示存在未完成 session；resume 列表支持切换 |
| suggested_actions 误触 | 用户误点高风险按钮 | 高风险动作必须二次确认；低风险动作（navigate）可即时执行 |
| Provider 失败 | AI 回复不可用 | rule_fallback 必须给出有意义的规则化回复，不展示空白 |
| 费用统计 | 对话新增 Provider 调用成本 | 每条 AI 回复记录 token_count/estimated_cost；CL-16 仍需真实 provider-reported cost |
| 仓库写入 | 用户可能要求 AI 直接改代码 | GAP-02 限制仍生效；write_repo/git_commit 在 forbidden_actions 中 |
| 与现有状态机交互 | 对话可能绕过状态机直接推进流程 | 所有流程推进仍须通过现有端点（confirm/review/create-formal-project），对话只建议 |
