## 2. 阶段范围定义

### 2.1 本阶段是什么

**产品运行时 Git 写闭环**是 AI-Dev-Orchestrator 的最后一个未闭环能力域。它在 P4（Git Dry-run + Human Approval）和 P9（Controlled Runtime）的已有基础上，为产品内部**受控、可审计、可回滚的 Git 写操作**建立 gate 设计与实现路线。

### 2.2 本阶段不是什么

| # | 不是 | 说明 |
|---|------|------|
| 1 | **不是**开发流程中的 `git commit` / `git push` | 开发流程 Git 操作是外部工具行为，不在产品运行时管控范围 |
| 2 | **不是**直接允许产品内部 `git add` / `git commit` / `git push` | 所有 Git 写必须先经过 gate chain |
| 3 | **不是** PR / merge 自动化 | PR 创建和 merge 决策必须人工确认 |
| 4 | **不是**真实执行器可以自行写 Git | 即使 P9 real executor pilot 启动，也不能绕过 GitWrite gate |
| 5 | **不是**无审批自动推送 | 每次写操作必须独立审批 |
| 6 | **不是**绕过 P4 dry-run preview | P4 的 diff/operation dry-run 是写前必经步骤 |
| 7 | **不是**直接写在 main 分支 | 必须有分支策略和 target branch allowlist |
| 8 | **不是**无限写入权限 | 每次写入是 one-shot token，用完即废 |

### 2.3 核心设计原则

```text
产品运行时 Git 写 = DiffPreview
                    + HumanApproval（当轮独立确认）
                    + SafetyGate（feature flag / workspace / budget / secret scan）
                    + OneShotApprovalToken（用完即废）
                    + AppendOnlyAuditEvent
                    + RollbackPlan
                    + DryRunBeforeWrite
```

## 4. 产品运行时 Git 写安全模型草案

### 4.1 安全原则总表

| # | 原则 | 规则 |
|---|------|------|
| 1 | **Feature Flag Default Off** | `PRODUCT_RUNTIME_GIT_WRITE_ENABLED` 默认 False |
| 2 | **Human Approval Required** | 每次写操作需要当轮独立的人类审批 |
| 3 | **Diff Preview Required** | 必须先生成并展示 diff preview |
| 4 | **Target Branch Allowlist** | 只能写入白名单中的分支 |
| 5 | **Workspace/Worktree Bound** | 必须绑定合法的 repository workspace |
| 6 | **No Direct Main Write** | 除非显式审批通过，否则不能写 main/master |
| 7 | **No Force Push** | 一律禁止 `--force` push |
| 8 | **No Reset/Rebase/Stash/Tag/Branch Delete** | 全线禁止破坏性操作 |
| 9 | **No CI Trigger Unless Explicit** | 不自动触发 CI pipeline |
| 10 | **No Secret in Diff** | 写前扫描 diff 内容，拒写含 secret 的变更 |
| 11 | **No Unreviewed Files** | 所有待提交文件必须在 diff preview 中展示 |
| 12 | **Audit Event Append-Only** | 每一步操作记录不可篡改的审计事件 |
| 13 | **Rollback Plan Required** | 写前必须准备回滚方案 |
| 14 | **Dry-Run Before Write** | 每次写前先执行 dry-run（如 `git add --dry-run`） |
| 15 | **One-Shot Approval Token** | 每次审批生成一次性 token，消耗后立即失效 |
| 16 | **No Product Runtime Git Write Without Gate Chain** | gate chain 任何一步失败 = 写操作拒绝 |

### 4.2 Git 写安全流程图

```text
User/Executor Intent → GitWriteIntent 创建
  → GitWritePreview 生成（diff + file list + target branch）
  → SecretScanner 扫描 diff（reject if secret found）
  → TargetBranchAllowlist 检查（reject if not in allowlist）
  → WorkspaceGate 验证（reject if workspace not bound）
  → ForcePushDetection（reject if force push detected）
  → RollbackPlan 生成
  → GitWriteSafetyGate 评估（13+ gates）
  → HumanApproval（用户当轮独立确认）
  → OneShotApprovalToken 生成
  → GitWriteAdapter.dryRun()（git add --dry-run + git commit --dry-run）
  → GitWriteAdapter.execute()（实际 git add + git commit）
  → AuditEvent 记录（append-only）
  → PostWriteVerification（确认 commit 存在、diff 匹配）
  → Cleanup（token 失效、rollback plan 保存）
```

### 4.3 禁止操作清单

| # | 禁止操作 | 原因 |
|---|---------|------|
| 1 | `git push --force` / `git push -f` | 不可逆覆盖远程历史 |
| 2 | `git reset --hard` | 不可逆丢失本地变更 |
| 3 | `git rebase` | 改写历史，风险高 |
| 4 | `git stash`（自动） | 可能隐藏用户变更 |
| 5 | `git tag`（自动） | 标签不应自动创建 |
| 6 | `git branch -D` / `git branch --delete` | 不可逆删除分支 |
| 7 | `git checkout` / `git switch`（自动切换分支） | 可能丢失未提交变更 |
| 8 | CI 自动触发（如 push 触发 GitHub Actions） | 除非用户显式配置 |
| 9 | 自动创建 PR | 必须人工触发 |
| 10 | 自动 merge PR | 必须人工触发 |

### 4.4 Feature Flag 设计

```text
PRODUCT_RUNTIME_GIT_WRITE_ENABLED: bool = False           # 总开关，默认关闭
GIT_WRITE_ALLOWED_BRANCHES: list[str] = []                 # 空白名单 = 全禁止
GIT_WRITE_MAX_FILES_PER_COMMIT: int = 50                   # 单次提交最大文件数
GIT_WRITE_MAX_DIFF_SIZE_BYTES: int = 1_000_000             # 单次 diff 最大字节数
GIT_WRITE_REQUIRE_SIGNED_COMMITS: bool = False             # 是否需要签名提交
GIT_WRITE_APPROVAL_TOKEN_TTL_SECONDS: int = 300            # 审批 token 有效期 5 分钟
GIT_WRITE_AUTO_PUSH_AFTER_COMMIT: bool = False             # commit 后是否自动 push
GIT_WRITE_AUTO_CREATE_PR: bool = False                     # 是否自动创建 PR
```

### 5.2 各子阶段详细说明

#### GitWrite-A（本文档）

- **输入**: P1-P9 scoped closure Pass、P4 Human Approval Gate Pass
- **产出**: GitWrite-A ledger、现状审计矩阵、安全模型草案、子阶段拆分、Gate 草案
- **不产出**: 任何代码、domain、service、API、frontend
- **Gate**: 本文档完成即 GitWrite-A Pass

#### GitWrite-B：GitWriteIntent / GitWritePreview / GitWriteApproval 纯 domain contract

- **输入**: GitWrite-A ledger、P4 domain models
- **产出**:
  - `GitWriteIntent` 域模型（intent_id / workspace_id / target_branch / files / commit_message / author）
  - `GitWritePreview` 域模型（diff_summary / file_list / estimated_commit_hash / safety_flags）
  - `GitWriteApproval` 域模型（approval_id / intent_id / scope: git_write_execute / one_shot_token / expires_at / consumed_at）
  - `GitWriteSafetyGate` 域模型（16 gates from §4.1）
  - `OneShotApprovalToken` 域模型
- **不产出**: service 实现、API、Git 命令执行
- **Gate**: domain contract freeze 完成，无安全字段泄露

#### GitWrite-C：dry-run diff/commit preview service（no write）

- **输入**: GitWrite-B domain contracts
- **产出**:
  - `GitWritePreviewService`（生成 diff preview，不执行）
  - `SecretScanner`（扫描 diff 内容检测 secret）
  - `TargetBranchValidator`（检查 target branch 是否在 allowlist）
  - `ForcePushDetector`（检测是否有 force push）
  - `RollbackPlanGenerator`（生成回滚计划）
- **不产出**: 真实 git add/commit/push
- **Gate**: service 通过 targeted tests，不触发任何 Git 写

#### GitWrite-D：read-only API + approval readback（no write）

- **输入**: GitWrite-C services
- **产出**:
  - `POST /git-write/intents` — 创建写意图（返回 preview，不执行）
  - `GET /git-write/intents/{id}` — 查看写意图 + preview

## 11. GitWrite-C-R1 preview-ready / write-ready gate 语义修正

GitWrite-C 原始实现存在 preview-ready / write-ready gate 语义混淆：`GitWritePreviewService` 在 preview 阶段把 `HUMAN_APPROVAL` 与 `ONE_SHOT_TOKEN` 直接标为 Pass，导致 `GitWriteSafetyGateSnapshot.all_passed=True`，容易把“preview 可展示”误读为“已审批 / 已可写”。

R1 已修正：

- `GitWriteSafetyGateSnapshot` 新增 preview-scope gate 判断：`preview_gates_passed()`。
- preview-scope gate 排除 `HUMAN_APPROVAL` 与 `ONE_SHOT_TOKEN`。
- `GitWritePreview.status == READY` 只代表 preview ready，可进入后续展示 / API 阶段。
- `HUMAN_APPROVAL` 与 `ONE_SHOT_TOKEN` 在 GitWrite-C preview 阶段保持 `PENDING`。
- pending 的 human approval / one-shot token 不阻塞 preview ready，但会让 full write gate `all_passed=False`。
- approved intent / approved decision 仍必须要求 full `all_passed=True`，不得只凭 `preview_gates_passed()` 通过。

| `runtime/orchestrator/app/services/git_write_preview_service.py` | 修改 | preview 阶段 human approval / one-shot token 改为 pending；preview status 使用 preview-scope gate |
