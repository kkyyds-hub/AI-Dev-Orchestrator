## 5. 当前路线 P2～P7 能解决什么，不能解决什么

### 5.1 逐阶段分析

| 阶段 | 能解决什么 | 不能解决什么 |
|------|----------|------------|
| **P2: 前端只读展示 + Runtime 使用 workspace_path** | 把执行上下文与既有 worktree 生命周期接上；让现有 agent 视图做成可信只读观察面 | 不会自动长出 conversation list、director inbox 和跨页面消息路由 |
| **P3: AI runtime in worktree** | 让 `RuntimeCreateConfig.workspacePath` 在 AI-Dev 模型体系里有对应落点，实现 AO 式 runtime/workspace 耦合执行 | 不会解决"像 GPT 一样持续对话" |
| **P4: 失败回流** | 把失败从日志变成可治理对象，向 AO 的 lifecycle/reactions 靠近 | 只能做到"失败可回流"，不能替代 human challenge + conversation 闭环 |
| **P5: worktree diff → commit candidate** | 把 worktree 内发生了什么变成可审阅、可存档、可审批的中间产物 | 不会自动变成"会和你讨论为什么要改计划的主管" |
| **P6: 审批/commit/delivery** | 把"关键动作需用户审批"原则扩大到更完整的交付治理 | 审批链与会话链是两条不同闭环；没有 Conversation Hub，系统更安全但不会更像主管 |
| **P7: 前端可观测** | 让用户"看到" agent 在做什么、哪里卡住、workspace 状态 | 前端可观测不等于 GPT 式交互体验；没有后端对话路由/上下文汇总/提案应用链路，前端只是强观测面板 |

### 5.2 关键结论

**P2～P7 是理想 AI 主管的必要条件，但不是充分条件。** 除非额外建设 Conversation Hub，否则系统最终仍会更接近一个能执行代码任务的 Agent 编排系统，而不是理想中的 AI 项目主管。

UAT 能验证"现有链路是否闭环"，能证明 worktree / runtime / failure / diff / approval / delivery 是否串起来，但**不能自动补齐一个根本没有建设的 Conversation Hub**。它可以发现缺陷，不能凭空发明缺失的中枢层。

## 6. 距离理想 AI 主管还缺的关键环节

### 6.1 12 个缺失/待强化环节

| # | 环节 | 当前证据 | 为什么必需 | 不做会造成什么 |
|---|------|---------|----------|-------------|
| 1 | **ProjectDirectorConversation** — AI 主管对话会话 | `runtime/orchestrator/app/domain/project_director_session.py` 存在但偏 intake/clarification 模式 | 所有讨论必须有一个长期存在的归宿 | 工作台对话只能是阶段性表单或单次消息流 |
| 2 | **DirectorConversationList** — 工作台对话列表 | **未发现** | 理想中的主管不是"一条单线程聊天" | 跨主题、跨阶段、跨页面的问题只能混在一个线程里 |
| 3 | **DirectorInbox** — 跨页面消息汇总 | `apps/web/src/features/agents/components/BossInterventionForm.tsx` 存在但对 Director 无汇总 | 其他页面新增质疑/note 需统一汇总给 Director | 用户必须手动在多个页面来回搬运上下文 |
| 4 | **ConversationRouter** — 对话路由 | `runtime/orchestrator/app/domain/project_director_message.py` 有关联字段但无路由层 | 没有 router 就不可能实现跨页面自动归并 | 无法控制消息归属 |
| 5 | **DirectorContextAssembler** — 现状汇总器 | `runtime/orchestrator/app/services/context_builder_service.py` 等通用服务存在，但无 Director 专用汇总器 | "现在项目进展是什么"是聚合问题 | 主管回答只能是松散猜测或页面抄写 |
| 6 | **UserChallenge / Intervention** — 用户质疑机制 | `apps/web/src/features/agents/components/BossInterventionForm.tsx` 存在；`runtime/orchestrator/app/domain/agent_session.py` 有 `latest_intervention_type` | 质疑需结构化类型 + Director 接收处理 + 工作台归档 | 用户质疑只能是局部中断操作 |
| 7 | **PlanRevision** — 计划变更机制 | `runtime/orchestrator/app/domain/project_director_plan_version.py` + `change_plan.py` 存在 | 讨论结果要能变成计划修订提案并审批应用 | 工作台讨论停在意见层，不影响真实项目 |
| 8 | **DirectorActionProposal** — AI 主管行动建议 | **未发现统一对象** | AI 主管不能直接执行关键动作，应先产出可审批提案 | 主管建议分散在不同页面和 DTO 里，难以统一治理 |
| 9 | **ProposalApproval** — 建议审批应用 | `runtime/orchestrator/app/domain/approval.py` 存在；`WorktreePlan`/`WorktreeCleanup` 有 `requires_user_confirmation` | 需从 workspace 级扩展到 Director 提案级 | 主管退化成"告诉你去点哪个按钮" |
| 10 | **Conversation-to-Task / Conversation-to-Plan** — 影响链路 | **未发现** (`ProjectDirectorMessage` 有关联字段但无应用链) | 主管不只是聊天，讨论结果需作用到真实编排 | 工作台对话与执行系统是两套平行宇宙 |
| 11 | **工作台主对话和 AgentSession/Task/Run/Approval/Deliverable 的关联模型** | `runtime/orchestrator/app/domain/project_director_message.py` 有 `related_plan_version_id`、`related_project_id`、`related_task_id` | 工作台要成为总入口，需要系统性关联全套对象 | 工作台只是总览页，不是主管对话中枢 |
| 12 | **前端工作台多会话交互** | `apps/web/src/pages/workbench/WorkbenchPage.tsx` 更像单页布局，不是多会话系统 | 理想工作台需要手动开新对话、自动追加相关对话、查看对话列表 | 工作台更像总览页，不像主管对话中枢 |

### 6.2 归纳

真正缺的不是"小模块再多几个"，而是一个新层级：**Project Director Conversation Hub**。这一层要站在现有 `Project / Task / Run / Worker / AgentSession / Approval / Deliverable / RepositoryWorkspace` 之上，而不是替换它们；它负责汇总、路由、提案、审批应用和对话治理。

## 8. 不能误判的地方

| # | 误判 | 正确理解 |
|---|------|---------|
| 1 | **P1 Pass = AI Project Director Pass** | P1 是 Coding Session + Worktree Lifecycle 机制级闭环。AI Project Director 总闭环需要 conversation hub + runtime + delivery + UAT。 |
| 2 | **有 AgentMessage audit = 有工作台对话中枢** | Audit 是 timeline 事件记录。对话中枢需要 conversation list + inbox + router + context assembler。 |
| 3 | **有 AgentSession = 有 ProjectDirectorConversation** | `AgentSession` 偏执行会话与 workspace/runtime 观测；`ProjectDirectorSession` 偏澄清目标与主管消息。二者职责不同。 |
| 4 | **有失败日志 = 有用户质疑机制** | AO 的 lifecycle/reactions 处理自动反馈路由；理想 challenge 机制还需"用户从任意页面质疑 → Director 接收 → 核对现状 → 建议 → 讨论 → 应用"。 |
| 5 | **有 worktree create = AI runtime 能编码** | 当前 AI-Dev 强证据在 `dry_run / plan_hash / confirmation / preflight / guarded create / guarded cleanup`；AO 有 `Runtime.create / sendMessage / isAlive / getOutput / workspacePath`。两者不能画等号。 |
| 6 | **有 commit candidate = 能 PR** | AO 的 `SCM` 接口覆盖 PR lifecycle、CI、review、mergeable 检测等完整链条。AI-Dev 有 `commit_candidate`、`approval`、`deliverable` 但无完整 PR 管理链路。 |
| 7 | **前端只读展示 = GPT 式交互体验** | 可观测性与会话治理不是一回事。没有对话列表、收件箱、路由和提案应用，前端只是强大浏览器。 |
| 8 | **复用 AO 机制 = 自动得到理想 AI 主管** | AO 定位是"parallel AI agents 的 orchestration layer"，不是"GPT 式可持续主管对话中枢"。即使学完整，也只会靠近强执行编排器。 |

---

## 9. 建议后的阶段规划

不改变当前主路线，但必须把"理想 AI 主管体验"显式补成后续阶段。

| 阶段 | 目标 | 说明 |
|------|------|------|
| **P2** | 前端只读展示 + Runtime 使用 `workspace_path` | 执行上下文与 worktree 生命周期接上；现有 agent 视图做成可信只读观察面 |
| **P3** | AI runtime in worktree | AO 式 runtime/workspace 耦合执行 |
| **P4** | 失败回流 | 把失败从日志变成可治理对象，向 AO lifecycle/reactions 靠近 |
| **P5** | worktree diff → commit candidate | 把执行结果变成 reviewable、approvable 的候选产物 |
| **P6** | 审批 / commit / delivery | 把"关键动作需用户审批"扩大到完整交付治理 |
| **P7** | **Project Director Conversation Hub** | 建设 `ProjectDirectorConversation`、`ConversationList`、`Inbox`、`Router`、`ContextAssembler`、`UserChallenge`、`PlanRevision`、`DirectorActionProposal`、`ProposalApproval` 和 conversation-to-plan/task 影响链路 |
| **P8** | 端到端 UAT / 总 Gate | 验证"讨论—提案—审批—应用—执行—回流—再讨论"的完整总闭环 |

**P7 Conversation Hub 不能被忘记**，否则系统最后会变成"能执行的编排器"，但不是"可讨论、可质疑、可影响计划的 AI 项目主管"。

---

## 10. 最终结论

### 10.1 如果只做完 P2～P6

AI-Dev-Orchestrator 会接近一个**能执行代码任务、能做工作区隔离、能做审批与交付治理的 Agent 编排系统**。这条路线本身是对的，而且与 P1 worktree lifecycle 基础设施是连续的、相容的。

### 10.2 但它仍然不是理想中的 AI 项目主管

原因不在于 execution layer 不强，而在于理想中的体验要求一个额外层级：要能持续讨论、能承接用户质疑、能从其他页面自动汇总消息、能基于真实状态给建议、并把讨论结果作用到计划/任务/审批/重试/暂停上。

### 10.3 明确结论

**当前主路线仍应该继续复用 Agent Orchestrator 的优秀机制，但必须把 Project Director Conversation Hub 作为后续明确阶段纳入闭环。** 不这样做，系统会越来越像一个强大的执行编排器，却不会真正达到理想中的"像 GPT 一样可持续讨论、可质疑、可影响计划、所有消息汇总到工作台的 AI 项目主管"。

### 10.4 Gate

| Gate | 结论 |
|------|------|
| AI Project Director ideal-gap evidence report | **Pass** |
| Coding Session + Worktree Lifecycle P1 机制级闭环 | **Pass** (保持) |
| AI Project Director 总闭环 | **仍为 Partial**，不能标记为 Pass |
