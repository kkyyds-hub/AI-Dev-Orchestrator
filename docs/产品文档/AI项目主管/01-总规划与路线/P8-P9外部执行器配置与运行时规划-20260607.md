## 2. P8 定义：Executor Config & Runtime Adapter Design

### 2.1 P8 是什么

P8 只做配置、发现、抽象、launch preview。**不真实启动执行器。**

P8 回答的问题：
- 系统里有哪些可用的外部执行器？
- 每个执行器的能力边界是什么？
- 如何只读发现本地 CLI 安装状态和原生配置？
- 如何生成 launch command preview 而不执行？
- Settings 页面如何展示执行器就绪状态？

### 2.2 P8 必须覆盖的域模型

#### ExecutorRegistry

```text
ExecutorRegistry
├── 管理所有已注册的 ExecutorProfile
├── 支持按 capability 查询可用执行器
├── 支持按 executor_id 获取单个 profile
└── 注册来源：built-in manifest + project managed config
```

#### ExecutorProfile

```text
ExecutorProfile
├── executor_id: str                    # "codex" | "claude_code" | "deepseek"
├── display_name: str                   # "OpenAI Codex" | "Claude Code" | "DeepSeek"
├── description: str
├── provider: str                       # "openai" | "anthropic" | "deepseek"
├── binary_name: str                    # "codex" | "claude" | N/A (DeepSeek 走 API)
├── binary_discovery_strategy: str      # "path_lookup" | "managed_path" | "api_only"
├── capabilities: ExecutorCapability
├── config_discovery: ExecutorConfigDiscovery
├── permission_model: str               # "default_deny" | "accept_edits" | "bypass_permissions"
├── status: ExecutorStatus              # "available" | "not_installed" | "not_configured" | "unknown"
└── created_at / updated_at
```

### 2.3 P8 不做

| # | 不做 | 原因 |
|---|------|------|
| 1 | 真实启动外部执行器（codex / claude 进程） | 属于 P9 |
| 2 | 执行任何 CLI 命令 | P8 只做 preview |
| 3 | 写入原生配置文件 | 只读发现 |
| 4 | 自动安装执行器 | 用户自行管理 |
| 5 | 实现 executor session / run state | 属于 P9 |
| 6 | terminal log capture | 属于 P9 |
| 7 | activity detection | 属于 P9 |
| 8 | cost / token usage 回填 | 属于 P9 |

### 2.4 P8 安全边界

| # | 规则 |
|---|------|
| 1 | Secret / token / API key 不明文暴露在任何 response 或日志中 |
| 2 | 原生配置文件内容不完整 dump 到前端 |
| 3 | 环境变量完整 dump 禁止 |
| 4 | 只读发现原生配置时不做任何写入 |
| 5 | launch preview 只生成命令字符串，不调用 subprocess / exec |
| 6 | feature flag + guardrail 控制 executor config 功能是否启用 |
| 7 | human confirmation 在进入 P9 真实启动前必须先通过 |

## 3. P9 定义：Controlled External Agent Runtime

### 3.1 P9 是什么

P9 才允许设计受控启动外部执行器。P9 的核心是"受控"——有 feature flag、有用户确认、有 workspace 验证、有成本限制、有并发限制、有超时、有 cleanup。

### 3.2 P9 必须覆盖的域模型

#### ExecutorSession

├── started_at: datetime | None
├── finished_at: datetime | None

### 3.3 Activity Detection 设计

参考 Agent Orchestrator 的 activity detection 模式：

| 检测来源 | Claude Code | Codex | 通用 fallback |
|---------|------------|-------|---------------|
| Process probe | `isClaudeProcessAlive()` | `isProcessRunning()` | `ps` / `/proc` 检查 |
| Native JSONL | `~/.claude/projects/{slug}/*.jsonl` | `~/.codex/sessions/**/*.jsonl` | N/A |
| Hook events | `~/.claude/projects/{slug}/.claude/hooks/*.jsonl` | N/A | N/A |
| Terminal output | `classifyTerminalOutput()` | `detectActivity()` | regex pattern matching |
| State derivation | 6-state: active/idle/waiting_input/blocked/exited/error | approval_request/error/tool_call/active/idle/exited | idle/active/waiting_input/exited |

P9 应设计统一的 `ExecutorActivityDetector` 接口，各执行器适配器实现各自的检测逻辑：

```text
ExecutorActivityDetector
├── probe_process(pid: int) → ProcessProbeResult
├── read_native_session_file(workspace_path: str, session_id: str) → ActivityState | None
├── parse_terminal_output(output: str) → ActivityState
├── classify_state(probe: ProcessProbeResult, native: ActivityState | None, terminal: ActivityState) → ActivityDetection
└── last_activity_at(session: ExecutorSession) → datetime | None
```

### 3.4 Cost / Token Usage 回填

| 执行器 | 数据来源 | 回填目标 |
|--------|---------|---------|

### 3.5 P9 不做

| # | 不做 | 原因 |
|---|------|------|
| 1 | 产品运行时 git add / commit / push / PR / merge | 全线禁止 |
| 2 | 自动启动执行器不经用户确认 | 必须 feature flag + 用户确认 |
| 3 | 绕过 P4/P5/P6/P7 已建立的 evidence / gate | 不可绕过 |
| 4 | 执行器自动获得 Git 写权限 | 默认 deny |
| 5 | AI 主管偷偷启动执行器 | 只可建议，不可执行 |
| 6 | 用户确认"启动执行器"被理解为授权 git push / merge | 确认范围严格限定 |
| 7 | 无限并发 / 无限预算 | 必须有限制 |
| 8 | 无超时运行 | 必须可配置超时 |

### 3.6 P9 安全边界

| # | 规则 |
|---|------|
| 1 | P9 启动必须 feature flag + 用户确认 + workspace 验证 + 成本限制 + 并发限制 |
| 2 | 默认禁止产品运行时 git add / commit / push / PR / merge |
| 3 | 外部执行器不能绕过 P4/P5/P6/P7 已建立的 evidence / gate |
| 4 | AI 主管可以建议启动执行器，但不能偷偷启动 |
| 5 | 用户确认"启动执行器"不等于授权 git push / merge |
| 6 | 执行器 session 必须可被 user interrupt / timeout / cleanup |
| 7 | 执行器进程退出后必须清理 workspace 绑定 |
