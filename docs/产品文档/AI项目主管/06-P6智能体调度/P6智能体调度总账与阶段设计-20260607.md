## 2. P6 核心定位

### 2.1 P6 是什么

P6 Agent Orchestration 的目标是让 AI Project Director 具备调度决策能力——不是自动执行调度，而是生成结构化调度建议。

P6 的核心工作是：在 P5 FailureRecoveryDecision 的基础上，生成 `AgentDispatchDecision`，回答"这个任务下一步应该由谁来做"，并把决策建议透传到前端供用户只读查看。

P6 不自动派发 Codex / DeepSeek，不自动创建任务，不自动启动 Worker。调度建议是只读的、可审计的、需要用户确认的。

### 2.2 P6 与 P5 的边界

| P5（已完成） | P6（本阶段） |
|-------------|-----------|
| 生成 FailureRecoveryDecision | 基于 FailureRecoveryDecision 生成 AgentDispatchDecision |
| 判断 recommended_owner | 转化为 dispatch recommended_agent |
| 生成 next_instruction_kind | 转化为 instruction_kind + instruction_draft |
| 判断 retry_allowed | 作为 dispatch_status 的输入信号 |
| 写入 failure recovery audit event | 写入 dispatch audit event |
| 前端只读展示 recovery decision | 前端只读展示 dispatch decision |

### 2.3 P6 与 P7 的边界

| P6（本阶段） | P7（后续） |
|-------------|-----------|
| 生成 dispatch decision | Project Director Conversation Hub 中讨论调度决策 |
| 推荐 agent 分工 | 用户通过 Conversation Hub 质疑/确认调度建议 |
| 前端只读展示 dispatch suggestion | Conversation Hub 中用户与 AI 主管协作调整调度 |
| 写入 dispatch audit event | 主管根据 conversation 修改调度 |
| 调度建议可审计 | 用户质疑/修改调度建议 |

**重要澄清**：

- P6 不做自动派发。自动派发 Codex / DeepSeek、自动创建任务、自动启动 Worker 均不在 P6 范围。
- P7 当前定义是 Project Director Conversation Hub + Governance，**不等于自动派发**。
- 未来如需进入真实自动调度阶段，必须另行设计 feature flag、guardrail、human confirmation、安全 Gate 和 evidence 链。不得把自动派发写成 P7 的既定范围。
- 不得把 AI Project Director 总闭环写成 Pass。

## 5. P6 对 P5 FailureRecoveryDecision 的复用关系

### 5.1 复用字段

P6 AgentDispatchDecision 从 P5 FailureRecoveryDecision 复用以下字段：

| P5 字段 | P6 复用方式 | 说明 |
|---------|-----------|------|
| `recommended_owner` | 映射为 `recommended_agent` | codex / deepseek / user / blocked |
| `next_instruction_kind` | 复用为 `instruction_kind` | code_fix / test_fix / config_fix / evidence_fix / replay / pause / replan / human_question |
| `next_instruction_draft` | 复用为 `instruction_draft` | 只读透传已有中文指令草案 |
| `requires_human_decision` | 映射为 `dispatch_status` 的 needs_user_decision | 需要人工决策的调度不自动建议 |
| `retry_allowed` | 映射为 dispatch safety 信号 | 自动 retry 不在当前任何阶段范围，P6 只标记不可执行 |
| `recoverable` | 映射为 dispatch safety 信号 | 不可恢复的调度建议 blocked |
| `safety_flags` | 合并到 P6 dispatch safety_flags | 保留 P5 安全标记，叠加 P6 调度 safety |
| `rule_codes` | 作为内部判断依据 | 不直接变成用户误导文案 |
| `evidence_refs`（P5 中 rule_codes 承载） | 转为 `evidence_refs` | 调度建议可追溯到 P5 recovery 证据 |

### 5.2 不直接暴露的字段

以下 P5 内部字段在 P6 前端展示中**不直接暴露**给用户（与 P5-F R1 保持一致）：

| 字段 | 处理方式 |
|------|---------|
| `recommended_owner`（enum 原值） | 映射为中文 label 后展示 |
| `next_action`（enum 原值） | 映射为中文 label 后展示 |
| `failure_category`（enum 原值） | 不展示 |
| `reason_code`（enum 原值） | 不展示 |
| `rule_codes`（内部码） | 不展示，仅用于内部判断 |
| `version` | 不展示 |
| `audit_event_type` | 不展示 |

## 6. AgentDispatchDecision 设计草案

### 6.1 建议域模型

```text
AgentDispatchDecision
├── dispatch_decision_id: str
├── source_failure_recovery_decision_id: str | None
├── source_run_id: UUID | None
├── source_task_id: UUID | None
├── recommended_agent: DispatchAgent
├── dispatch_status: DispatchStatus
├── dispatch_reason_code: str
├── dispatch_reason_cn: str
├── instruction_kind: InstructionKind
├── instruction_draft: str | None
├── evidence_refs: list[str]
├── safety_flags: AgentDispatchDecisionSafetyFlags
├── created_at: datetime
├── created_by: str
└── api_response_exposed: bool
```

### 6.2 DispatchAgent 枚举草案

```text
DispatchAgent
├── codex: 代码实现、后端 domain、API、Worker、前端实现、复杂测试修正
├── deepseek: 文档、ledger、证据整理、测试结果回填、路线梳理、设计说明
├── user: 需要真实业务判断、授权、确认、缺少关键信息、成本/风险选择
└── blocked: 违反安全 gate、缺证据、缺前置状态、产品运行时 Git 写风险、总闭环误判风险
```

### 6.3 DispatchStatus 枚举草案

```text
DispatchStatus
├── suggested: 调度建议已生成，AI 主管推荐此 agent
├── needs_user_decision: 调度无法自动判断，需要用户决策
├── blocked: 违反安全 gate 或缺少前置条件，不可调度
└── not_applicable: 当前不适用调度（如成功路径）
```

### 6.4 AgentDispatchDecisionSafetyFlags 草案

```text
AgentDispatchDecisionSafetyFlags
├── runs_git: bool = False
├── runs_write_git: bool = False
├── git_add_triggered: bool = False
├── git_commit_triggered: bool = False
├── git_push_triggered: bool = False
├── pr_opened: bool = False
├── merge_triggered: bool = False
├── branch_deleted: bool = False
├── git_reset_triggered: bool = False
├── git_checkout_triggered: bool = False
├── git_switch_triggered: bool = False
├── git_stash_triggered: bool = False
├── git_rebase_triggered: bool = False
├── git_tag_triggered: bool = False
├── ci_triggered: bool = False
├── execution_enabled: bool = False
├── worker_dispatch_triggered: bool = False
├── api_response_exposed: bool = False
├── agent_message_written: bool = False
├── task_created: bool = False
├── retry_triggered: bool = False
└── auto_dispatch_triggered: bool = False
```

所有 safety_flags 在 P6 纯域模型中默认为 `False`。P6-E 透传时 `api_response_exposed` 设为 `True`。

### 6.5 Codex / DeepSeek / user / blocked 判定边界

#### Codex 判定

| 触发条件 | 说明 |
|---------|------|
| 代码实现缺陷 | 后端 / 前端 / API / Worker 逻辑错误 |
| 测试失败 | 单元测试 / targeted test 失败 |
| 类型错误 | Python type hint / TypeScript type 错误 |
| 输出格式问题 | Provider 返回格式不合法 |
| 复杂测试修正 | 需要修改测试代码本身 |
| 域模型实现 | domain model 代码实现 |
| Provider 调用失败（可重试型） | 临时性网络/服务错误 |

#### DeepSeek 判定

| 触发条件 | 说明 |
|---------|------|
| 文档不一致 | 设计文档与实现偏离 |
| 证据链断裂 | evidence 缺失或不完整 |
| Ledger 过期 | 阶段总账状态未更新 |
| Gate 状态错误 | Gate 结论与证据不匹配 |
| 测试结果回填 | 将测试结果整理为证据写入 ledger |
| 路线梳理 | 规划文档更新 |
| 设计说明 | 设计方案文档化 |
| 配置修正 | 验证模板/规则配置错误 |

#### user 判定

| 触发条件 | 说明 |
|---------|------|
| 需求不清 | 任务需求不明确，需要补充验收标准 |
| 预算变更 | 需要确认是否增加预算或更换模型 |
| 权限决策 | 涉及密钥、权限变更 |
| 安全边界决策 | 高风险方向改变 |
| 超过预设重试阈值 | 连续失败超过 N 次 |
| 缺少关键信息 | AI 主管无法根据现有 evidence 给出可信判断 |
| 成本/风险选择 | 涉及成本或风险 trade-off |
| 项目方向改变 | 当前计划方向可能需要调整 |

#### blocked 判定

| 触发条件 | 说明 |
|---------|------|
| 违反安全 gate | runtime_launch_gate / delivery_gate 未通过 |
| 缺证据 | 缺少必要的 evidence 链 |
| 缺前置状态 | P5 failure recovery decision 未生成 |

| 总闭环误判风险 | 可能把 Partial 误判为 Pass |

| Feature flag 未开启 | 所需功能未启用 |
| 当前阶段不允许继续 | 阶段 Gate 未通过 |
