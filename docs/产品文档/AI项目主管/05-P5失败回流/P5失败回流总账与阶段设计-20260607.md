## 2. P5 核心定位

### 2.1 P5 是什么

P5 Failure Recovery 的目标不是把失败丢给用户判断。

P5 的目标是让 AI Project Director 先生成**结构化 failure recovery decision**：AI 主管默认判断失败类型、是否可恢复、是否允许重试、推荐交给 Codex / DeepSeek / User / Blocked、下一步动作和下一条修复指令草案。

用户只处理高风险、不可自动判断、需要权限/预算/需求澄清/项目方向改变的决策。

### 2.2 P5 与 P6 的边界

| P5（本阶段） | P6（后续） |
|-------------|-----------|
| 生成 recovery decision | 自动派发执行 |
| 判断推荐 owner | 自动路由到正确角色 |
| 生成 next instruction draft | 自动创建/更新任务 |
| 写入 audit event | 主管根据证据自动决策 |
| 前端只读展示 | 主管主动调度 |

P5 只生成 recovery decision 和 instruction draft，不自动派发执行。自动派发属于 P6。

### 2.3 参考机制

参考 Agent Orchestrator lifecycle-manager 的三个核心理念（不照搬代码）：

| AO 机制 | P5 映射 |
|---------|--------|
| Reaction engine：对状态变更触发自动反应 | P5 对 run.failure_category 变更触发 recovery decision |
| Auto-handle vs escalate：可自动处理 vs 升级到人工 | P5 的 recommended_owner=codex/deepseek vs user/blocked |
| Lifecycle reason：结构化原因码，不丢失信息 | P5 的 reason_code + failure_category 联合推导 |

### 5.2 当前缺口

当前系统有丰富的失败信号，但**没有统一的 P5 failure recovery decision contract**。

缺失的能力：

| 缺失 | 说明 |
|------|------|
| 没有 `recoverable` 判断 | 当前无法从 failure_category 推导是否可恢复 |
| 没有 `retry_allowed` 判断 | 重试上限超限和预算超限的语义不同，但没有统一判断 |
| 没有 `recommended_owner` | 不知道应该交给 Codex、DeepSeek、用户还是阻塞 |
| 没有 `next_action` | 下一步应该 retry / fix / pause / replan / wait_human |
| 没有 `next_instruction_draft` | 没有可复制给执行端的修复指令草案 |
| 没有 `requires_human_decision` | 哪些失败必须用户决策 |
| 没有 recovery audit event | 失败回流建议不可审计 |

## 6. P5 Failure Recovery Decision 草案

### 6.1 建议域模型

```text
FailureRecoveryDecision
├── failure_category: RunFailureCategory
├── reason_code: TaskBlockingReasonCode | None
├── recoverable: bool
├── retry_allowed: bool
├── recommended_owner: RecoveryOwner
├── next_action: RecoveryNextAction
├── next_instruction_kind: InstructionKind
├── next_instruction_draft: str | None
├── requires_human_decision: bool
├── human_decision_reason: str | None
├── user_visible_summary_cn: str
└── audit_event_type: str
```

### 6.2 RecoveryOwner 枚举草案

```text
RecoveryOwner
├── codex: 代码修复（后端/前端/测试/类型/实现缺陷）
├── deepseek: 文档/证据/ledger/审计/规划修正/Gate 证据整理
├── user: 需求不清/预算变更/权限/安全边界/高风险方向改变
└── blocked: 信息不足/安全禁止/依赖缺失/当前阶段不允许继续
```

### 6.3 RecoveryNextAction 枚举草案

```text
RecoveryNextAction
├── retry: 允许自动或手动重试当前任务
├── fix_and_retry: 需要先生成修复指令，然后重试
├── pause_and_wait: 暂停并等待外部条件满足
├── replan: 建议重新规划任务或子任务
├── escalate_to_human: 升级到用户决策
├── block_permanently: 永久阻塞，不应重试
└── archive: 归档为已知不可修复
```

### 6.4 InstructionKind 枚举草案

```text
InstructionKind
├── code_fix: 代码修复指令
├── test_fix: 测试修复指令
├── config_fix: 配置修复指令
├── evidence_fix: 证据/文档修正指令
├── replay: 回放/重新执行指令
├── pause: 暂停建议
├── replan: 重新规划建议
└── human_question: 需要用户回答的问题
```

## 7. recommended_owner 规则草案

### 7.1 决策矩阵

| failure_category | 典型原因 | recommended_owner | 说明 |
|-----------------|---------|-------------------|------|
| EXECUTION_FAILED | Provider 异常、模型错误、代码缺陷 | codex | 执行层修复 |
| EXECUTION_FAILED | 输出格式错误、验证解析失败 | codex | 输出格式修复 |
| VERIFICATION_FAILED | 质量门未通过、输出不合规 | codex | 质量修复 |
| VERIFICATION_CONFIGURATION_FAILED | 验证模板/规则配置错误 | deepseek | 配置修正 |
| DAILY_BUDGET_EXCEEDED | 日预算耗尽 | user | 预算决策 |
| SESSION_BUDGET_EXCEEDED | 会话预算耗尽 | user | 预算决策 |
| RETRY_LIMIT_EXCEEDED | 重试次数超限 | user | 超过阈值需人工 |
| EXECUTION_FAILED | 依赖缺失、工作区异常 | blocked | 外部条件不满足 |
| — | 需求不清、目标矛盾 | user | 需用户澄清 |
| — | Git 写操作被阻断 | blocked | 安全边界 |

### 7.2 owner 规则细化

| owner | 触发条件 |
|-------|---------|
| **codex** | failure 来自代码实现缺陷、测试失败、类型错误、输出格式问题、provider 调用失败（可重试型） |
| **deepseek** | failure 来自文档不一致、证据链断裂、ledger 过期、Gate 状态错误、规划与实现偏离 |
| **user** | 需求不清、预算/权限/安全边界决策、高风险方向改变、超过预设重试阈值、连续失败超过 N 次 |
| **blocked** | 信息不足、安全禁止、依赖缺失、feature flag 未开启、当前阶段不允许继续 |

## 8. requires_human_decision 规则草案

### 8.1 默认值

`requires_human_decision` 默认 `false`。AI 主管先判断，不需要用户为每个失败决策。

### 8.2 必须为 true 的场景

只有以下情况 `requires_human_decision` 才为 `true`：

| # | 场景 | human_decision_reason 示例 |
|---|------|--------------------------|
| 1 | 需要改变项目方向 | "失败表明当前计划方向可能需要调整" |
| 2 | 需要用户补充需求 | "当前任务需求不明确，需要补充验收标准" |
| 3 | 涉及预算/密钥/权限 | "需要用户确认是否增加预算或更换模型" |
| 4 | 涉及产品运行时 Git 写操作（当前不执行） | "需要用户确认是否允许进入写入前 guardrail" |
| 5 | 涉及删除、覆盖、reset、merge、push、PR | "需要用户确认高风险仓库操作" |
| 6 | AI 主管无法根据现有 evidence 给出可信判断 | "现有证据不足以判断下一步" |
| 7 | 连续失败超过预设阈值 | "任务已连续失败 N 次，需要人工判断是否继续" |

### 8.3 安全约束

即使 `requires_human_decision=true`，也只能请求用户决策，**不能自动执行高风险动作**。必须等待用户显式确认后才能进入下一阶段。

## 9. P5 子阶段规划

| 子阶段 | 内容 | 负责人 | 说明 |
|--------|------|--------|------|
| **P5-A** | Ledger + Stage Design | DeepSeek | 本文档 |
| **P5-B** | FailureRecoveryDecision 纯域模型 + targeted tests | Codex | 域模型实现 |
| **P5-C** | Worker failed run → recovery decision builder | Codex | 从 Worker 失败路径生成 recovery decision |
| **P5-D** | AgentMessage / audit event 记录失败回流建议 | Codex | 将 recovery decision 写入可审计事件 |

| **P5-F** | 最小前端只读展示 / P5 closure | Codex → DeepSeek | 前端展示 recovery decision；Gate 证据收口 |

### 9.1 P5 子阶段顺序

```text
P5-A（本文档）
  ↓
P5-B FailureRecoveryDecision 纯域模型
  ↓
P5-C Worker failure → recovery decision builder
  ↓
P5-D Audit event / AgentMessage
  ↓
P5-E WorkerRunOnceResponse 只读透传
  ↓
P5-F 前端只读展示 / P5 final closure
```
