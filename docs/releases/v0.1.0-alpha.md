# AI-Dev-Orchestrator v0.1.0-alpha - Maintainer Preview

> **Draft only.** This file prepares release notes for maintainer review. It does
> not indicate that a tag, package, artifact, or GitHub Release has been published.

## Maturity

`v0.1.0-alpha` is proposed as an **Early-stage / Maintainer Preview /
Experimental** snapshot. It is suitable for local inspection and contribution,
not for production or security-sensitive deployment without independent review
and additional isolation.

## Verified Implemented Capabilities

- FastAPI backend and React/Vite local control surface for project, task, run,
  approval, repository, deliverable, agent, cost, and console workflows.
- SQLite-backed domain state, JSONL run logs, and audit-oriented application
  messages.
- Simulated and local-command execution, verification templates, task routing,
  retry limits, budget guards, and local worker concurrency.
- Mock and OpenAI-compatible provider service abstractions and masked provider
  configuration summaries.
- External-executor preflight, launch, supervision, timeout, cleanup, and readback
  components with dry-run and controlled-smoke paths.
- Sandbox-oriented workspace roots, path checks, manifests, candidate-file
  writes, and diff-generation stages for selected Project Director flows.
- Read-only reviewer handoff, transport, output validation, disposition, and
  freshness-consumption stages for selected candidate-diff flows.
- Human approval, escalation, blocking-reason, and evidence-record structures.

These statements describe repository code and focused tests, not universal
end-to-end or production guarantees.

## Security And Review Controls

- Explicit allow/deny fields and blocked-state results in guarded contracts.
- Backend-owned sandbox workspace roots and path-containment checks.
- Diff, prompt, scope, schema, and review-fingerprint binding in selected review
  chains.
- Freshness revalidation and single-consumption records for selected review
  dispositions.
- Timeouts, reviewer output limits, process-supervisor and cleanup components.
- Human confirmation and audit-oriented records for selected transitions.

Application-level controls do not replace OS/container isolation. Independent
review is not universal, local evidence is not tamper-proof against host
compromise, and AI-generated changes still require human inspection. Read
[Security Policy](../../SECURITY.md) and [Threat Model](../THREAT_MODEL.md).

## Development Setup

Requirements: Git, Python 3.11-3.13, `uv`, Node.js, and npm.

Backend:

```bash
cd runtime/orchestrator
RUNTIME_DATA_DIR="$(mktemp -d)" uv run --no-project --with-editable . \
  python -m uvicorn app.main:app --host 127.0.0.1 --port 8011
```

Web control surface:

```bash
cd apps/web
npm ci
VITE_BACKEND_URL=http://127.0.0.1:8011 npm run dev
```

Isolated simulated backend smoke:

```bash
cd runtime/orchestrator
uv run --no-project --with-editable . \
  python scripts/p9_run_backend_runnable_smoke.py --json
```

See the root [README](../../README.md) for configuration and trust boundaries.

## Known Limitations

- The complete backend pytest readiness run passes 3,472 tests but emits 3,005
  warnings. Warning reduction remains maintenance work; see
  [Project Status](../PROJECT_STATUS.md).
- The current npm lockfile reports eight audit findings, including three high
  findings in the production-dependency view. No dependency upgrade is included
  in this documentation-only preparation branch.
- There is no committed CI, stable package distribution, automated release
  pipeline, or supported deployment profile.
- Authentication, authorization, multi-tenancy, and hardened remote operation
  are not established.
- Native executor and reviewer behavior depends on local tools, explicit flags,
  and host permissions. Many tests use fakes, dry-run, or controlled smoke.
- Workspace containment is an application policy, not an OS sandbox guarantee.
- Provider calls require operator credentials and have external privacy, account,
  availability, and model-compatibility dependencies.
- Some cost values are estimates, not reconciled billing data.
- GitHub private vulnerability reporting is not currently enabled.

## Compatibility Notes

- Backend package metadata declares Python `>=3.11,<3.14`.
- The web application uses the committed npm lockfile and is not published as a
  package.
- Runtime state uses local SQLite and filesystem data. This draft does not define
  a migration or long-term data compatibility promise.
- Provider services use OpenAI-compatible endpoints, but compatibility with a
  specific third-party endpoint or model must be verified by its operator.
- This open-source readiness work does not change APIs, schemas, dependencies,
  provider integrations, or runtime behavior.

## Upgrade Guidance

There is no prior public release to upgrade from. For evaluation:

1. clone the exact tag only after the maintainer publishes one;
2. use a new disposable runtime data directory;
3. do not copy secrets or production repositories into the evaluation environment;
4. start with simulated execution and the isolated smoke workflow;
5. review configuration and provider data before enabling external calls.

## Rollback Guidance

Because no migration or release is performed by these notes, rollback means
stopping local processes and returning to a previously reviewed commit. Back up
runtime data before experimenting. Do not reuse runtime data across revisions if
schema compatibility has not been verified.
