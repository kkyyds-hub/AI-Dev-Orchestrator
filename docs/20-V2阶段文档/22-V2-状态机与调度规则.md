# V2-A 状态机与调度规则

- 项目：`AI-Dev-Orchestrator`
- 阶段：`V2-A`
- 文档版本：`v1.3`
- 冻结日期：`2026-03-10`
- 文档性质：**阶段实施说明文档**
- 目标：统一 `Task / Run / Event` 的状态语义、允许转移路径、非法转移处理方式和后续服务拆分边界。

## 0. 本阶段要解决什么

1. 把当前散落在 `TaskService`、`TaskWorker`、`TaskRepository`、`TaskRouterService`、`ContextBuilderService` 里的状态规则收敛成统一口径。
2. 明确 `Task` 关注“整体推进状态”，`Run` 关注“单次执行尝试状态”，`Event` 关注“状态变化与执行过程的观测信号”。
3. 为 Day 2 的状态守卫与统一入口改造冻结文件命名、职责边界和非法转移处理方式。

## 1. 当前状态现状盘点

### 1.1 `TaskStatus` 当前真实状态

| 状态 | 当前语义 | 当前主要写入入口 | 当前是否可直接进入路由 |
| --- | --- | --- | --- |
| `pending` | 任务处于待调度队列，允许被 `Router` 评估 | `TaskService.create_task`、`TaskService.retry_task`、`TaskService.resolve_human_review`、`TaskService.resume_task` | **条件允许时可** |
| `running` | 任务已经被 `Worker` 认领，正在执行本轮尝试 | `TaskRepository.claim_pending_task` | **否** |
| `paused` | 任务被人工显式暂停，不应再被调度 | `TaskService.create_task`、`TaskService.pause_task` | **否** |
| `waiting_human` | 任务正在等待人工介入或确认 | `TaskService.create_task`、`TaskService.request_human_review`、`TaskService.resume_task` | **否** |
| `completed` | 任务已成功完成当前目标 | `TaskWorker._finalize_execution` | **否** |
| `failed` | 本轮执行或验证失败，等待人工决定是否重试或转人工 | `TaskWorker._finalize_execution` | **否** |
| `blocked` | 执行前被预算 / 重试守卫阻断 | `TaskWorker._finalize_guard_blocked_run` | **否** |

### 1.2 `TaskHumanStatus` 当前控制维度

> 当前任务是否可执行，并不只由 `TaskStatus` 决定，还受 `human_status` 与 `paused_reason` 共同影响。

| 字段值 | 当前语义 | 对调度的影响 |
| --- | --- | --- |
| `none` | 没有人工介入标记 | 不额外阻塞 |
| `requested` | 已请求人工处理但未完成 | 当前视为阻塞信号 |
| `in_progress` | 人工处理中 | 当前视为阻塞信号 |
| `resolved` | 人工处理已完成 | 可恢复正常调度，并在路由打分中有轻微正向加成 |

### 1.3 `RunStatus` 当前真实状态

| 状态 | 当前语义 | 当前主要写入入口 | 备注 |
| --- | --- | --- | --- |
| `queued` | 预留的排队状态 | **当前未使用** | Day 1 仅冻结，不在本轮启用 |
| `running` | 单次执行尝试已创建并开始 | `RunRepository.create_running_run` | 与 `TaskStatus.running` 配对出现 |
| `succeeded` | 单次执行尝试成功完成 | `RunRepository.finish_run` | 仅在执行成功且验证通过 / 跳过时出现 |
| `failed` | 单次执行尝试失败 | `RunRepository.finish_run` | 包含执行失败和验证失败 |
| `cancelled` | 单次执行尝试被守卫阻断取消 | `RunRepository.finish_run` | 当前主要来自预算 / 重试守卫 |

### 1.4 `RunFailureCategory` 当前失败分类

| 分类 | 当前归属 | 当前典型来源 |
| --- | --- | --- |
| `execution_failed` | 执行失败 | `Executor` 返回失败 |
| `verification_failed` | 验证失败 | `Verifier` 校验未通过 |
| `verification_configuration_failed` | 验证配置失败 | `Verifier` 内部分类 |
| `daily_budget_exceeded` | 日预算耗尽 | `BudgetGuardService` |
| `session_budget_exceeded` | 会话预算耗尽 | `BudgetGuardService` |
| `retry_limit_exceeded` | 任务重试次数超限 | `BudgetGuardService` |

### 1.5 `Event` 当前观测类型

| `run_updated` | `EventStreamService.publish_run_updated` | 运行记录变化 |
| `log_event` | `RunLoggingService.append_event` | 运行过程结构化日志 |

## 2. 当前真实触发入口

| 触发方 | 当前入口 | 当前作用 |
| --- | --- | --- |
| API / Planner | `TaskService.create_task` | 根据 `human_status` 与 `paused_reason` 推导初始状态 |
| Console / API | `TaskService.retry_task` | 允许 `failed / blocked -> pending` |
| Console / API | `TaskService.pause_task` | 允许 `pending / failed / blocked -> paused` |
| Console / API | `TaskService.resume_task` | 允许 `paused -> pending / waiting_human` |
| Console / API | `TaskService.request_human_review` | 允许 `pending / failed / blocked / paused -> waiting_human` |
| Console / API | `TaskService.resolve_human_review` | 允许 `waiting_human -> pending` |
| Router | `TaskRouterService.route_next_task` | 只评估 `pending` 任务，不直接修改状态 |
| Repository | `TaskRepository.claim_pending_task` | 允许 `pending -> running` |
| Worker | `TaskWorker._finalize_guard_blocked_run` | 允许 `running -> blocked` 且 `run -> cancelled` |
| Worker | `TaskWorker._finalize_execution` | 允许 `running -> completed / failed` 且 `run -> succeeded / failed` |
| Context Builder | `ContextBuilderService._build_blocking_reasons` | 计算“是否可执行”和阻塞原因，不直接改状态 |

## 3. `Task` 状态转移矩阵（冻结）

### 3.1 允许转移

| From | Trigger | To | 触发方 | 当前入口 | 说明 |
| --- | --- | --- | --- | --- | --- |
| `new` | 创建普通任务 | `pending` | API / Planner | `TaskService.create_task` | 默认入口 |
| `new` | 创建时已带暂停说明 | `paused` | API / Planner | `TaskService.create_task` | 由 `_derive_initial_status` 推导 |
| `new` | 创建时 `human_status=requested / in_progress` | `waiting_human` | API / Planner | `TaskService.create_task` | 人工信号优先级高于暂停说明 |
| `pending` | `Worker` 认领 | `running` | Worker | `TaskRepository.claim_pending_task` | 当前通过仓储直接落库 |
| `running` | 执行成功且验证通过 / 无验证 | `completed` | Worker | `TaskWorker._finalize_execution` | 成功闭环 |
| `running` | 执行失败 | `failed` | Worker | `TaskWorker._finalize_execution` | 失败分类为 `execution_failed` |
| `running` | 验证失败 | `failed` | Worker | `TaskWorker._finalize_execution` | 失败分类走验证分类 |
| `running` | 预算 / 重试守卫阻断 | `blocked` | Worker / Budget Guard | `TaskWorker._finalize_guard_blocked_run` | 当前在认领后才阻断 |
| `failed` | 手动重试 | `pending` | Console / API | `TaskService.retry_task` | 受 `BudgetGuardService.build_retry_status` 限制 |
| `blocked` | 手动重试 | `pending` | Console / API | `TaskService.retry_task` | 受同一重试上限限制 |
| `pending` | 手动暂停 | `paused` | Console / API | `TaskService.pause_task` | 进入显式暂停态 |
| `failed` | 手动暂停 | `paused` | Console / API | `TaskService.pause_task` | 先停止后续处理 |
| `blocked` | 手动暂停 | `paused` | Console / API | `TaskService.pause_task` | 人工保留现场 |
| `paused` | 手动恢复且无人工阻塞 | `pending` | Console / API | `TaskService.resume_task` | 清空 `paused_reason` 后恢复路由 |
| `paused` | 手动恢复但 `human_status=requested / in_progress` | `waiting_human` | Console / API | `TaskService.resume_task` | 当前分支存在，属于保守恢复 |
| `pending` | 请求人工处理 | `waiting_human` | Console / API | `TaskService.request_human_review` | 进入人工等待态 |
| `failed` | 请求人工处理 | `waiting_human` | Console / API | `TaskService.request_human_review` | 失败转人工 |
| `blocked` | 请求人工处理 | `waiting_human` | Console / API | `TaskService.request_human_review` | 守卫阻断转人工 |
| `paused` | 请求人工处理 | `waiting_human` | Console / API | `TaskService.request_human_review` | 暂停任务转人工 |
| `waiting_human` | 人工处理完成 | `pending` | Console / API | `TaskService.resolve_human_review` | 同时设置 `human_status=resolved` |

### 3.2 当前存在的“非转移型阻塞”

> 当前系统里还有一类非常重要的规则：**任务保持 `pending`，但因为阻塞原因而暂时不可执行。**

| 当前状态 | 阻塞原因 | 当前行为 | 当前入口 |
| --- | --- | --- | --- |
| `pending` | 依赖任务未全部 `completed` | 继续保持 `pending`，`Router` 跳过 | `ContextBuilderService._build_blocking_reasons` |
| `pending` | `human_status=requested / in_progress` | 继续保持 `pending`，`Router` 跳过 | `ContextBuilderService._build_blocking_reasons` |
| `pending` | 存在 `paused_reason` 残留说明 | 继续保持 `pending`，`Router` 解释阻塞原因 | `ContextBuilderService._build_blocking_reasons` |

### 3.3 当前明确不允许的转移

1. `completed` 当前视为终态，不提供恢复、暂停、转人工、重试入口。
2. `running` 当前不允许通过控制台直接暂停、转人工或重试，必须等待 `Worker` 完成本轮落库。
3. `waiting_human` 当前不允许再次 `pause` 或 `retry`，必须先 `resolve-human` 回到 `pending`。
4. 任意未在表内声明的手动转移，都按非法转移处理。

## 4. `Run` 状态转移矩阵（冻结）

| From | Trigger | To | 当前入口 | 说明 |
| --- | --- | --- | --- | --- |
| `new` | 创建运行记录 | `running` | `RunRepository.create_running_run` | 当前直接创建为运行中 |
| `running` | 预算 / 重试守卫阻断 | `cancelled` | `RunRepository.finish_run` | 与 `TaskStatus.blocked` 配对 |
| `running` | 执行成功且验证通过 / 无验证 | `succeeded` | `RunRepository.finish_run` | 成功收口 |
| `running` | 执行失败 | `failed` | `RunRepository.finish_run` | 对应 `execution_failed` |
| `running` | 验证失败 | `failed` | `RunRepository.finish_run` | 对应验证类失败 |

补充规则：

1. `RunStatus.queued` 当前仅保留，不参与实际流转。
2. `RunStatus.cancelled` 当前不计入 `count_execution_attempts_by_task_id`，因此不会被当成一次真正执行尝试。
3. `Run` 一旦进入 `succeeded / failed / cancelled` 即视为终态，不做 reopen。

## 5. 失败、阻塞与人工介入主分流规则（冻结）

### 5.1 执行失败分流

1. `Executor` 失败：`Task -> failed`，`Run -> failed`，`failure_category=execution_failed`
2. `Verifier` 失败：`Task -> failed`，`Run -> failed`，`failure_category` 复用验证分类
3. 失败任务后续只允许三种保守动作：
   - `retry`
   - `pause`
   - `request-human`

### 5.2 守卫阻断分流

1. 预算超限或重试超限时，当前不是保持 `pending`，而是先让任务进入 `blocked`。
2. 同时本轮 `Run` 会落为 `cancelled`，并记录具体 `failure_category`。
3. `blocked` 后续只允许三种保守动作：
   - `retry`
   - `pause`
   - `request-human`

### 5.3 人工介入分流

1. 人工介入统一收敛到 `waiting_human`。
2. 人工处理完成统一回到 `pending`，并写入 `human_status=resolved`。
3. `resolved` 当前不是任务主状态，而是路由与观测层的辅助信号。

## 6. 非法转移处理约定（冻结）

1. 当前手动动作的非法转移在 `TaskService` 中抛出 `ValueError`。
2. API 层将该类错误统一映射为 `HTTP 409 Conflict`。
3. 仓储层 `set_status` / `update_control_state` 当前不承担状态守卫，只承担持久化。
4. Day 2 起应把“是否允许转移”的判断集中进单独状态机服务，避免规则继续散落。

## 7. 文件职责与命名收口（冻结）

### 7.1 计划新增文件

1. `runtime/orchestrator/app/services/task_state_machine_service.py`
2. `runtime/orchestrator/app/services/task_readiness_service.py`

### 7.2 职责冻结

| 文件 | 责任 |
| --- | --- |
| `task_state_machine_service.py` | 维护 `Task / Run` 合法转移矩阵、触发方、错误信息和统一转移入口 |
| `task_readiness_service.py` | 维护依赖就绪判断、阻塞原因生成和“是否可路由”判断 |
| `task_service.py` | 保留 API 动作编排职责，不再继续堆积状态判断细节 |
| `task_worker.py` | 负责执行循环与状态机调用，不继续内联扩散规则 |
| `event_stream_service.py` | 负责事件发布，不承担状态决策 |

### 7.3 外部接口命名冻结

1. 保留显式动作接口：
   - `POST /tasks/{task_id}/retry`
   - `POST /tasks/{task_id}/pause`
   - `POST /tasks/{task_id}/resume`
   - `POST /tasks/{task_id}/request-human`
   - `POST /tasks/{task_id}/resolve-human`
2. 不新增万能状态修改接口，例如：`PATCH /tasks/{task_id}/status`
3. 后续控制台与路由解释能力都应复用同一套状态机 / 就绪判断结果。
