# V2 实施排期与每日计划

- 项目：`AI-Dev-Orchestrator`
- 阶段：`V2`
- 模块：`调度 / 状态机 / 预算 / 观测 / 复盘`
- 文档版本：`v1.0`
- 冻结日期：`2026-03-10`
- 文档性质：**V2 实施排期冻结文档**
- 目标：把 `V2` 从阶段路线拆成按天可执行的计划，并统一目录、文件、接口和命名规范。

---

## 0. 范围定位（冻结）

1. `V2` 的重点不是新增大量功能，而是把现有 `V1` 闭环做得更稳、更清晰、更可解释。
2. 本文档负责冻结 `V2` 的总天数、阶段顺序、每日主题和交付边界。
3. 本文档不替代每日计划；每日实现时仍以对应日期目录下的 `01-今日计划.md` 为准。

---

## 1. 排期原则（冻结）

1. `V2` 按 `V2-A / V2-B / V2-C` 三个阶段推进，不交叉并行开发。
2. 每天只解决一个明确主题，避免同一天同时推进状态机、预算、UI 和并发。
3. 每天都要有可交付物，交付物至少包含以下之一：`代码`、`接口`、`文档`、`验证记录`。
4. 每个阶段结束时必须有一次收口日，用来做统一验证和文档冻结。
5. 如果某天未完成，优先顺延一天，不在当天硬塞后续主题。

---

## 2. 命名与接口规范（冻结）

### 2.1 文档目录命名

1. 每日计划目录统一使用：`YYYY-MM-DD-V2阶段主题中文名`
2. 每日计划主文件统一使用：`01-今日计划.md`
3. 后续如需补充完成记录，可在同目录追加：`02-完成记录.md`

### 2.2 Step 编码规范

1. `P2A1-S1` 表示：`V2-A` 第 `1` 天的第 `1` 个步骤。
2. `P2B3-S2` 表示：`V2-B` 第 `3` 天的第 `2` 个步骤。
3. 所有 `V2` 日计划统一使用该编码规则，避免重复编号。

### 2.3 代码文件命名

1. 文档标题与目录使用中文，便于管理阶段计划。
2. 代码文件继续使用英文 `snake_case`，例如 `task_state_machine_service.py`。
3. 新增服务文件要体现职责，不使用模糊命名，例如禁止 `helper.py`、`manager.py` 这类过宽名称。

### 2.4 接口命名规范

1. 外部接口优先使用“显式动作”命名，不新增含义模糊的万能接口。
2. 任务动作继续保持显式语义，例如：`POST /tasks/{task_id}/pause`。
3. 新增控制台聚合和观测接口时，优先收敛到 `runtime/orchestrator/app/api/routes/console.py`。
4. 新增接口名必须能直接回答“这是给谁看的”和“返回什么”，禁止无语义缩写。

## 4. Phase `V2-A`：调度与状态机强化排期

- 阶段目标：把 `Task / Run / Event` 的状态语义、转移入口、依赖判断和人工恢复规则统一下来。
- 阶段产出：
  1. `docs/20-V2阶段文档/22-V2-状态机与调度规则.md`
  2. `runtime/orchestrator/app/services/task_state_machine_service.py`
  3. `runtime/orchestrator/app/services/task_readiness_service.py`
  4. `V2-A` 最小验证记录
- 阶段 DoD：
  1. 主要状态转移路径可列成表并可解释。
  2. 阻塞原因有统一口径，不再分散在多个服务里隐式判断。
  3. 人工介入、暂停、恢复、重试都能被统一规则解释。

---

## 5. Phase `V2-B`：成本、路由与观测增强排期

- 阶段目标：把模型路由、预算守卫、成本指标和失败分布从“内部逻辑”提升为“用户可见信息”。
- 阶段产出：
  1. `docs/20-V2阶段文档/23-V2-路由预算与观测策略.md`
  2. `runtime/orchestrator/app/api/routes/console.py`
  3. `runtime/orchestrator/app/services/console_metrics_service.py`
  4. `apps/web/src/features/console-metrics/*`
- 阶段 DoD：
  1. 用户可以看到为什么任务被这样路由。
  2. 用户可以看到预算使用和失败分布。
  3. 超预算时系统有清晰的降级或阻断行为。

---

## 6. Phase `V2-C`：复盘体系与有限并行排期

- 阶段目标：在不牺牲恢复能力和成本控制的前提下，增强复盘能力并引入有限并行。
- 阶段产出：
  1. `docs/20-V2阶段文档/24-V2-复盘与有限并行方案.md`
  2. `runtime/orchestrator/app/services/failure_review_service.py`
  3. `runtime/orchestrator/app/services/decision_replay_service.py`
  4. `runtime/orchestrator/app/services/worker_slot_service.py`
- 阶段 DoD：
  1. 同类失败可以被追踪和归类。
  2. 历史决策可以回放，不再只看最终结果。
  3. 单机有限并行不会破坏状态一致性和预算控制。

---

## 7. 计划使用方式（冻结）

1. 每天开工前，先打开对应日期目录下的 `01-今日计划.md`。
2. 当天如需调整，只允许微调步骤内容，不允许直接跨阶段切换主题。
3. 每个阶段最后一天必须补充实际完成证据和顺延说明。
4. 如果出现新的重大瓶颈，先更新本总计划，再修改后续每日计划。

---

## 8. 最终结论

`V2` 先冻结为 `13` 天实施排期，并按 `V2-A -> V2-B -> V2-C` 的顺序推进。  
后续开发优先遵守本排期文档和对应日期的每日计划，不再临时拆散主题。
