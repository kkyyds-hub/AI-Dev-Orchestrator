# 结构减债 vs 整站翻修判断规则

## 1. 目的

避免把本应低风险推进的结构治理，做成高风险整站翻修。

## 2. 属于结构减债的动作

以下动作默认属于当前 skill 可处理范围：

- 抽 section / panel / component
- 下沉 hooks / api / types / lib
- 补 `data-testid`
- 保持接口合同不变的代码整理
- 保持入口关系基本不变的文件拆分
- 页面 owner 收口
- 说明联调 / 测试 / 文档影响

## 3. 属于整站翻修的动作

以下动作默认不应在当前 skill 中推进：

- 重建导航体系
- 大改路由关系
- 重排页面主入口
- 全局视觉统一与重做
- 大改标题文案体系
- 整体信息架构重构

## 4. 快速判断问题

动手前先问：

1. 这次会不会改用户主路径？
2. 这次会不会改变页面 owner？
3. 这次会不会让现有 smoke / 证据脚本大面积失效？
4. 这次是不是主要在调整视觉，而不是收口结构？
5. 这次是不是要重建页面关系而不是抽文件？

如果大多回答“会”，这就已经越过结构减债边界了。

## 5. 一句话纪律

当前 skill 只做“减债”，不做“翻修”。
