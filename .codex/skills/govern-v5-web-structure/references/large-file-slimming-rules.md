# 大文件瘦身规则

## 1. 目的

把“前端大文件瘦身”从个人感觉，收敛成可重复执行的工程规则。

## 2. 核心原则

### 原则 1：瘦身优先于翻修
当前优先目标是：
- 减少文件膨胀
- 收口职责
- 下沉逻辑
- 降低继续失控风险

当前不优先：
- 视觉统一
- 页面重设计
- 导航翻新

### 原则 2：瘦身时尽量保持行为不变
默认应尽量保持：
- 接口合同不变
- 关键按钮文案不变
- 页面入口顺序基本不变
- 联调路径不变
- 当前 feature owner 不乱跳

## 3. 触发瘦身的典型信号

遇到下面任一情况，应优先考虑瘦身：

- `App.tsx` 或 `ProjectOverviewPage.tsx` 继续新增一整块结果区 / 表格区 / 详情区
- 页面中同时出现请求、字段转换、状态逻辑和大段 JSX
- 同一页面出现多个重复的小工具函数、badge tone 映射、字段拼装逻辑
- 新增逻辑会让入口页继续明显变胖
- 测试对页面结构和文案越来越脆弱

## 4. 强制拆分门槛

满足任意一条，默认优先拆分：

- 单文件新增超过 120 行结构性 UI 代码
- 单文件新增两个及以上大区块
- 新增逻辑同时涉及请求、状态、渲染三类职责
- 该区块天然可以成为独立 section / panel
- 该区块已经开始被测试或浏览器证据单独关注

## 5. 推荐瘦身动作

### 动作 A：抽 section
适合：
- 首页某个独立结果区
- 某个 feature 下的大块渲染区

### 动作 B：抽 panel / card / component
适合：
- 可复用或半复用的区块
- 结构稳定但内容较长的视图块

### 动作 C：下沉 hooks
适合：
- 查询
- mutation
- 状态机
- 条件分支多的行为逻辑

### 动作 D：下沉 lib
适合：
- 纯格式化
- 字段拼装
- tone / label 映射
- URL / hash 处理

## 6. 禁止动作

以下动作不应伪装成“瘦身”：

- 一边抽文件，一边大改站点导航
- 一边拆 section，一边重做所有文案
- 一边下沉逻辑，一边改掉接口合同
- 以“清理代码”为名，把用户路径一起改掉

## 7. 一句话纪律

瘦身的本质是“收纳和分层”，不是“借机翻新整栋楼”。
