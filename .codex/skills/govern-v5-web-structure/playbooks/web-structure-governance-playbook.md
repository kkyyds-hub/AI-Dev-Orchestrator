# V5 前端结构治理 playbook

## 1. 目的

把 V5 期间的前端结构治理收敛成一个可重复执行的最小工作流，避免每次都靠临场发挥决定“该不该拆、怎么拆、会不会影响联调”。

## 2. 标准流程

### 步骤 1：先界定这是结构治理，不是视觉重做
先写清：
- 当前问题是什么
- 当前不做什么
- 为什么这次属于低风险瘦身

### 步骤 2：锁定最小治理切片
推荐切片：
- 一个过胖区块
- 一组重复逻辑
- 一个应该抽出的 section
- 一组应补的测试锚点

### 步骤 3：先看入口页边界
确认：
- `App.tsx` 是否继续承担了不该承担的逻辑
- `ProjectOverviewPage.tsx` 是否继续堆了不该堆的区块
- 本次能否把逻辑下沉，而不改变外部行为

### 步骤 4：选拆分落点
从下面四类里选：
- section
- component / panel
- hooks / api / types
- lib

### 步骤 5：执行最小拆分
要求：
- 保持接口合同不变
- 尽量保持关键文案不变
- 尽量保持联调路径不变
- 保持页面入口关系基本不变

### 步骤 6：补测试锚点
为本次涉及的关键区域补充或统一 `data-testid`。

### 步骤 7：评估联调 / 测试 / 文档影响
至少写清：
- 联调是否受影响
- 现有 smoke / 浏览器证据是否需同步注意
- 是否需要文档回填

### 步骤 8：做最小验证
至少推荐：
- `npm run build`
- 最小页面路径自检
- 对关键 testid 或关键区块做一次核对

## 3. 最小输出要求

每次线程结束时，至少输出：

- 本轮治理对象
- 文件级改动
- 行为保持不变的说明
- 联调/测试/文档影响
- 验证结果
- 下一线程建议

## 4. 红线提醒

- 不要在瘦身线程里顺手大改导航
- 不要把视觉重做混进来
- 不要让入口页继续做胖
- 不要不补锚点就拆页面
- 不要改完后不给 verify / docs 交接说明
