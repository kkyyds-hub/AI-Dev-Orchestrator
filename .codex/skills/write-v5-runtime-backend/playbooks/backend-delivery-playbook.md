# V5 后端交付 playbook

## 1. 目标

把一次 V5 后端改动，收敛成 **对象清楚、主链可落、兼容性可交代、最小验证可复述** 的后端交付切片。

## 2. 先把任务翻译成后端对象

开始前先明确：

1. 属于哪个 Phase / 工作包
2. 最终落在哪些对象上：worker、service、route、domain、repository、schema、日志还是脚本
3. 现有代码里最接近的能力是什么
4. 这次是沿已有结构扩展，还是确实需要新增一条后端子链

## 3. 先确认当前真实状态

至少确认：

- 当前仓库是否已有半成品逻辑
- 当前 worker 主链是否已经调用到该能力
- 当前 API 是否已有入口或观察面
- 当前 DB / JSON / 日志落点分别在哪里
- 当前前端是否已经依赖某些字段

不要只靠猜测开始写。

## 4. 锁定单线程可交付切片

每轮尽量只选一个清晰切片，例如：

- 给 `ExecutorService` 引入 provider adapter 抽象
- 给 `TaskWorker` 默认接入 `include_project_memory`
- 给 `Run` / 运行日志增加 provider usage 与 model tier 口径
- 给 checkpoint / session / team control 增加最小 domain + route + service 主链

切片必须回答：

- 入口在哪
- 主处理链在哪
- 结果存在哪
- 谁会消费这个结果
- 失败时如何降级或回退

## 5. 先列文件级改动，再动手

至少明确：

- 要改哪些文件
- 每个文件承担什么职责
- 哪些是新增文件，哪些是扩展旧文件
- 是否需要 migration 或数据兼容
- 是否需要 DTO / route / worker / script 同步

## 6. 显式处理兼容性与迁移风险

必须检查：

- 是否需要 migration
- 老数据是否允许为空
- 新字段是否需要默认值
- API 响应是否会破坏旧前端
- `simulate` 和真实 provider 模式如何共存
- 失败时是否能退回旧链路

## 7. 做最小验证

最低建议：

- 至少做一次导入级或脚本级验证
- 如改了后端启动入口，确认应用可启动
- 如改了 route，至少做一次请求级验证或构造数据验证
- 如改了 worker 主链，至少跑一次最短链路
- 如改了 schema / 持久化，至少确认写入与读取没有明显断裂

## 8. 线程结束前自检

结束前至少确认：

- 我有没有把任务翻译成明确后端对象？
- 我有没有沿现有结构增量扩展，而不是平行造系统？
- 我有没有交代 migration / 默认值 / 回退逻辑？
- 我有没有做最小验证，而不是空口宣布完成？
- 我有没有给出下一棒 owner？
