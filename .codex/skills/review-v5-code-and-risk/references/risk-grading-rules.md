# V5 风险分级规则

## 1. 目的

统一 `review-v5-code-and-risk` 的风险表达，避免所有问题都被写成同一等级。

## 2. 高风险

满足任一情况，可优先标高风险：

- 会导致功能不可用
- 会导致数据错误或状态错误
- 会让阶段判断明显失真
- 主链实际上没接上，却被当成已落地
- schema / migration / 默认值缺失会直接打坏旧数据或旧页面
- 文档或页面把未验证内容写成已完成

### 高风险输出要素

- 风险点
- 影响范围
- 为什么是高风险
- 建议先交给哪个 skill 修

## 3. 中风险

适用于：

- 当前也许能跑，但扩展成本高
- 架构边界模糊，后续容易分叉
- 字段与页面勉强可用，但口径不够稳
- 需要补 verify 或补文档，否则后续易误判

## 4. 低风险

适用于：

- 命名、组织方式、局部重复
- 注释不足
- 风格不统一但不直接影响交付判断

## 5. 结论表达建议

优先写成：

- 当前不建议进入 accept，先回实现
- 当前不建议冻结为已完成，先补 verify
- 当前结构基本可靠，但仍有中风险待补
- 当前可继续推进，但需要记录低风险尾项

不要只写“有点问题”或“整体还行”。
