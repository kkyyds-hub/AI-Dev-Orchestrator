# V5 风险审查 playbook

## 1. 目标

把一次 V5 代码与实现审查，收敛成 **范围明确、风险分级清楚、修复归属明确、可直接接力** 的审查结果。

## 2. 先界定审查边界

开始前先说清：

1. 当前审查的是哪个 Phase / 工作包
2. 当前审查哪些目录 / 文件
3. 这次更重架构、数据、兼容性，还是更重口径与合同一致性

不要无限扩大审查范围。

## 3. 先看是否贴合 V5 真实缺口

优先确认：

- 这次改动是否对应母本里的真实缺口
- 是否贴合正确的 Phase / 工作包
- 有没有把工作包做偏
- 有没有把展示态误写成运行态落地

如果方向本身就偏了，再精致的实现也有风险。

## 4. 检查架构、schema 与主链风险

重点检查：

- 是否沿现有 `service / route / domain / repository / worker / feature` 体系增量扩展
- DB 字段是否真实存在，是否需要 migration
- 默认值 / 空值 / 旧数据兼容是否考虑到
- worker / route / page 主链是否真的接上
- 是否只有 DTO 或页面，实际上没走主链

## 5. 检查口径一致性

重点看：

- 前端字段与后端字段是否一致
- 文档有没有夸大完成程度
- verify 结论是否真的覆盖实现声明
- 页面上的“已完成 / 已接入 / 可用”等表述是否诚实

## 6. 做风险分级

建议至少分为：

- 高风险：会导致不可用、数据错误、阶段判断失真、主链断裂
- 中风险：当前也许能跑，但扩展成本高、兼容性差、容易翻车
- 低风险：命名、风格、局部重复、可后补的小问题

不要把所有问题都写成同一等级。

## 7. 写修复归属

每个高 / 中风险最好补：

- 影响范围
- 为什么是风险
- 建议修复方向
- 建议由哪个 skill 接手

本 skill 不只挑毛病，还要让下游知道该怎么接。

## 8. 线程结束前自检

结束前至少确认：

- 我有没有界定审查范围？
- 我有没有检查架构、schema、兼容性与主链风险？
- 我有没有检查前后端 / 文档 / verify 口径一致性？
- 我有没有做高 / 中 / 低风险分级？
- 我有没有写清修复归属？
