# V5 交接与冻结检查清单

## 1. 用途

用于在 V5 文档线程结束前，检查当前结果是否已经具备：

- 正式冻结资格
- 诚实回填资格
- 可交给下一线程直接接手的资格

## 2. 线程结束前必须明确的 8 项

### 2.1 阶段定位

- 当前属于哪个 Phase
- 属于哪个工作包
- 对应母本哪一节

### 2.2 改动范围

- 改了哪些正式文档
- 关联了哪些代码目录/文件
- 哪些范围明确不在本轮内

### 2.3 真实状态

只能使用：

- 已完成
- 已实现待验证
- 已规划
- 进行中
- 阻塞

### 2.4 风险说明

至少写：

- 未处理风险
- 兼容性风险
- 假设条件
- 是否依赖后续 skill 才能闭环

### 2.5 验证证据

至少说明以下之一：

- 已有 build/test/API/页面/日志证据
- 目前缺少哪些证据
- 为什么当前只能写成待验证/进行中/阻塞

### 2.6 下一步建议

必须明确下一线程先接哪个 skill。

推荐写法：

- 下一线程 owner：`write-v5-runtime-backend`
- 如需跨层升级：`drive-v5-orchestrator-delivery`
- 如需先验证事实：`verify-v5-runtime-and-regression`

### 2.7 文档回填位置

必须说明：

- 是否已经回填到 `docs/01-版本冻结计划/V5/`
- 还是仍停留在临时草案
- 如未回填，阻塞原因是什么

### 2.8 冻结结论

要明确当前文档属于哪一类：

- 正式冻结基线
- 工作中回填
- 执行前交接说明
- 待验证暂存结论

## 3. 何时允许写“已完成”

同时满足以下条件时才允许：

1. 已有明确实现或明确文档收口结果
2. 已有最低验证证据，或该项本身纯属文档治理且已完成正式回填
3. 没有把跨线程才能闭环的部分偷偷吞进“已完成”

## 4. 何时只能写“已实现待验证”

出现下列任一情况：

- 代码已改，但运行证据不足
- 文档已写，但相关实现尚未验证
- 已能看见主链雏形，但没有足够证据证明稳定

## 5. 何时只能写“已规划”

出现下列任一情况：

- 只有切片与实施方案
- 尚未有实际实现
- 当前线程的主要价值只是把边界、owner、完成口径写清楚

## 6. 交接建议模板

```md
## 交接路线
- 当前状态：
- 下一线程 owner：
- 下一线程开始前先读：
- 最低需要验证：
- 当前不要误判为完成的点：
```

## 7. 冻结时的常见误判

### 误判 1：把“有 UI 壳子”写成已完成

纠正：没有真实状态或真实接口，不算完成。

### 误判 2：把“有 service 草案”写成已完成

纠正：没有 worker/API/route/调用链接入，不算完成。

### 误判 3：把“有桌面草案”写成正式冻结

纠正：没有回填仓库正式目录，不算正式冻结。

### 误判 4：把“某一侧完成”写成“整包完成”

纠正：跨层包必须明确剩余侧面和下一线程 owner。

## 8. 文档线程的 done checklist

- 已绑定 V5 母本
- 已绑定仓库正式冻结目录
- 已说明正式 / 草案边界
- 已说明当前状态词及其依据
- 已说明风险与缺口
- 已说明下一线程 owner
- 已说明最小验证要求
- 已避免夸大完成度
- 已让后续线程可以直接接力
