# V5 冻结文档线程 playbook

## 1. 目标

把一次 V5 的计划 / 冻结 / 回填 / 交接文档任务，收敛成 **口径清楚、状态诚实、边界明确、可直接接力** 的正式文档结果。

## 2. 先判文档类型

先明确当前属于哪一种：

1. 总纲型
2. 工作包型
3. 进度回填型
4. 冻结裁定型
5. 线程接力型

同一份文档只承担一个主类型；如果必须混合，也要显式分段。

## 3. 先对齐母本、正式文档体系与代码现实

开始前先回答：

1. 当前内容属于哪个 Phase / 工作包？
2. V5 母本要求的能力和边界是什么？
3. 仓库正式文档体系里应该落到哪里？
4. 当前代码现实已经有了什么、还缺什么？
5. 哪些话可以写成已完成，哪些只能写成已规划 / 已实现待验证 / 阻塞？

## 4. 写清完成定义与非完成定义

每次冻结或回填都必须同时写：

- 完成定义
- 非完成定义

禁止把下列情况直接写成完成：

- 只有计划，没有实现
- 只有页面壳子，没有真实链路
- 只有 service / interface 草案，没有真实接线
- 只有代码提交，没有最小验证

## 5. 用诚实状态词回填

状态只能从下面集合里选：

- `已完成`
- `已实现待验证`
- `已规划`
- `进行中`
- `阻塞`

写状态时必须同时补：

- 证据路径
- 风险说明
- 缺口说明
- 下一步接力建议

## 6. 明确兄弟 skill 交接路线

文档线程结束时，至少明确：

- 后端实现 → `write-v5-runtime-backend`
- 前端控制面 → `write-v5-web-control-surface`
- 跨层交付 → `drive-v5-orchestrator-delivery`
- 运行验证 → `verify-v5-runtime-and-regression`
- 风险审查 → `review-v5-code-and-risk`
- 里程碑裁定 → `accept-v5-milestone-gate`

## 7. 回写正式文档体系

回写时遵守：

- 正式输出根目录是 `docs/01-版本冻结计划/V5/`
- 延续仓库既有冻结文档风格与层级
- 只创建当前线程真正需要的最小正式文档集合
- 不批量制造空白占位文档

## 8. 线程结束前自检

结束前至少确认：

- 我有没有引用 V5 母本，而不是脱离母本自由发挥？
- 我有没有最小核对代码现实？
- 我有没有把文档类型说清？
- 我有没有同时写完成定义和非完成定义？
- 我有没有给出下一线程 owner？
