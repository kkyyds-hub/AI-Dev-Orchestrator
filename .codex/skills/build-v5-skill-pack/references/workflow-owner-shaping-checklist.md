# workflow owner 化检查清单

## 1. 目的

把能力主题、草案描述或半成品说明，收敛成一个真正能带线程的 workflow owner skill。

这份清单用于快速判断：当前 skill 是否已经具备 owner 资格，而不是还停留在“会描述主题”。

## 2. 先问 4 个关键问题

### 这个 skill 到底负责什么？

如果一句话说不清它接哪类线程，就还不是 owner。

### 这个 skill 明确不负责什么？

如果边界说不清，就会和兄弟 skill 抢 owner。

### 线程开始时必须先读什么？

如果没有最小必读入口，新线程就无法稳定起步。

### 线程结束时应该留下什么？

如果没有明确交接物、证据或下一棒建议，就不适合作为 workflow skill。

## 3. 必须出现的结构要素

至少应有：

- 使命与 owner
- 强绑定的权威输入
- V5 母本绑定原则
- 技能边界（什么时候使用 / 不要使用）
- 正式落盘边界
- 开始入口
- 核心工作流
- 与兄弟 skill 的协作契约
- 非完成定义 / 红线
- done checklist

## 4. owner 边界判断题

至少要能回答：

- 它管理的是哪类线程，而不是哪类能力名词
- 它的主产物是什么
- 它最容易和哪个兄弟 skill 冲突
- 冲突时应该如何让位
- 什么时候应该先调用它，什么时候不该再继续停留在它

如果这些问题答不出来，就还没完成 owner 化。

## 5. 常见失败写法

- 只描述能力主题，不描述线程 owner
- 只写“能做什么”，不写“不能做什么”
- 只写理想状态，不写开始入口、文件范围和交接规则
- 只会讲概念，不会讲目录、产物、验证与 handoff
- 用抽象大词把多个 owner 面糊成一个 skill

## 6. 与本轮 4 个关键兄弟 skill 的最小分界

- 如果主问题是前端控制面真实落地 → 不该继续停留在 skill-pack owner
- 如果主问题是前端结构治理 → 不该继续停留在 skill-pack owner
- 如果主问题是跨层交付编排 → 不该继续停留在 skill-pack owner
- 如果主问题是运行验证与回归事实 → 不该继续停留在 skill-pack owner

只有当这些 owner 的 **skill 本身写坏了、边界失真了、包装残缺了**，才应该先由 `build-v5-skill-pack` 介入。

## 7. 一句话纪律

**能直接指导下一线程接棒，才算完成了 workflow owner 化。**
