# V5 跨层交付 playbook

## 1. 目标

把一次 V5 的 backend / web / docs / verify 跨面任务，收敛成 **切片明确、动作有序、状态诚实、证据可交接** 的最小跨层交付结果。

## 2. 先判 owner

开始前先回答：

1. 这次任务是否真的跨了至少两个 owner 面？
2. 当前主问题是跨层推进，还是其实只属于 backend / web / docs / verify / accept 其中一面？
3. 如果只剩单面动作，是否应该立刻降级给对应 skill？

只有当任务确实跨层时，才继续使用本 playbook。

## 3. Intake：先收齐最小输入

最少收这些输入：

- Phase / 工作包归属
- 当前切片目标
- backend / web / docs / verify 各面现状
- 已知阻塞、风险、依赖
- 当前已有证据与缺证项

如果连工作包归属和切片目标都说不清，就先缩小范围，不要直接做大总控。

## 4. 锁定最小跨层切片

优先选择这种切片：

- 一条 runtime 能力 + 一个前端入口 + 一层 verify
- 一个 team / strategy / memory 子能力的 backend-web 闭环
- 一个工作包的“实现 + 验证 + 文档回填准备”闭环

切片必须同时满足：

- 小到能在一轮线程内推进
- 大到足以形成真实交接

## 5. 先排动作顺序，再排文件改动

常见动作顺序：

1. 先收口依赖链
2. 再补最小验证
3. 再补 docs 回填与交接包装

常见动作面：

- backend：service / route / worker / schema
- web：页面 / 面板 / 抽屉 / 状态反馈
- docs：完成定义、状态回填、证据路径
- verify：build / API / page / smoke 证据

本 skill 的职责不是平均铺开，而是决定：

- 哪些面本轮必须纳入
- 哪些面本轮明确不做
- 哪些面留给下一线程

## 6. 保证状态诚实

线程结束时，至少同时写清：

- 已完成什么
- 明确没完成什么
- 当前是已完成 / 部分完成 / 阻塞
- 为什么还不能上升为更大范围结论

总控线程最忌讳把“推进了一截”写成“整条链已打通”。

## 7. 写证据与交接

至少留下：

- 关键证据
- 关键风险
- 缺证项
- 下一棒 owner
- 是否已经具备进入 `accept-v5-milestone-gate` 的材料

## 8. 线程结束前自检

结束前至少确认：

- 我有没有证明这是跨层任务？
- 我有没有把切片压到最小？
- 我有没有把 backend / web / docs / verify 的边界写清？
- 我有没有给出真实状态，而不是空泛推进感？
- 我有没有明确下一棒是谁？
