# V5 跨层交付 playbook

## 1. 目的

把一次 V5 的 backend / web / docs / verify 跨面任务，收敛成 **最小交付切片明确、动作顺序清楚、状态诚实、下一棒明确** 的跨层推进结果。

这个 playbook 只负责跨层工作包编排，不替 backend、web、verify、docs 或 accept 抢 owner。

## 2. 进入前先判断什么

开始前先回答：

1. 当前任务是否真的跨了至少两个 owner 面？
2. 当前属于哪个 Phase / 工作包？
3. 当前主问题是跨层整链推进，还是其实只属于 backend / web / docs / verify / accept 其中一面？
4. 如果只剩单面动作，是否应该立即降级回对应 owner skill？

如果不是跨层任务，就不要继续使用本 playbook。

## 3. 最小执行步骤

### 步骤 1：锁定最小交付切片

优先把范围压成：

- 一条 runtime 能力 + 一个前端入口 + 一层 verify
- 一个 team / strategy / memory 子能力的 backend-web 闭环
- 一个工作包的“实现 + 验证 + 文档回填准备”闭环

切片必须同时满足：小到能推进，大到足以交接。

### 步骤 2：列出本轮纳入的面

明确写清本轮是否纳入：

- backend
- web
- docs
- verify

同时写清本轮明确不做哪些面和哪些子项。

### 步骤 3：先排动作顺序

默认优先顺序：

1. 先收口依赖链
2. 再补最小验证
3. 再补 docs 回填与交接包装

### 步骤 4：同步记录真实状态

推进过程中持续记录：

- 已完成什么
- 部分完成什么
- 哪些点仍阻塞
- 当前为什么不能上升为更大范围结论

### 步骤 5：给下一棒留接棒位

结束前至少明确：

- 如果还需补实现，回哪个 skill
- 如果还需补验证，回哪个 skill
- 如果还需补文档回填，回哪个 skill
- 如果已经具备裁定材料，是否进入 `accept-v5-milestone-gate`

## 4. 执行时重点检查什么

- 这次是否真的属于跨层任务，而不是单面任务误升格
- 切片是否压到最小，没有为了“看起来完整”摊太大
- backend / web / docs / verify 的职责是否分清
- 依赖链是否先于“好看程度”被收口
- 状态是否诚实，没有把“推进了一截”写成“整条链完成”

## 5. 完成后怎么输出

结束时至少输出：

- 背景归属：Phase / 工作包 / 为什么属于跨层任务
- 本轮切片：目标、纳入的面、明确不做的范围
- 动作与产出：backend / web / docs / verify 各面做了什么
- 当前真实状态：已完成 / 部分完成 / 阻塞 / 当前不能宣布完成的范围
- 关键证据、关键风险、缺证项
- 下一线程建议与第二顺位

## 6. 什么时候应该让位给兄弟 skill

- 范围已经缩回纯后端实现 → `write-v5-runtime-backend`
- 范围已经缩回纯前端控制面实现 → `write-v5-web-control-surface`
- 范围已经缩回纯文档冻结与状态回填 → `manage-v5-plan-and-freeze-docs`
- 主问题变成纯运行验证与回归确认 → `verify-v5-runtime-and-regression`
- 主问题变成纯代码 / 合同 / 主链风险审查 → `review-v5-code-and-risk`
- 只剩里程碑通过 / 部分通过 / 阻塞裁定 → `accept-v5-milestone-gate`
