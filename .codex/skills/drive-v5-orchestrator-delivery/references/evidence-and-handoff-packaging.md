# V5 跨层交付证据与交接打包规则

## 1. 目的

总控线程最重要的价值，不是“看起来推进了很多”，而是：

- 有真实状态
- 有关键证据
- 有可接力的交接包

本清单用于规范线程结束时必须留下什么。

## 2. 最低交接包

线程结束时，至少应留下：

- Phase / 工作包定位
- 本轮切片定义
- 改动范围
- 已完成 / 部分完成 / 阻塞
- 关键证据
- 关键风险
- 遗留项
- 下一线程建议 skill

## 3. 证据优先级

优先保留：

1. build / 命令 / API / 页面 / smoke 输出
2. review 风险结论
3. freeze / 回填建议

如果没有强证据，就不要把状态写得太满。

## 4. 推荐的线程结束表达

优先写成：

- 本轮推进到哪里
- 哪些面已经闭合
- 哪些面仍未闭合
- 还不能宣布什么
- 下一步该由谁接

## 5. 最常见的错误

- 只写“已完成”，不写完成范围
- 只写“待补”，不写由谁补
- 只写“已验证”，不写验证层级
- 只写“建议下一步继续推进”，不写先接哪个 skill

## 6. 一句话纪律

总控线程的好坏，不看它动了多少目录，而看它是否让下一线程无需重新猜测上下文。
