# V5 交付切片与路由规则

## 1. 目的

V5 的总控线程最容易失败在两个地方：

- 切片太大，最后什么都没真正完成
- 明明只该交给单一 owner，却错误升级成跨层线程

本清单用于控制这两个风险。

## 2. 什么时候升级成总控线程

优先满足至少一条：

- 当前任务至少稳定跨两个 owner 面
- 如果不在同一线程里串联，交付闭环根本成立不了
- 用户目标明确要求“整条链推进到可交付”

## 3. 什么时候降级回单一 owner

满足任一情况可优先降级：

- 纯 backend
- 纯 web
- 纯 docs
- 纯 verify
- 纯 review
- 纯 accept

## 4. 切片规则

一个好切片通常满足：

- 能在一轮线程里说清楚目标
- 能有至少一层最小验证
- 能留下明确遗留项
- 不需要同时推翻多个系统边界

一个坏切片通常表现为：

- 涉及 4 个面，但每个面只动一点点
- 线程结束后仍说不清到底完成了什么
- 需要下游重新判断这次线程到底在干嘛

## 5. 路由规则

总控线程内的动作路由建议：

- 缺 runtime 主链 → `write-v5-runtime-backend`
- 缺控制面 / 观察面 → `write-v5-web-control-surface`
- 缺完成定义或状态回填 → `manage-v5-plan-and-freeze-docs`
- 缺运行事实 → `verify-v5-runtime-and-regression`
- 缺风险澄清 → `review-v5-code-and-risk`
- 已具备裁定材料 → `accept-v5-milestone-gate`

## 6. 一句话纪律

总控不是“多做一点”，而是“把多面动作编排成一个诚实可交付的最小闭环”。
