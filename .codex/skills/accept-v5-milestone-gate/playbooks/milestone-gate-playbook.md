# V5 里程碑裁定 playbook

## 1. 目标

把一次“能不能算完成 / 能不能放行”的请求，压成 **对象清楚、证据清楚、分级清楚、遗留项清楚、下一棒清楚** 的正式 gate 结果。

这个 playbook 面向四类高频场景：

1. 工作包或单轮切片的正式裁定
2. 阶段级裁定前的收口判断
3. 前端结构治理后的放行判断
4. 证据不足时的“无法裁定”处理

## 2. 第一步先判 owner

先回答三件事：

1. 当前主任务真的是“裁定”，还是还在修 skill / 补实现 / 跑 verify / 做 review？
2. 当前对象是一个切片、一个工作包，还是整个 Phase？
3. 这次结论最后要交回谁继续推进？

判定规则：

- 修 skill 包本身 → `build-v5-skill-pack`
- 做前端控制面实现 → `write-v5-web-control-surface`
- 做前端结构治理 → `govern-v5-web-structure`
- 做跨 backend / web / docs / verify 的整链推进 → `drive-v5-orchestrator-delivery`
- 查 build / 页面 / API / 日志 / 回归事实 → `verify-v5-runtime-and-regression`
- 做最终放行或不放行判断 → 继续本 playbook

## 3. Intake：先收齐最小输入

最少收这些输入：

- V5 Phase / 工作包 / 切片定义
- 完成定义或 freeze 口径
- 实现说明
- 最近一轮 review 结论
- 最近一轮 verify 结论
- 当前已知遗留项、风险项、依赖项

如果这六类里缺关键项，就先记录缺口，不要急着下 Pass。

## 4. 先锁定“这次只裁定什么”

每次裁定都先写：

- 本次裁定对象
- 本次裁定层级
- 本次明确不裁定的更大范围
- 本次结论会影响哪个状态口径

默认纪律：

- 先裁定工作包 / 切片
- 不默认直接上升为整个 Phase
- 只有当多工作包证据闭合时，才考虑 Phase 级结论

## 5. 把证据分层

建议把当前材料分成四层：

1. 完成定义层：母本、计划、freeze、工作包目标
2. 实现层：代码、页面、接口、改动说明
3. 风险层：review 结论、高风险、合同 / 主链 / 结构风险
4. 事实层：verify、build、页面、API、日志、脚本、回归结果

纪律：

- 只有实现层，没有事实层，通常不能给 Pass。
- 只有事实层，没有完成定义层，也不能说“已经完成”。
- 风险层缺失时，通常最多给 `无法裁定` 或保守 `Partial`，不要乐观放行。

## 6. 先判断“缺证”还是“真阻塞”

### 该判为 `无法裁定`

- 验收对象不清
- 没有完成定义
- 缺 review 或 verify 到无法负责任分级
- 证据互相冲突但还没核清

### 该判为 `Blocked`

- 对象清楚
- 已知失败事实清楚
- 失败或高风险足以阻断放行
- 现在已经知道必须先回退谁去处理

## 7. 再判断最多能说到哪一步

典型收敛方式：

- 最多能说“后端切片已闭合” → 不要升格成“整链可交付”
- 最多能说“结构治理切片 build + 入口 + 最小回归通过” → 不要升格成“前端整体稳定”
- 最多能说“控制面页面已具备最小闭环” → 不要升格成“阶段已完成”

## 8. 分级口诀

### Pass

- 对象明确
- 范围闭合
- 关键证据充分
- 风险可接受
- 遗留项不会推翻当前结论

### Partial

- 主体已成
- 缺口明确
- 可以继续推进
- 但更大范围不能一起放行

### Blocked

- 对象明确
- 阻塞明确
- 当前不能放行
- 必须先回退处理

### 无法裁定

- 分级基础不够
- 先补材料
- 不能硬判

## 9. 结构治理放行场景的专门做法

当上游来自 `govern-v5-web-structure` 时，至少追加判断：

- 本次治理切片到底是什么
- 入口页 / 关键页面 / build / 脚本 / 锚点有没有 verify 事实
- 是否只是“结构更好看了”，还是已经证明没打坏旧链路
- 结论是“本次治理切片可放行”，还是只能说“需要继续补 verify / 锚点 / 文档”

推荐规则：

- 有 build + 入口 + 最小回归 + 风险可接受 → 可考虑 `Pass`
- 结构主体完成，但证据只覆盖最小面 → 多数应给 `Partial`
- 结构改动打坏入口或旧链路 → `Blocked`
- 只有 diff，没有 verify / review / 目标说明 → `无法裁定`

## 10. 线程结束前必须写清的 7 件事

1. 结论等级
2. 验收对象
3. 依据材料
4. 能确认完成的范围
5. 不能确认完成的范围
6. 遗留项 / 缺证项 / 前置条件
7. 下一棒 owner

## 11. 最后一轮自检

结束前至少确认：

- 我有没有把“能不能过”翻译成明确对象？
- 我有没有区分 `无法裁定` 和 `Blocked`？
- 我有没有把结论限制在证据真正覆盖的范围内？
- 我有没有把结构治理场景和控制面交付场景分开？
- 我有没有明确下一棒是谁，而不是只给一句空泛建议？
