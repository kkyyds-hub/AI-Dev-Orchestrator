# V5 验收证据与汇总检查清单

## 1. 目的

很多错误验收不是因为不会判断，而是因为材料不齐、证据失衡，或者把不同层级的事实混在一起。

这个清单用于快速判断：当前材料够不够支撑一个可靠裁定，以及当前最多能把结论说到哪一步。

## 2. 最低材料要求

理想情况下，至少同时具备：

- 完成定义或计划口径
- 实现摘要
- review 结论
- verify 结论

如果缺其中任一项，要明确写：

- 缺了什么
- 这会限制你把结论说到哪一步
- 当前应该先回退给哪个 skill

## 3. 四层证据框架

### A. 完成定义层

常见材料：

- V5 母本章节
- 当前工作包计划
- freeze / 回填文档
- 用户明确授权范围

没有这一层，通常不能说“完成”。

### B. 实现层

常见材料：

- 代码 diff
- 页面改动
- 接口合同
- 上游实现说明

没有这一层，通常不能说“已落地”。

### C. 风险层

常见材料：

- `review-v5-code-and-risk` 结论
- 合同风险、结构风险、主链风险
- 高风险是否已消化或隔离

没有这一层，通常不能乐观 Pass。

### D. 事实层

常见材料：

- `verify-v5-runtime-and-regression` 结果
- build 输出
- API 响应
- 页面访问结果
- 日志 / 脚本 / 锚点结果

没有这一层，通常不能把“实现存在”写成“已验证可用”。

## 4. 证据强度

建议按强到弱记录：

1. 实际命令输出 / build 输出 / 启动日志
2. API 响应与状态码
3. 页面打开、交互、脚本运行结果
4. review 风险结论
5. 代码阅读后的合理推断

如果只有第 5 类，必须显式写“仅代码核对，未形成运行事实”。

## 5. 工作包级裁定最低要求

优先至少确认：

- 该工作包目标是什么
- 当前实现覆盖了哪些目标
- review 有没有高风险阻断
- verify 覆盖了哪些目标
- 当前遗留项会不会推翻“工作包基本闭合”的结论

## 6. Phase 级裁定额外要求

除了工作包级材料，还要确认：

- Phase 下关键工作包是否基本闭合
- 主要遗留项是否仍可能推翻阶段结论
- 是否已有足够 build / smoke / page / API 证据支撑更大范围结论
- 文档口径是否已经与现实同步

没有这些条件，就优先降回工作包级裁定。

## 7. 前端结构治理放行的额外检查

当对象涉及 `govern-v5-web-structure` 产物时，至少额外确认：

- 本次治理切片是什么
- 入口页、关键页面或关键脚本有没有 verify 事实
- `data-testid` / 脚本选择器是否需要同步
- build、入口或最小回归是否说明旧链路没被打坏

如果只有“结构拆分完成”的自述，没有回归事实，通常最多给 `Partial` 或 `无法裁定`。

## 8. 缺证与阻塞的区分清单

### 更像缺证，应优先考虑 `无法裁定`

- 没有 review 结论
- 没有 verify 结论
- 完成定义不清
- 不知道对象是切片还是 Phase
- 上游说法互相打架

### 更像阻塞，应优先考虑 `Blocked`

- build 失败
- 页面入口断裂
- API 不通
- 关键回归失败
- 高风险明确未解且足以阻断放行

## 9. 遗留项检查

每次裁定至少留下：

- 风险
- 缺证项
- 必补项
- 下一阶段前置条件
- 下一线程建议 skill

没有这些信息的裁定，不适合做接力基线。

## 10. 推荐交接规则

- 缺完成定义 → `manage-v5-plan-and-freeze-docs`
- 缺实现 → `write-v5-runtime-backend` / `write-v5-web-control-surface`
- 缺结构治理收口 → `govern-v5-web-structure`
- 缺风险澄清 → `review-v5-code-and-risk`
- 缺运行证据 → `verify-v5-runtime-and-regression`
- 本质跨层 → `drive-v5-orchestrator-delivery`
- 需要修 accept skill 包本身 → `build-v5-skill-pack`

## 11. 一句话纪律

验收不是为了“宣布成功”，而是为了把当前真实边界、缺口和下一棒说清楚。
