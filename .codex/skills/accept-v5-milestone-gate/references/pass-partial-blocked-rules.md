# V5 Pass / Partial / Blocked / 无法裁定 规则

## 1. 目的

统一 `accept-v5-milestone-gate` 的裁定表达，避免因为表达模糊导致后续线程误解状态。

## 2. Pass

优先适用于：

- 范围明确且闭合
- 关键实现已经完成
- 最小必要验证通过
- 重大风险已消化、可接受或已明确隔离
- 文档、代码、review、verify 口径一致

### Pass 结论里必须写

- 到底什么范围 Pass 了
- 哪些更大范围并没有一起 Pass
- 剩余遗留项为什么不会推翻当前结论

## 3. Partial

优先适用于：

- 核心部分完成，但有明确残留缺口
- 切片完成，但阶段未完成
- 可以继续往下推进，但必须保留未完成项和前置条件

### 最典型的 Partial 场景

- backend OK，web 未闭合
- web 有入口，联调证据不足
- API 存在，但主链未完全接上
- 结构治理主体完成，但脚本 / 锚点 / 次级页面 / 文档仍待同步
- verify 只覆盖最小切片，尚不足以支撑阶段级 Pass

### Partial 结论里必须写

- 当前到底完成到了哪一步
- 还差什么才可能升级为 Pass
- 下一棒 owner 是谁

## 4. Blocked

优先适用于：

- 验收对象明确
- 关键实现未完成，或关键依赖 / 高风险明确阻断放行
- 已有明确失败事实，而不是单纯缺证
- 当前必须先回退给实现 / 治理 / verify / review 线程处理

### 最典型的 Blocked 场景

- build 失败
- 页面入口断裂
- API / worker 主链不通
- review 明确指出高风险且未解决
- 结构治理打坏旧链路或关键脚本

### Blocked 结论里必须写

- 被什么阻塞
- 哪份证据证明它被阻塞
- 先补什么
- 先由哪个 skill 接手

## 5. 无法裁定

优先适用于：

- 验收对象不清
- 完成定义缺失
- review / verify 缺失到无法负责任分级
- 关键证据之间冲突，但当前还没查清
- 只有实现自述，没有足够事实材料

### 最典型的 无法裁定 场景

- “帮我看看能不能过”但没说过哪个对象
- 只有代码 diff，没有 review / verify
- 结构治理只给出重构结果，没有 build / 入口 / 回归事实
- 工作包目标与 freeze 口径互相打架

### 无法裁定 结论里必须写

- 缺了什么材料
- 为什么这些缺口使你无法负责任分级
- 先补哪类材料
- 先回退给哪个 skill

## 6. 一个基本纪律

当你犹豫是 `Pass` 还是 `Partial` 时，默认先给 `Partial`。

当你犹豫是 `Blocked` 还是 `无法裁定` 时，先问自己：

- 是不是已经有明确失败事实？有的话更像 `Blocked`
- 还是只是材料不够？那更像 `无法裁定`

因为 V5 的验收目标是诚实收口，不是乐观表态或武断定性。
