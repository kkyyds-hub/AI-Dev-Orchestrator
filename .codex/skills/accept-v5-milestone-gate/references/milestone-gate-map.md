# V5 里程碑裁定地图

## 1. 目的

把 `accept-v5-milestone-gate` 绑定到 V5 母本里的真实阶段与验收口径，避免线程一上来就凭感觉宣布“过了”。

这个参考文件回答四个问题：

1. 当前最适合被裁定的对象有哪些
2. 什么情况下适合裁定工作包，什么情况下才适合裁定 Phase
3. 裁定要依赖哪些材料
4. 如何避免把局部完成误判成整阶段完成

## 2. 优先裁定对象

### 工作包 / 单轮切片

优先适用于：

- 某条后端切片已经过实现、review、verify
- 某条前端控制面切片已经有页面、合同、验证结果
- 某条 provider / token / memory / team-control 子能力需要明确当前状态

这是默认最安全的裁定层级。

### Phase 级裁定

只有在下列条件更接近满足时，才建议提升到 Phase 级：

- 该 Phase 的关键工作包已经基本闭合
- review 与 verify 结论覆盖面足够
- 主要遗留项不会推翻阶段结论
- 文档口径已经与现实同步

否则，应优先给工作包级 `Pass/Partial/Blocked`，而不是整 Phase 裁定。

## 3. 与母本验收章节的关系

V5 母本已经明确三类验收口径：

- 产品能力验收
- 工程验收
- 成本验收

里程碑裁定时，不要求每次都覆盖三类全部细项，但至少要说明：

- 本次裁定主要参考了哪一类
- 哪些类目前还没有足够材料

## 4. 最常见的误判

### 误判 1：把“切片完成”写成“阶段完成”

例如：

- Provider 最小链路通了，就宣布 Phase 1 全部完成
- team control 页面有了，就宣布老板控制中心完成

### 误判 2：没有 verify 也给 pass

例如：

- 代码写完了，但没有最小构建 / 运行 / 页面 / API 证据

### 误判 3：高风险仍未消化，却给 pass

例如：

- schema / migration 风险未解
- worker 主链未全接通
- 页面与后端合同不一致

## 5. 推荐裁定顺序

建议顺序：

1. 先确认验收对象
2. 再收齐完成定义、review、verify
3. 再判断最多能说到哪里
4. 最后给 `Pass / Partial / Blocked`

如果还没回答“最多能说到哪里”，就不要先写结论。
