# V5 里程碑裁定地图

## 1. 目的

把 `accept-v5-milestone-gate` 绑定到 V5 母本里的真实阶段、工作包和放行口径，避免线程一上来就凭感觉宣布“过了”。

这个参考文件回答五个问题：

1. 当前最适合被裁定的对象有哪些
2. 什么情况下适合裁定工作包，什么情况下才适合裁定 Phase
3. 哪些场景应该先去别的 owner skill，而不是直接做 accept
4. 前端结构治理后如何判断是否具备放行条件
5. 如何避免把局部完成误判成更大范围完成

## 2. 优先裁定对象

### 工作包 / 单轮切片

优先适用于：

- 某条后端切片已经过实现、review、verify
- 某条前端控制面切片已经有页面、合同、验证结果
- 某条 provider / token / memory / team-control 子能力需要明确当前状态
- 某条结构治理切片已经完成，需判断是否可以作为稳定基线放行

这是默认最安全的裁定层级。

### Phase 级裁定

只有在下列条件更接近满足时，才建议提升到 Phase 级：

- 该 Phase 的关键工作包已经基本闭合
- review 与 verify 结论覆盖面足够
- 主要遗留项不会推翻阶段结论
- 文档口径已经与现实同步

否则，应优先给工作包级 `Pass / Partial / Blocked / 无法裁定`，而不是整 Phase 裁定。

## 3. 先别急着进 accept 的场景

下面这些情况更应该先去别的 owner：

- skill 本身失真、乱码、模板缺失 → `build-v5-skill-pack`
- 还在写控制面实现 → `write-v5-web-control-surface`
- 还在做结构治理方案和落地 → `govern-v5-web-structure`
- 还在做跨层协调和整链推进 → `drive-v5-orchestrator-delivery`
- 还在补运行事实或回归证据 → `verify-v5-runtime-and-regression`

只有当主问题已经收敛成“现在最多能放行到哪一步”，才适合进入 accept。

## 4. 与母本验收章节的关系

V5 母本已经明确三类验收口径：

- 产品能力验收
- 工程验收
- 成本验收

里程碑裁定时，不要求每次都覆盖三类全部细项，但至少要说明：

- 本次裁定主要参考了哪一类
- 哪些类目前还没有足够材料
- 当前结论能不能上升到更大范围

## 5. 前端结构治理放行地图

当上游来自 `govern-v5-web-structure` 时，accept 的判断重点不是“结构是否优雅”，而是：

- 这次治理切片是否定义清楚
- 这次治理是否已由 review / verify 形成最小事实闭环
- 入口页、构建、关键页面、脚本、锚点或回归是否仍稳定
- 本次最多能宣布“结构治理切片通过”还是只能宣布“仍需继续治理 / 验证”

### 结构治理场景常见结论

- **Pass**：切片明确，build / 入口 / 最小回归通过，遗留项不会推翻本切片结论
- **Partial**：治理主体完成，但脚本 / 锚点 / 次级页面 / 文档仍待同步
- **Blocked**：入口、构建、关键链路或高风险明确阻断放行
- **无法裁定**：缺完成定义、缺 verify、缺 review，连分级基础都不够

## 6. 最常见的误判

### 误判 1：把“切片完成”写成“阶段完成”

例如：

- provider 最小链路通了，就宣布 Phase 1 全部完成
- team control 页面有了，就宣布老板控制中心完成
- 结构治理切片收敛了，就宣布整个前端控制面稳定

### 误判 2：没有 verify 也给 Pass

例如：

- 代码写完了，但没有最小 build / 页面 / API / 回归证据

### 误判 3：高风险仍未消化，却给 Pass

例如：

- schema / migration 风险未解
- worker 主链未全接通
- 页面与后端合同不一致
- 结构治理后旧入口或旧脚本已断裂

### 误判 4：材料不齐，却硬判 Blocked

例如：

- 其实只是没有 verify 结果，却被写成“已经阻塞”
- 其实对象都没定义清楚，却强行下阻塞结论

## 7. 推荐裁定顺序

建议顺序：

1. 先确认验收对象
2. 再确认 owner 是否正确
3. 再收齐完成定义、review、verify
4. 先判断是“缺证”还是“阻塞”
5. 再判断最多能说到哪里
6. 最后给 `Pass / Partial / Blocked / 无法裁定`

如果还没回答“最多能说到哪里”，就不要先写结论。
