# V5 验证证据分级与回归检查清单

## 1. 目的

V5 验证线程最容易出问题的地方，不是不会跑命令，而是：

- 跑了点东西，却没有形成可引用证据
- 只验证新功能，不看旧功能是否回归
- 把代码阅读推断当成强证据

本清单用于统一证据强度和回归口径。

## 2. 证据强度分级

按强到弱建议这样分：

### A 级：强运行证据

- 命令输出
- build 输出
- 启动成功 / 失败日志
- API 响应与状态码
- 结构化运行日志

### B 级：强观察证据

- 页面打开结果
- 关键交互结果
- 实时事件流或日志回显

### C 级：弱推断证据

- 代码阅读
- 目录检查
- 配置检查

规则：

- 只要最终结论依赖 C 级证据，就必须明确写“未实际运行验证”。
- 能用 A 级证据时，不要只给 C 级推断。

## 3. 回归检查最低要求

每次验证至少回答：

- 这次改动可能影响哪条旧链路
- 我有没有对那条旧链路做最小检查
- 如果没查，为什么没查

典型回归面：

- 后端还能否启动
- 关键 API 路由是否仍可访问
- worker 最小链路是否仍可触发
- 前端 `npm run build` 是否仍通过
- 旧页面是否因字段变化直接报错

## 4. 失败记录最低要求

如果验证失败，至少记录：

- 哪一步失败
- 报错原文或要点
- 推测根因
- 阻断了哪个 V5 结论
- 下一线程该交给谁

如果失败了但没有这些信息，这个验证结果就不可复用。

## 5. 可复用结论写法

优先写成：

- 已验证通过
- 已部分验证通过
- 已实现但当前环境未验证
- 验证失败
- 环境阻塞无法验证

不要只写“差不多可以”“应该没问题”。
