# V5 验证证据分级与回归检查清单

## 1. 目的

V5 验证线程最容易出问题的地方，不是不会跑命令，而是：

- 跑了点东西，却没有形成可引用证据
- 只验证新功能，不看旧功能是否回归
- 把代码阅读推断当成强证据
- 接到结构治理或锚点变更产物时，没有补最小同步验证

本清单用于统一证据强度、回归口径和交接写法。

## 2. 证据强度分级

按强到弱建议这样分：

### A 级：强运行证据

- 命令输出
- build 输出
- 启动成功 / 失败日志
- API 响应与状态码
- 结构化运行日志
- 脚本运行结果

### B 级：强观察证据

- 页面打开结果
- 关键交互结果
- 实时事件流或日志回显
- 目标元素、锚点或状态区域的页面观察结果

### C 级：弱推断证据

- 代码阅读
- 目录检查
- 配置检查
- 仅查看 `data-testid` 或脚本源码但未执行

规则：

- 只要最终结论依赖 C 级证据，就必须明确写“未实际运行验证”。
- 能用 A 级证据时，不要只给 C 级推断。
- 接到结构治理或锚点变更场景时，不能只靠“代码看起来没问题”就写成通过。

## 3. 回归检查最低要求

每次验证至少回答：

- 这次改动可能影响哪条旧链路
- 我有没有对那条旧链路做最小检查
- 如果没查，为什么没查

典型回归面：

- 后端还能否启动
- 关键 API 路由是否仍可访问
- worker 最小链路是否仍可触发
- 前端 `npm run build` 是否仍通过
- 旧页面是否因字段变化直接报错
- 原有 `data-testid`、脚本选择器或页面观察点是否失效

## 4. 场景化最小回归要求

### 后端启动 / API / worker

至少补一项：

- `health` 仍可用
- 原有 worker / run / log 观察面仍可见
- 相关 `*_smoke.py` 仍可跑或至少能定位阻塞点

### 前端 build / 页面入口 / 关键交互

至少补一项：

- `npm run build` 仍通过
- 原有页面入口仍可达
- 旧交互路径没有立即报错

### 前端结构治理后的联调影响

至少补一项：

- `App.tsx` / `ProjectOverviewPage.tsx` 挂载入口仍正常
- 被拆分后的页面仍能通过原入口到达
- 受影响脚本、锚点或页面最小观察面仍正常

### 测试锚点变更后的最小同步验证

至少补一项：

- 受影响脚本实际运行结果
- 页面目标元素观察结果
- 受影响 `data-testid` 与脚本选择器核对结果

如果只完成最后一项，也必须标明“仅代码核对，未实际运行”。

## 5. 上游接棒补充要求

### 来自 `write-v5-web-control-surface`

至少补：

- build 或页面入口中的一种强证据
- 一条关键交互结果
- 一条旧链路回归判断

### 来自 `govern-v5-web-structure`

至少补：

- 入口页 / 挂载关系中的一种事实
- 受影响脚本或锚点中的一种同步结果
- 明确这次验证的是“影响事实”，不是“治理方案优劣”

### 来自 `drive-v5-orchestrator-delivery`

至少补：

- 最小闭环中的一条强证据
- 当前未覆盖的层面
- 下一棒是否还需要回到 backend / web / docs

### 来自 `build-v5-skill-pack`

至少补：

- 本次真正要验证的运行对象
- 明确不把 skill 包维护结果当成运行事实

## 6. 失败记录最低要求

如果验证失败，至少记录：

- 哪一步失败
- 报错原文或要点
- 推测根因
- 阻断了哪个 V5 结论
- 下一线程该交给谁

如果失败了但没有这些信息，这个验证结果就不可复用。

## 7. 可复用结论写法

优先写成：

- 已验证通过
- 已部分验证通过
- 已实现但当前环境未验证
- 验证失败
- 环境阻塞无法验证

同时补上：

- 已验证范围
- 未验证范围
- 最小回归情况
- 下一棒 owner

不要只写“差不多可以”“应该没问题”。
