# V5 运行验证 playbook

## 1. 目的

把一次“有人声称已经完成”的线程，压成 **验证范围明确、证据等级明确、失败事实可引用、最小回归已交代** 的事实确认结果。

这个 playbook 只负责运行事实和最小回归确认，不负责继续补实现、做结构治理、修 skill 包或做阶段裁定。

## 2. 进入前先判断什么

开始前先回答：

1. 当前主任务是“查事实”，还是“补实现 / 做治理 / 修 skill 包 / 做阶段裁定”？
2. 当前来源是哪个上游 owner：`write-v5-web-control-surface`、`govern-v5-web-structure`、`drive-v5-orchestrator-delivery`，还是普通实现线程？
3. 这次验证结果最后要交回谁继续推进？
4. 如果现在不做 verify，而去补实现或治理，会不会混淆事实结论？

如果主问题已经不是事实确认，就不要继续留在 verify。

## 3. 最小执行步骤

### 步骤 1：先收齐最小输入

最少收这些输入：

- 上游来源 skill / 线程 / 工作包
- 上游声称已完成的点
- V5 母本或冻结口径里的对应目标
- 代码入口、页面入口、脚本或日志入口

### 步骤 2：把宣称拆成验证点

把上游声称逐条翻译成可验证点，例如：

- provider / worker / run / log 是否真的有痕迹
- 页面入口是否可达
- 关键交互是否至少走通一条最小路径
- 结构治理后旧入口、旧脚本、旧锚点是否仍可用

拆不开就先降级成“需要澄清”，不要直接验证通过。

### 步骤 3：选择最小足够的验证层级

优先从小到大选：

1. 静态层
2. 构建层
3. 接口层
4. 页面层
5. 联调层
6. 回归层

不要每次都默认做最大层级。

### 步骤 4：执行最小验证并补最小回归

按当前场景至少覆盖一层强证据，并在以下场景补最小回归：

- 后端场景：至少确认一条旧的 health / worker / run 观察面
- 前端控制面场景：至少确认 build 或旧入口仍可达
- 结构治理场景：至少确认入口、挂载关系、原脚本或原锚点中的一项仍可用
- 锚点变更场景：至少确认受影响脚本、页面或选择器已有一层同步结果

### 步骤 5：记录结果并做分级

至少区分：

- 已验证通过
- 已部分验证通过
- 已实现但当前环境未验证
- 验证失败
- 环境阻塞无法验证

同时写清已验证范围、未验证范围和下一棒 owner。

## 4. 执行时重点检查什么

- 验证范围是否清楚，是否没有夸大成全量通过
- 证据等级是否足够，是否区分了“实际运行”与“仅代码推断”
- 失败事实是否被完整记录，没有被回避
- 当前场景是否覆盖了最小回归，特别是结构治理后的入口 / 锚点 / 脚本影响
- 结论是否仍停留在 verify 边界内，没有越权做实现或验收裁定

## 5. 完成后怎么输出

结束时至少输出：

- 上游接棒来源与本次验证归属
- 本次验证范围与未验证范围
- 执行动作：命令 / API / 页面 / 脚本 / 锚点
- 结果事实：成功 / 失败 / 阻塞
- 证据摘要：关键输出、日志路径、报错要点、仅代码推断部分
- 回归判断：受影响旧链路、最小回归动作、是否发现回归
- 结论分级与下一线程建议

## 6. 什么时候应该让位给兄弟 skill

- 发现主问题是后端实现缺口 → `write-v5-runtime-backend`
- 发现主问题是前端控制面实现缺口 → `write-v5-web-control-surface`
- 发现主问题是前端结构治理、入口边界或锚点治理 → `govern-v5-web-structure`
- 发现任务已经自然跨 backend / web / docs / verify 多面收口 → `drive-v5-orchestrator-delivery`
- 主问题是 skill 包本身损坏 → `build-v5-skill-pack`
- 主问题是冻结文档与状态回填 → `manage-v5-plan-and-freeze-docs`
- 主问题是代码 / 合同 / 主链风险审查 → `review-v5-code-and-risk`
- 主问题是阶段 Pass / Partial / Blocked 裁定 → `accept-v5-milestone-gate`
