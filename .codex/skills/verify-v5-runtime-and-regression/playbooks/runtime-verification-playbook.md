# V5 运行验证 playbook

## 1. 目标

把一次“有人声称已经完成”的线程，压成 **范围明确、证据可引用、最小回归已交代、下一棒 owner 清楚** 的事实确认结果。

这个 playbook 面向四类高频场景：

1. 后端启动 / API / worker 的最小事实确认
2. 前端 build / 页面入口 / 关键交互的最小事实确认
3. 前端结构治理后的联调影响确认
4. 测试锚点变更后的最小同步验证

## 2. 先做 owner 判断

先回答三件事：

1. 当前主任务是“查事实”还是“补实现 / 做治理 / 修 skill 包 / 做阶段裁定”？
2. 当前来源是哪个上游 owner：`write-v5-web-control-surface`、`govern-v5-web-structure`、`drive-v5-orchestrator-delivery`、普通实现线程，还是 `build-v5-skill-pack` 的维护结果？
3. 这次验证的结论最终要交还给谁继续推进？

判定规则：

- 如果当前主任务是修 verify skill 包本身，立刻切到 `build-v5-skill-pack`。
- 如果当前主任务是继续补页面、补后端、做结构治理，也不要留在 verify。
- 只有当当前主任务是“确认事实、确认回归、确认是否阻塞下一棒”时，才继续使用本 playbook。

## 3. Intake：先收齐最小输入

最少收这些输入：

- 上游来源 skill / 线程 / 工作包
- 上游声称已完成的点
- V5 母本或冻结口径里的对应目标
- 代码入口
- 现有脚本、日志、页面入口或锚点入口

如果来源不同，优先补读：

### 来自 `write-v5-web-control-surface`

优先补读：

- 新增或改动的页面 / panel / drawer / form
- 对应 `hooks.ts` / `types.ts` / `api.ts`
- 页面入口挂载点
- 上游自述的关键交互路径

### 来自 `govern-v5-web-structure`

优先补读：

- `App.tsx` / `ProjectOverviewPage.tsx` 或其他入口页改动
- 新拆出的 `sections/*`、`panel/*`、`hooks.ts`、`api.ts`
- 受影响的 `data-testid`
- 受影响的 `apps/web/scripts/*.spec.mjs`

### 来自 `drive-v5-orchestrator-delivery`

优先补读：

- 工作包切片定义
- backend / web / docs / verify 的声称完成项
- 当前最小闭环依赖链路

### 来自 `build-v5-skill-pack`

优先补读：

- 仅与本次实际运行验证有关的 handoff 说明
- 不要把 skill 文件维护结果误当成运行事实

## 4. 把宣称拆成验证点

把上游声称逐条翻译成“可以被证据支持或否定”的验证点。

例子：

- “worker 已接主链” → 能否触发最小 worker；是否看到 run / log / event 痕迹
- “控制面已可用” → build 是否通过；页面入口是否可达；关键交互是否至少走通一条最小路径
- “结构治理不影响联调” → 入口页是否仍挂得上；旧脚本或旧锚点是否仍可用；旧路径是否回归
- “测试锚点已同步” → 受影响的 `data-testid`、脚本选择器、页面元素是否一致

如果一句话拆不开，就先降级成“需要进一步澄清”，不要直接进入验证通过。

## 5. 选择最小足够的验证层级

优先按下面顺序选最小足够层级：

1. 静态层：入口、路径、脚本、锚点、导入可读
2. 构建层：后端启动 / 前端 build
3. 接口层：health / workers / runs / events 等接口请求
4. 页面层：页面可打开、入口可达、关键交互可观察
5. 联调层：前后端最小链路跑通
6. 回归层：原有链路、脚本或锚点的最小复核

纪律：

- 不要为了“显得全面”一上来做最大层级。
- 也不要只做代码阅读就宣布通过。
- 如果当前场景是结构治理或锚点变更，至少补一层与旧链路有关的回归判断。

## 6. 场景化执行建议

### A. 后端启动 / API / worker

最小建议动作：

- 后端启动或导入确认
- `health` 最小可达性确认
- `workers` / `runs` / `events` 其中一条观察面确认
- 如有现成 `*_smoke.py`，优先复用

强证据优先级：

1. 启动输出 / 错误输出
2. API 状态码与响应
3. run log / JSONL / event 证据

### B. 前端 build / 页面入口 / 关键交互

最小建议动作：

- `npm run build`
- 页面入口是否仍可挂载
- 当前改动最相关页面是否至少可达
- 一条关键交互路径是否有结果或明显报错

强证据优先级：

1. build 输出
2. 页面访问结果
3. 关键交互结果

### C. 前端结构治理后的联调影响

最小建议动作：

- 确认 build 是否通过
- 确认入口页和被拆分页面是否仍可达
- 确认原脚本、原锚点或原入口中至少一条旧链路是否仍可用
- 记录“这次只确认影响是否可见，不替治理线程给结构方案背书”

强证据优先级：

1. build 输出
2. 页面入口 / 挂载结果
3. 受影响脚本或旧链路结果

### D. 测试锚点变更后的最小同步验证

最小建议动作：

- 列出受影响 `data-testid`
- 列出受影响脚本，如 `apps/web/scripts/*.spec.mjs`
- 确认脚本、选择器、页面元素至少有一层同步结果
- 如果没实际跑脚本，明确写“仅代码核对，未实际运行”

强证据优先级：

1. 脚本运行结果
2. 页面元素观察结果
3. 锚点代码核对结果

## 7. 失败时怎么写

失败不是坏事；坏的是没有形成可复用失败事实。

至少记录：

- 失败步骤
- 报错原文或要点
- 推测根因
- 阻断了哪个 V5 结论
- 下一棒回退给谁

典型回退：

- 页面或字段缺口 → `write-v5-web-control-surface`
- 结构入口或锚点治理缺口 → `govern-v5-web-structure`
- 后端链路缺口 → `write-v5-runtime-backend`
- 跨层协调缺口 → `drive-v5-orchestrator-delivery`

## 8. 结论写法

结论优先用下列分级：

- 已验证通过
- 已部分验证通过
- 已实现但当前环境未验证
- 验证失败
- 环境阻塞无法验证

并且总是补上：

- 已验证范围
- 未验证范围
- 最小回归是否已覆盖
- 下一棒 owner

## 9. 线程结束前自检

结束前至少确认：

- 我有没有把“宣称”拆成验证点？
- 我有没有给出至少一种强证据？
- 我有没有交代旧链路是否受影响？
- 我有没有清楚说明下一棒是谁？
- 我有没有越权把 verify 变成实现、治理、skill 维护或验收线程？
